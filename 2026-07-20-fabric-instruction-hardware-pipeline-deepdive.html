1. 问题定义：从 PTX 到物理内存的完整路径

CUDA CFT 编程模型在 PTX ISA 9.3 / CUDA 13.3 中引入了 6 条 fabric 指令，使 GPU 线程能够通过 NVLink 织物发起远程内存操作。一个 fabric 操作的完整路径涉及 7 个硬件/软件层级：

┌─────────────────────────────────────────────────────────────────────┐
│  层级 1: PTX 指令层                                                   │
│  fabric.try_get/put/red/pullred + fabric.submit + fabric.wait        │
│  → 定义操作的语义（读/写/归约）和完成追踪（mbarrier）                   │
├─────────────────────────────────────────────────────────────────────┤
│  层级 2: SM 发射层                                                     │
│  Load-Store Unit (LSU) 识别 fabric 操作，构造 fabric 请求包             │
│  → 提取 CFT Handle = {leId (32-bit), offset (64-bit)}                │
├─────────────────────────────────────────────────────────────────────┤
│  层级 3: GPU MMU + Swizzle 层                                         │
│  VA → FLA (FAMLA) 翻译 → Swizzle 地址变换 → NVLink 链路选择           │
│  → 将 leId+offset 编码到 NVLink 请求包中                              │
├─────────────────────────────────────────────────────────────────────┤
│  层级 4: NVLink 链路层                                                 │
│  128-bit flit 封装 → 信用流控 → CRC 校验 → 链路级重传                  │
│  → 将请求包传输到 NVSwitch                                            │
├─────────────────────────────────────────────────────────────────────┤
│  层级 5: NVSwitch 路由 + 地址变换层                                    │
│  入口路由表 (Map Slot) → TgtID 查找 → De-swizzle → Compaction         │
│  → 将全局地址翻译为 FABM 压缩地址 (FAMCA)                              │
├─────────────────────────────────────────────────────────────────────┤
│  层级 6: 目标 GPU / FAMM Donor GPU 层                                 │
│  FAMCA → 物理 DRAM 地址 → 执行读/写/原子操作                           │
│  → 硬件 RMW 支持，无需软件模拟                                         │
├─────────────────────────────────────────────────────────────────────┤
│  层级 7: 完成追踪 + 错误报告层                                         │
│  mbarrier 硬件 + GNIC 跨处理器通知 → 报告完成/故障状态                  │
│  → 应用通过 report predicate 和 opaque report value 检查              │
└─────────────────────────────────────────────────────────────────────┘

关键问题：leId 和 offset 在这 7 个层级中分别扮演什么角色？硬件需要哪些机制才能让这套指令从上到下 work 起来？

2. 层级 1-2：PTX 指令语义与 SM 发射

2.1 PTX Fabric 指令集

PTX ISA 9.3 Section 9.7.10 定义了 6 条 fabric 指令，分为三类：

操作指令（4 条）：

指令 语义 类比 RDMA leId 要求
fabric.try_get 从远端异步读取到本地 shared memory RDMA READ 必须 unicast
fabric.try_put 从本地 shared memory 异步写入远端 RDMA WRITE 无 .multimem：unicast；有 .multimem：multicast
fabric.try_red 向远端执行归约写入 RDMA Atomic Reduction 同 try_put
fabric.try_pullred 从 multicast 端点拉取并归约 无直接 RDMA 对应 必须 multicast

控制指令（2 条）：

指令 功能
fabric.submit 提交之前 issue 的 fabric 操作到 fabric proxy
fabric.wait 等待 shared memory 读取完成

来源：[官方文档] PTX ISA 9.3 Table of Contents, Section 9.7.10.5.1-9.7.10.5.6

2.2 操作生命周期

PTX ISA 9.3 Section 9.7.10.3 定义了 fabric 操作的 4 阶段生命周期：

阶段 1: Issue
  GPU 线程执行 fabric.try_get/put/red/pullred
  → LSU 将操作放入内部队列（类似 cp.async 的异步提交）
  → 此时操作尚未发送到 fabric

阶段 2: Submit + Arrive
  线程执行 fabric.submit → 将队列中的操作提交到 fabric proxy
  线程在 mbarrier 上 arrive + expect_tx → 声明预期传输量

阶段 3: Wait
  线程执行 fabric.wait（或 mbarrier wait）→ 阻塞等待完成
  → 硬件在后台处理 fabric 请求

阶段 4: Inspect
  检查 mbarrier 的 report predicate → 判断是否出错
  → 如果出错，读取 opaque report value
  → 通过 cudaFabricOpErrorStatusGet/Count API 解码错误详情

来源：[官方文档] PTX ISA 9.3 Section 9.7.10.3

2.3 完成机制

PTX ISA 9.3 Section 9.7.10.4 定义了三种完成机制：

  1. Fabric Reporting Mechanism ([官方文档] Section 9.7.10.4.1)：通过 mbarrier 的 report predicate 和 opaque report value 报告完成/故障状态。这是 CFT 容错能力的核心——与传统指针式 P2P 不同（NVLink 故障导致致命崩溃），fabric 操作将故障报告给应用。

  2. Fabric-read Completion Mechanism ([官方文档] Section 9.7.10.4.2)：通过读取结果本身来检测完成（用于 try_get）。

  3. Counted Completion Mechanism ([官方文档] Section 9.7.10.4.3)：使用 mbarrier 的 expected transaction count 进行计数完成。支持 .mbarrier::complete_tx::bytes（按字节计数）和 .mbarrier::complete_tx::16B（按 16 字节块计数）。

2.4 SM 发射层的硬件行为

[推断] 基于 SM 架构的通用设计模式和 cp.async 的已知行为：

当 SM 的 Load-Store Unit (LSU) 遇到 fabric 指令时，推测执行以下操作：

  1. 指令解码：识别 fabric.try_* 操作码，提取操作数：
  2. CFT Handle 寄存器对：{leId_reg, offset_reg}
  3. 源/目标 shared memory 地址
  4. 传输大小

  5. mbarrier 地址

  6. 请求构造：将 fabric 操作封装为内部请求包，包含：

  7. 操作类型（read/write/reduction）
  8. leId（32-bit）
  9. offset（64-bit）
  10. 数据载荷（对于 write/reduction）

  11. 完成标记（mbarrier 地址 + 预期计数）

  12. 异步提交：请求被放入 LSU 的内部队列，线程可以继续执行其他指令。fabric.submit 触发队列刷写（queue flush），将累积的请求提交到 GPU 的 fabric 接口。

关键设计约束：try_ 前缀表明操作可能失败——如果 fabric 无法接受请求（如队列满、链路故障），操作不会致命崩溃，而是通过 mbarrier 的 report 机制报告失败。这与 cp.async 的"try"语义一致。

3. 层级 3：GPU MMU + Swizzle 地址变换

3.1 虚拟地址到织物线性地址（VA → FAMLA）

[官方文档] CUDA Programming Guide — Virtual Memory Management 描述了地址翻译的第一步：

"The driver connects the application's virtual location to the hardware's physical location, and subsequently to the network's fabric location."

这个过程建立 VA → PA → FA (Fabric Address) 的三级映射：

应用虚拟地址 (VA)
    ↓  [GPU MMU 页表翻译]
物理地址 (PA) — 本地物理内存或 peer aperture 窗口
    ↓  [Peer Aperture PTE 重定向]
织物线性地址 (FAMLA / FLA) — 全局地址空间

Peer Aperture 机制 ([专利] US11182309B2)：

"Source processor utilizes a 'peer aperture,' functioning as a specific window into a remote memory slice."

Source GPU 的 MMU 页表中，某些 PTE 不指向本地物理内存，而是指向一个 peer aperture——一个映射到远程 FAM 内存的窗口。当 MMU 翻译命中这类 PTE 时，产生的物理地址实际上是一个 织物线性地址 (FAMLA)，标识了远程内存在全局地址空间中的位置。

[官方文档] Grace Hopper Architecture Blog 进一步确认：

"NVLink Network endpoints do not share a common memory address space" — 因此需要新的地址翻译硬件 "to isolate all GPU address spaces" from each other and the network.

[推断] FAMLA 的编码方式：FAMLA 是一个全局唯一的地址，覆盖所有 FAMM 的总内存空间。例如，6 个 FAMM 各 2 GB，则 FAMLA 范围为 0~12 GB。FAMLA 的高位标识目标 FAMM（间接对应 leId），低位标识该 FAMM 内的偏移（对应 offset）。

3.2 leId 在 MMU 层的角色

[推断] leId 在 MMU 层的作用机制有两种可能的设计：

设计 A（直接映射）：leId 直接编码在 FAMLA 地址的特定比特位中。MMU 翻译时，PTE 中已包含 leId 信息。这种设计的优点是硬件简单，缺点是 leId 空间受限于地址位宽。

设计 B（间接映射）：MMU 翻译产生 FAMLA，leId 由独立的硬件表（leId → FAMLA 基址映射）提供。Source GPU 维护一个 leId 查找表（可能位于 L2 cache 或专用 SRAM 中），将 leId 翻译为 FAMLA 基址，然后加上 offset 得到完整的 FAMLA。

[推断] 基于 CUDA Driver API 的设计（cuLogicalEndpointBindAddr 将 leId + offset 绑定到具体的 ptr），设计 B 更可能：Driver 在创建端点时，将 leId → (FAMLA 基址, 目标设备, NVSwitch Map Slot 索引) 的映射编程到硬件表中。运行时，GPU 线程只需提供 leId + offset，硬件自动查表得到 FAMLA。

3.3 Swizzle 地址变换（Source GPU 端）

[专利] US11182309B2 明确描述了 swizzle 机制：

"Source processor employs a 'swizzle' technique. By manipulating address bits via mathematical strings, the system generates entropy. This pseudo-randomly distributes requests across multiple physical links."

Swizzle 的具体实现 ([专利] US11182309B2, [论文] DySHARP arXiv:2605.05607)：

  1. 伽罗瓦域 (Galois Field) 数学变换：对 FAMLA 地址的特定比特位执行 GF 乘法。
  2. XOR 累积：将变换后的位与原始位做 XOR，产生伪随机的地址分布。
  3. 链路选择：变换后的低位用于从 k 条 NVLink 链路中选择一条（如 k=12）。

目的：使连续的地址请求分散到不同的 NVLink 链路上，实现负载均衡（"喷洒/spraying"），避免单链路过载。

[论文] DySHARP (arXiv:2605.05607) 提供了更详细的硬件描述：

"Load-Store Unit fetches destinations into a small MultimemQ before transmitting."

这表明 SM 的 LSU 内部有一个专用的 MultimemQ 队列，用于缓冲 fabric 请求，然后在发送前进行地址变换和链路选择。

3.4 Swizzle 后的 NVLink 请求包

经过 swizzle 后，请求包包含以下信息：

NVLink 请求包格式（推测）:
┌──────────────────────────────────────────────────────┐
│  Header:                                              │
│    - Packet Type (Memory Request)                     │
│    - Source GPU ID                                    │
│    - Swizzled Address (包含链路选择信息)                │
│    - Operation Type (Read/Write/Atomic/Reduction)     │
│    - leId 或编码在地址中的路由信息                      │
│    - Request Size                                     │
│    - Completion Tag (mbarrier 地址)                    │
│  Payload:                                             │
│    - Data (for write/reduction operations)            │
│    - Reduction Operator (for atomic operations)       │
│  Footer:                                              │
│    - CRC (链路级校验)                                  │
└──────────────────────────────────────────────────────┘

[论文] NVLink 包格式参考 (MDPI Future Internet 2025)：NVLink 使用 128-bit flits（flow control digits），最大包大小 256 字节，有效载荷效率 >94%。

4. 层级 4：NVLink 链路层传输

4.1 NVLink 物理层

[官方文档] 各代 NVLink 规格：

代际 产品 每 GPU 链路数 单向带宽/GPU 信令
NVLink 3 A100 12 300 GB/s PAM4
NVLink 4 H100 18 450 GB/s PAM4
NVLink 5 B200 18 900 GB/s PAM4
NVLink 5 (NVL72) GB200 18 1.8 TB/s PAM4

来源：[官方文档] NVIDIA 各代架构白皮书

4.2 链路层协议

[专利] US9954984B2 (NVIDIA) 描述了 NVLink 链路层的关键机制：

  1. 信用流控 (Credit-based Flow Control)：接收端向发送端分配信用值，表示可用的缓冲区空间。发送端每发送一个 flit 消耗一个信用，收到信用返回后恢复。这确保了不会发生缓冲区溢出。

  2. CRC 校验：每个 flit 包含 CRC 校验码。接收端验证 CRC，如果发现错误，请求重传。

  3. 重传缓冲区 (Replay Buffer)：发送端维护已发送 flit 的副本，直到收到接收端的确认。如果 CRC 失败，接收端请求重传特定 flit。

[论文] NVBleed (安全研究论文) 确认：NVLink 的传输开销极低，有效载荷效率 >94%，重传率通常 <0.1%。

4.3 链路层对 fabric 操作的影响

[推断] NVLink 链路层对 fabric 操作是透明的——它只负责可靠地传输比特流，不感知上层是 fabric 操作还是普通 P2P 操作。fabric 操作的语义（leId 路由、地址变换、完成追踪）由更高层处理。

但链路层对 fabric 操作的容错能力至关重要：

  • NVLink 故障检测：链路层通过 CRC 错误率、信用超时等机制检测链路故障。
  • 故障报告：当链路故障被检测到时，正在传输中的 fabric 操作会被标记为失败。
  • 完成报告：失败的 fabric 操作通过 mbarrier 的 report predicate 通知应用。

这实现了 CFT 的核心价值——将硬件故障报告给应用而非致命崩溃。

5. 层级 5：NVSwitch 路由 + 地址变换（核心）

5.1 NVSwitch 架构概述

[官方文档] 第三代 NVSwitch (H100) ([官方文档] NVIDIA Developer Blog)：

  • 工艺：定制 TSMC 4N 工艺，251 亿晶体管，294 mm²
  • 端口：64 个 NVLink 端口
  • 带宽：3.2 TB/s 全双工
  • 信令：50 Gbaud PAM4，每差分对 100 Gbps
  • 拓扑：中央 crossbar + 两侧端口逻辑块和加速器，128 条 PHY 通道

[官方文档] 第四代 NVSwitch (B200 NVL72)：

  • 带宽：57.6 Tbps 全双工（机架级）
  • 拓扑：非阻塞计算织物，最多 576 GPU 无阻塞连接

5.2 NVSwitch 入口路由表（Ingress Routing Table）

[专利] US11182309B2 是理解 NVSwitch 路由机制的核心文献。专利描述了 入口路由表重映射表 (Ingress Port Route Remap Table)，表中的每个条目称为一个 Map Slot。

Map Slot 字段定义 ([专利] US11182309B2)：

字段 位宽 功能
Base 可配置 地址范围的起始值，用于匹配检查
Limit 可配置 地址范围的结束值，用于匹配检查
TgtID 可配置 目标标识符 — 指定请求应路由到哪个目标（FAMM/GPU）
Shuffle Mode 可配置 控制 de-swizzle 和 compaction 的具体模式
Offset 可配置 可编程偏移量，加到压缩后的地址上

来源：[专利] US11182309B2 — "The central routing switch manages traffic via ingress tables containing 'map slots.' These entries define address boundaries and utilize software-programmable 'Target IDs' to direct traffic to specific hardware targets."

5.3 NVSwitch 路由流程

[专利] US11182309B2 描述了完整的路由流程：

NVSwitch 入口端口收到 NVLink 请求包
    ↓
步骤 1: 从请求地址中提取特定比特位作为索引
    ↓
步骤 2: 查找该入口端口对应的 Route Remap Table
    ↓
步骤 3: 找到对应的 Map Slot，验证 Base ≤ 地址 < Limit
    ↓
步骤 4: 读取 TgtID → 确定目标 FAMM/GPU
    ↓
步骤 5: 根据 Shuffle Mode 执行 De-swizzle（逆向 swizzle）
    ↓
步骤 6: 执行 Compaction（地址压缩）
    ↓
步骤 7: 加上 Offset 字段值 → 得到最终的 FAMCA 地址
    ↓
步骤 8: 将请求转发到 TgtID 指定的目标

来源：[专利] US11182309B2 — "The switch evaluates incoming physical addresses against these programmed parameters to determine the correct egress path."

5.4 leId → TgtID 映射（关键桥梁）

这是连接软件层（CUDA CFT）和硬件层（NVSwitch）的核心映射。

[专利] US11182309B2 使用 "TgtID"（Target ID）作为路由目标标识符，而 [官方文档] CUDA Driver API 使用 "leId"（Logical Endpoint ID）。两套文档使用不同术语描述同一概念。

[推断] leId → TgtID 的映射机制：

  1. Driver 层配置：当应用调用 cuLogicalEndpointCreate(leId, prop) 时，CUDA Driver 分配一个 leId，并确定该端点对应的物理目标设备。Driver 随后通过 RM Control 接口（闭源）将 leId → TgtID 的映射编程到 NVSwitch 的 Map Slot 表中。

  2. Map Slot 编程：每个 leId 对应一个或多个 Map Slot 条目。Map Slot 的 TgtID 字段指向目标设备的硬件标识符（如 NVSwitch 端口号或设备物理 ID）。

  3. 运行时查找：当 fabric 请求到达 NVSwitch 时，NVSwitch 硬件从请求地址中提取 leId 相关信息，查找 Map Slot，得到 TgtID，然后路由到目标。

关键证据：

  • [官方文档] MNNVL User Guide 确认："NVLink Subnet Manager assigns logical identifiers (LID) to the GPU and NVSwitch NVLink ports" 并 "calculates and programs the NVSwitch forwarding table"。这里的 LID 与 leId 概念相似，都是逻辑标识符。
  • [官方文档] Fabric Manager 文档确认："configures the NVSwitch memory fabric to form a single memory fabric among the participating GPUs" 并管理 "NVLink port mapping and routing between compute nodes and NVLink switches"。
  • [专利] US11182309B2 的 Map Slot 中 TgtID 字段是 "software-programmable"，说明它由驱动/Fabric Manager 在初始化时配置。

5.5 De-swizzle（NVSwitch 端）

[专利] US11182309B2 描述了 de-swizzle 机制：

"The switch must first reverse the initial entropy manipulation."

De-swizzle 是 swizzle 的逆操作：

  1. Source GPU 对地址执行了 GF 域变换 + XOR 累积。
  2. NVSwitch 根据 Map Slot 的 Shuffle Mode 字段，知道 Source GPU 使用了哪种 swizzle 模式。
  3. NVSwitch 执行逆向变换，恢复原始 FAMLA 地址。

[推断] Shuffle Mode 字段的作用：不同的 Source GPU 可能使用不同的 swizzle 模式（例如，不同代际的 GPU 或不同配置的链路数）。Shuffle Mode 告诉 NVSwitch 应该使用哪种逆向算法。这确保了 NVSwitch 可以与多种 Source GPU 兼容。

5.6 Compaction（地址压缩）

[专利] US11182309B2 的核心创新之一：

"Divides the address by the number of active links so the memory module receives consecutive, usable addresses without collisions."

为什么需要 Compaction：

Source GPU 看到的是全局的 FAMLA 地址空间（如 0~12 GB），覆盖所有 FAMM 的总容量。但单个 FAMM 只有 2 GB 物理内存。如果不做压缩，FAMM 的内存控制器需要处理 12 GB 的地址范围，其中大部分是空洞（无效地址），浪费 TLB 和页表资源。

Compaction 的数学：

假设：
  - 全局 FAMLA 空间 = 12 GB（6 个 FAMM × 2 GB）
  - 每个 FAMM 通过 L=2 条 NVLink 连接到 NVSwitch
  - Source GPU 通过 k=12 条 NVLink 连接到 NVSwitch

对于 FAMM 2（全局地址范围 8~10 GB）：
  1. De-swizzle 后恢复 FAMLA（如 8.5 GB）
  2. Compaction: FAMCA = (FAMLA - Base_of_FAMM2) / L
                = (8.5 GB - 8 GB) / 2
                = 0.25 GB
  3. 加上 Map Slot Offset: Final_addr = 0.25 GB + Offset

结果：FAMM 2 只需要处理 0~2 GB 的紧凑地址空间（FAMCA），
      而不是 8~10 GB 的全局空间。

[专利] US11182309B2 Claim 9 明确保护了 "地址变换器包含除法或压缩电路"。

5.7 NVSwitch 多播支持

[官方文档] NVSwitch 第三代 Blog 确认了 in-fabric 多播和归约能力：

"In-fabric multicast and reductions provide up to 2x throughput gain."

[论文] TokenWeave (arXiv:2505.11329) 描述了多播 fabric 操作的工作方式：

"The network fabric duplicates packets for all subscribed devices and executes in-network reduction directly inside the switch ASIC."

[推断] 对于 multicast leId 的 fabric 操作（fabric.try_put.multimem / fabric.try_red.multimem）：

  1. NVSwitch 维护一个 multicast group table，记录每个 multicast group 的成员端口列表。
  2. 当请求到达 NVSwitch 时，硬件查找 multicast group，将请求复制到所有成员端口。
  3. 对于 reduction 操作，NVSwitch 的 SHARP 引擎 在 crossbar 内执行归约（如 FP32 加法），只将最终结果写回目标内存。

[官方文档] NVSwitch 第三代 Blog 确认 SHARP 引擎的归约能力：

"Embedded ALUs and SRAM directly in the fabric execute collective operations without routing data back to GPUs."
"400 GFlops FP32 capability."

5.8 NVSwitch SHARP 引擎详解

[官方文档] NVSwitch 第三代 Blog 提供了 SHARP 引擎的技术细节：

  • 位置：直接嵌入在 NVSwitch crossbar 的数据通路中
  • 能力：逻辑运算、min/max、加法，支持 FP16/FP32/FP64/BF16/整数
  • 组管理：专用控制器同时管理最多 128 个 SHARP 组
  • 性能：400 GFlops FP32 计算能力，通信量减少从 4N 到 2N+2

[论文] DySHARP (arXiv:2605.05607) 进一步描述了 switch 内的归约硬件：

"Records the number of targets and tracks incoming partial responses to return final aggregated value."

这表明 NVSwitch 内部有 Merging Table 用于跟踪归约会话状态：

Merging Table 条目（推测）:
┌──────────────────────────────────────────────────────┐
│  Session ID:     归约操作的唯一标识                    │
│  Target Count:   预期参与的 GPU 数量                   │
│  Received Count: 已收到的部分结果数量                   │
│  Operator:       归约操作类型（add/min/max/and/or）    │
│  Data Type:      数据类型（FP32/FP64/INT32...）        │
│  Accumulator:    累积值缓冲区                          │
│  Dest Address:   最终结果的写入地址                     │
│  Completion Tag: 完成通知标记                          │
└──────────────────────────────────────────────────────┘

6. 层级 6：目标 GPU / FAMM Donor GPU 处理

6.1 Donor GPU 架构

[专利] US11182309B2 描述了 Donor GPU（Floor-swept GPU）的内部结构：

FAMM (Fabric Attached Memory Module)
┌──────────────────────────────────────────────────────┐
│  Donor GPU (Floor-swept GPU):                         │
│  ┌─────────────────────────────────────────────────┐ │
│  │  Boot ROM:          启动固件                      │ │
│  │  DDR Memory Ctrl:   硬件加速原子操作 (RMW)         │ │
│  │  Row Remapper:      坏行重映射管理                 │ │
│  │  Data Cache:        数据缓存（缓冲热点数据）        │ │
│  │  Crossbar:          内部数据通路                   │ │
│  │  Fabric Interface:  NVLink 织物接口               │ │
│  │  计算引擎 (SM):     被禁用（熔断或软件配置）        │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  离散 DRAM 阵列:     大容量内存（如 16 GB）            │
│  L 条 NVLink 链路:   连接到 NVSwitch（如 L=2）        │
└──────────────────────────────────────────────────────┘

来源：[专利] US11182309B2 Claims 1-7

6.2 请求处理流程

[专利] US11182309B2 描述了 Donor GPU 的处理流程：

"Donor controller acts strictly as a slave device."

  1. 接收压缩请求：NVSwitch 将 FAMCA 地址的请求发送到 Donor GPU。
  2. FAMCA → 物理地址翻译：Donor GPU 的内存控制器将 FAMCA 地址翻译为物理 DRAM 地址。
  3. 执行操作：
  4. Read：从 DRAM 读取数据，沿原路返回 Source GPU。
  5. Write：将数据写入 DRAM。
  6. Atomic (RMW)：硬件执行 read-modify-write，无需软件模拟。这是 Donor GPU 与普通内存控制器的关键区别。
  7. 返回结果：对于 read 和某些 atomic 操作，结果通过 NVSwitch 返回 Source GPU。

6.3 硬件原子操作支持

[专利] US11182309B2 Claim 1 的关键限定：

"该处理器被构造为在所述内存上执行至少一个 read-modify-write 原子内存访问命令。"

为什么硬件原子操作至关重要：

  • 分布式训练中的梯度归约（AllReduce）需要在多个 GPU 之间执行原子加法。
  • 如果 Donor GPU 不支持硬件原子操作，就需要 Source GPU 执行 read → modify → write 三步操作，这在远程内存场景下会导致严重的性能问题（3 倍延迟 + 一致性问题）。
  • Donor GPU 的硬件 RMW 确保原子操作的 线性一致性 (linearizability)——所有 GPU 看到的操作顺序是一致的。

[专利] US11182309B2 Claim 16 列出了 Donor GPU 上被禁用的计算能力清单：

  • 浮点原子加法（全局和共享内存）
  • Warp vote 和 ballot 函数
  • 内存围栏函数
  • 同步函数
  • Surface 函数
  • 3D 线程块网格
  • 漏斗移位
  • 动态并行
  • 半精度浮点运算
  • 加减乘比较、warp shuffle、转换
  • 张量核心 (Tensor Core)

这份清单实质上列出了 CUDA GPU 的主要计算特性，明确了 Donor GPU 只保留内存控制和 fabric 接口能力。

6.4 Data Cache 和 Row Remapper

[专利] US11182309B2 还描述了两个重要的辅助机制：

Data Cache：Donor GPU 包含数据缓存，用于缓冲热点数据。对于频繁访问的远程内存区域，Data Cache 可以减少 DRAM 访问延迟。

Row Remapper：内存坏行重映射器。当 DRAM 中出现坏行时，Row Remapper 将坏行的地址重映射到备用行，确保应用无需感知硬件缺陷。这与 Floor-swept GPU 的理念一致——即使是"残次品"也能可靠工作。

7. 层级 7：完成追踪与错误报告

7.1 mbarrier 硬件机制

[推断] 基于 PTX ISA 对 mbarrier 的描述和 cp.async 的已知行为：

mbarrier（memory barrier）是一个硬件同步原语，位于 shared memory 中。对于 fabric 操作，mbarrier 承担以下职责：

  1. 预期计数 (Expected Transaction Count)：线程在 arrive 时声明预期的传输量（expect_tx）。硬件维护一个计数器，初始值为 0，每次完成一个 fabric 操作时递增。

  2. 完成检测：当计数器达到预期值时，mbarrier 触发，唤醒等待的线程。

  3. Report Predicate：如果任何 fabric 操作失败，mbarrier 的 report predicate 被设置为 true。线程可以检查这个 predicate 来判断是否有错误。

  4. Opaque Report Value：错误发生时，硬件将错误详情编码到 opaque report value 中。应用通过 cudaFabricOpErrorStatusGet/Count API 解码。

[官方文档] PTX ISA 9.3 Section 9.7.10.4.1：

"If the operation succeeds, the mbarrier report predicate is not modified... Otherwise, the operation fails, the mbarrier report predicate is set, and the opaque report value may contain more information, which may be decoded via the cudaFabricOpErrorStatusGet/Count APIs."

7.2 跨处理器完成通知

[专利] US20230289242A1 (NVIDIA) 描述了跨处理器通知的硬件机制：

  • GNIC (GPU Network Interface Controller)：负责在 fabric 操作完成时向 Source GPU 发送完成通知。
  • 通知合并 (Notification Coalescing)：GNIC 将多个完成通知合并为一个，减少中断开销。
  • 零平衡计数器 (Zero-balanced Counter)：mbarrier 使用零平衡计数器——初始值为 0，每次完成递增，达到预期值时触发。

7.3 错误类型与恢复

[官方文档] CUDA Runtime API 定义了 fabric 操作错误报告：

API 功能
cudaFabricOpErrorStatusCount 返回 fabric 操作的错误总数
cudaFabricOpErrorStatusGet 提取具体错误详情到 cudaFabricOpStatusInfo 结构

[推断] 可能的错误类型：

  1. NVLink 链路故障：物理链路断开或不可用。
  2. NVSwitch 路由错误：Map Slot 查找失败（无效的 leId 或地址范围）。
  3. 目标设备故障：Donor GPU 无响应。
  4. 超时：fabric 操作在规定时间内未完成。
  5. 一致性错误：多播归约中的参与者数量不匹配。

恢复策略（由应用决定）：

  • 重试：暂时性故障（如链路瞬断）可以重试。
  • 降级：永久故障可以绕过故障设备，使用剩余设备继续计算。
  • 中止：严重故障可以中止整个计算任务。

8. 软件基础设施：Driver、IMEX 与 Fabric Manager

8.1 CUDA Driver 层的职责

[官方文档] CUDA Driver API — Logical Endpoint 定义了 12 个 API 函数。Driver 层承担以下关键职责：

  1. leId 管理：cuLogicalEndpointIdReserve/Release 分配和回收 leId 空间。

  2. 端点创建：cuLogicalEndpointCreate 根据属性创建端点，内部创建端点数据结构。

  3. 资源绑定：cuLogicalEndpointBindAddr/BindMem 建立 (leId, offset) → (device, physical_memory) 的映射。

  4. 硬件编程：[推断] Driver 通过 RM Control 接口将 leId → TgtID 映射编程到 NVSwitch Map Slot 表，配置 Source GPU 的 leId 查找表，设置 Donor GPU 的地址翻译参数。

  5. IPC 导出/导入：cuLogicalEndpointExport/Import 序列化/反序列化 leId 和绑定信息为 CUlogicalEndpointFabricHandle。

  6. 就绪查询：cuLogicalEndpointQuery 检查端点是否已完成构造（所有远端 GPU 已同步元数据）。

8.2 IMEX Daemon（跨进程/跨节点协调）

[官方文档] NVIDIA IMEX (Import/Export Service) Guide：

  • 角色："A privileged entity capable of communicating across OS/Node domains" — 跨 OS/节点域协调内存映射。
  • 工作机制：当节点 A 分配内存时，创建 "Virtual Address (VA) to Physical Address (PA) to Fabric Address (FA) mapping"。IMEX daemon 将此映射安全地传播到其他节点。
  • 通信方式：使用 TCP/IP 和 gRPC 进行节点间控制平面通信。
  • 安全隔离：通过 nvidia-caps-imex-channels 字符设备提供用户级内存隔离。

[官方文档] IMEX Channels Guide：

"User-based memory isolation in a multi-user environment."

IMEX channel 是 GPU 驱动特性，提供 /dev/nvidia-caps-imex-channels/channelN 字符设备。只有被授权访问同一 IMEX channel 的进程才能安全地共享 fabric 内存。

[开源代码] NVIDIA open-gpu-kernel-modules 中的 IMEX 相关参数：

NVreg_ImexChannelCount:    // IMEX channel 数量，默认 2048，最大 2^20
NVreg_CreateImexChannel0:  // 自动创建 channel 0（仅用于可信环境）

来源：[开源代码] kernel-open/nvidia/nv-reg.h

8.3 Fabric Manager 与 NVLink Subnet Manager

[官方文档] MNNVL User Guide：

  • Fabric Manager (FM)："configures the NVSwitch memory fabric to form a single memory fabric among the participating GPUs" — 配置 NVSwitch 内存织物，管理 NVLink 端口映射和路由。

  • NVLink Subnet Manager (NVLSM)："assigns logical identifiers (LID) to the GPU and NVSwitch NVLink ports" 并 "calculates and programs the NVSwitch forwarding table" — 分配逻辑标识符，计算和编程 NVSwitch 转发表。

[推断] Fabric Manager 和 NVLSM 在 fabric instruction 工作流中的角色：

  1. 初始化阶段：FM/NVLSM 发现 NVLink 拓扑，为每个 GPU 和 NVSwitch 端口分配 LID。
  2. 路由表计算：NVLSM 根据拓扑计算 NVSwitch 的转发表（包括 Map Slot 的 Base/Limit/TgtID/Shuffle Mode/Offset 字段）。
  3. 路由表编程：FM 通过 RM Control 接口将转发表编程到 NVSwitch 硬件中。
  4. 动态更新：当拓扑发生变化（如 GPU 故障、链路降级）时，FM/NVLSM 重新计算和编程路由表。

8.4 fence.proxy 指令

[官方文档] PTX ISA 9.3 定义了 3 条 fence.proxy 指令：

fence.proxy.generic::fabric.alias.release.sys;   // 普通内存 → fabric 可见
fence.proxy.fabric::generic.alias.acquire.sys;   // fabric → 普通内存可见
fence.proxy.fabric::fabric.alias.acquire.sys;    // fabric 内部排序

[论文] "Mixed-Proxy Extensions for the NVIDIA PTX Memory Consistency Model" (ISCA 2022) 解释了 mixed-proxy 内存一致性模型：

"The fabric proxy is a memory ordering concept."

[推断] fence.proxy 的作用：fabric 操作和普通内存操作位于不同的 代理域 (proxy domain)。fence.proxy 确保跨代理域的内存可见性顺序：

  • generic → fabric：确保在 fabric 操作之前，普通内存的写入对 fabric 可见。
  • fabric → generic：确保在普通内存操作之前，fabric 操作的写入对普通内存可见。
  • fabric → fabric：确保 fabric 操作之间的顺序。

这对于保证数据一致性至关重要——例如，在通过 fabric 发送数据之前，必须确保数据已经写入 shared memory。

9. 多播 Fabric 操作的硬件路径

9.1 Multicast vs Unicast 的区别

[官方文档] CUDA Driver API 定义了两种端点类型：

类型 路由行为 硬件机制
CU_LOGICAL_ENDPOINT_TYPE_UNICAST 路由到单个 GPU 标准 NVSwitch 单播路由
CU_LOGICAL_ENDPOINT_TYPE_MULTICAST 路由到多个 GPU NVSwitch 多播复制 + SHARP 归约

9.2 Multicast 硬件路径

[专利] US12326820B2 (Multicast and Reflective Memory) 描述了多播内存行为的硬件机制：

"将单个虚拟地址映射到多个物理地址并传播访问请求。"

[论文] MSCCL++ (arXiv:2504.09014) 描述了多播 fabric 操作的工作方式：

"Switch-Mapped I/O: Accelerator kernels utilize specialized PTX commands to interact with the network fabric. During aggregation, instruction fetches the values from all the virtual memory addresses pointed by the multimem address to the switch for hardware-level reduction."

[推断] 多播 fabric 操作的完整硬件路径：

1. Source GPU 执行 fabric.try_put.multimem [dstLeId, dstOffset], [src], size
   → leId 标识一个 multicast group

2. NVSwitch 收到请求，查找 multicast group table
   → 获取成员端口列表 [port_0, port_1, ..., port_N]

3. 对于普通 put（无归约）：
   → NVSwitch crossbar 将数据复制到所有成员端口
   → 每个目标 GPU 的 Donor GPU 将数据写入本地内存

4. 对于 reduction（try_red.multimem）：
   → NVSwitch SHARP 引擎从所有成员收集数据
   → 在 switch 内执行归约（如 FP32 加法）
   → 将最终结果写入目标内存

5. 对于 pull-reduction（try_pullred）：
   → Source GPU 发起拉取请求
   → NVSwitch 从所有成员收集数据
   → 在 switch 内执行归约
   → 将结果返回 Source GPU

9.3 NVSHMEM 中的 multimem 指令

[论文] NCCL NVLink SHARP 源码分析 (CSDN) 提供了 multimem 指令的实际使用方式：

// 从所有 multicast 成员加载并归约
multimem.ld_reduce.relaxed.sys.global.add.v4.f32
    [dst], [multimem_addr];
// → Switch 收集所有成员的值，执行加法，返回结果

// 向所有 multicast 成员存储（广播）
multimem.st.relaxed.sys.global.v4.f32
    [multimem_addr], [src];
// → Switch 将数据复制到所有成员

// 向所有 multicast 成员执行归约
multimem.red.relaxed.sys.global.add.v4.f32
    [multimem_addr], [src];
// → Switch 将 src 与所有成员的当前值做加法

来源：[论文] CSDN NCCL NVLink SHARP 源码分析

注意：multimem.* 指令是 PTX ISA 9.7.9 定义的（Section 9.7.9.13-9.7.9.26），与 Section 9.7.10 的 fabric.try_* 指令是不同的指令集。multimem.* 用于 NVLink SHARP (NVLS) 操作，fabric.try_* 用于更通用的 CFT 操作。两者共享底层硬件路径（NVSwitch crossbar + SHARP 引擎），但语义和完成机制不同。

  • 硬件隔离：新的地址翻译硬件隔离所有 GPU 地址空间

11.3 Blackwell (B200/GB200) — CFT 指令集引入

[官方文档] Blackwell 架构引入：

  • NVSwitch 4：72 端口，57.6 Tbps 全双工（NVL72 机架级）
  • NVLink 5：每 GPU 1.8 TB/s 双向带宽
  • PTX ISA 9.3 / CUDA 13.3：引入 fabric.try_* 指令集
  • sm_100：要求 Blackwell 数据中心架构
  • CFT 编程模型：从容错 fabric 操作到通用远程内存访问

[推断] Blackwell 相比 Hopper 的关键硬件变化：

  1. Source GPU 端：LSU 增加了 fabric 指令解码和执行能力（MultimemQ 扩展），MMU 增加了 leId 查找表支持。
  2. NVSwitch 端：路由表扩展支持 leId 路由（不仅仅是物理地址路由），SHARP 引擎增强支持更多数据类型和归约操作。
  3. 完成追踪：mbarrier 硬件扩展支持 fabric report predicate 和 opaque report value。

11.4 Rubin (R100) — 下一代展望

[官方文档] NVIDIA 路线图：

  • NVSwitch 5：36 物理端口，448G SerDes
  • NVLink 6：更高带宽
  • 预期：进一步扩展 fabric instruction 能力，可能支持跨节点 fabric 操作（目前仅限 NVLink 域内）

12.3 关键发现

  1. 专利先于 API 6 年：US11182309B2 的优先权日（2019-11-04）比 CFT 编程模型的推出（2025）早 6 年。专利描述的硬件架构是 CFT 的基础。

  2. leId → TgtID 是核心桥梁：软件的 leId 和硬件的 TgtID 是同一概念在不同抽象层的表达。Driver 通过 RM Control 接口将 leId 映射编程到 NVSwitch Map Slot 的 TgtID 字段。

  3. offset 经过完整的地址变换流水线：offset → FAMLA → Swizzled Address → (NVSwitch) → De-swizzled → Compacted (FAMCA) → + Offset → 物理地址。

  4. 没有专利显式使用 "CFT" 或 "Compute Fabric Transport"：CFT 是较新的软件 API 层。它依赖的硬件机制被已有专利覆盖，但 CFT 级别的抽象（fabric handle 作为软件对象、CFT 指令编码）可能是作为硬件原语之上的实现，尚未找到单独的专利覆盖。

  5. 容错是 CFT 的核心价值：传统指针式 P2P 在 NVLink 故障时致命崩溃，CFT 通过 mbarrier report 机制将故障报告给应用。这要求从 NVLink 链路层到 mbarrier 硬件的完整错误传播路径。
