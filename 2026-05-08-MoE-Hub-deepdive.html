论文概览

核心贡献

本文提出 MoE-Hub，一种在 GPU Hub 中构建的新型硬件加速通信架构，专门解决 MoE (Mixture of Experts) 模型在多 GPU 系统中的计算-通信重叠问题。核心创新包括：(1) 目的地无关范式 (Destination-Agnostic Paradigm)，通过新型 st.rowsp 指令实现 token 发送与目标地址解耦；(2) 三个专用硬件模块 —— AAU (地址分配单元)、RPM (运行时包管理器)、DAM (数据可用性管理器)，共同实现 token 数据的硬件级通信调度；(3) 用事件触发的 dispatch 替代软件忙等待轮询，消除软件中介开销。实验表明 MoE-Hub 在 8x H800 GPU 上实现了 1.40x-3.08x 每层加速 和 1.21x-1.98x 端到端加速，达到理论理想加速的 96.8%。

MoE 模型与通信瓶颈

Mixture of Experts (MoE) 架构通过稀疏激活策略显著扩大了模型容量，同时保持合理的计算成本。然而，MoE 的 动态 token 路由 特性在分布式推理/训练中引入了严重的 GPU 间通信挑战。每个 token 根据门控网络动态选择目标专家（可能位于不同 GPU），导致通信模式高度不规则且不可预测，传统通信库（如 NCCL）难以高效处理这种动态 All-to-All 通信。

研究动机

MoE 通信瓶颈

MoE 推理/训练包含两个核心通信阶段：dispatch（将 token 从生产者 GPU 发送到拥有目标专家的消费 GPU）和 combine（将专家处理后的输出聚合回原始生产者 GPU）。这两个阶段都需要跨 GPU 的 All-to-All 通信。在 MoE 模型中，通信开销可占总延迟的 40% 以上，成为关键性能瓶颈。

动态 Token 路由 vs 静态内存不匹配

MoE 的核心特征是每个 token 经过门控网络 (gate) 动态 路由到 K 个专家（Top-K）。这导致：

  • • 每个 token 的目标 GPU 是运行时决定的，编译时不可知
  • • 不同 GPU 接收的 token 数量极不均衡（token imbalance）
  • • 每个 expert 的输入 buffer 大小动态变化
  • • 传统静态内存分配无法适应这种动态性

软件中介开销

现有 MoE 通信系统（Megatron-TE, FasterMoE, Tutel, Comet, CCFuser）依赖软件层处理动态 token 路由和通信调度，引入多重开销：

  • • 目标地址解析: 每个 token 需要查找目标 GPU 地址
  • • 忙等待轮询: 等待数据就绪的软件轮询消耗 GPU 周期
  • • 内核启动开销: 频繁的 kernel launch 造成 pipeline bubbles
  • • 软件同步: CPU-GPU 同步延迟（~1us RTT）
  • • 包合并开销: 软件层面的数据打包效率低下

核心洞察

将 MoE 通信中 重复、规律 的底层操作（地址分配、包管理、依赖跟踪）从软件层卸载到 GPU 硬件中，可以消除软件中介开销，同时保持对动态 token 路由的灵活性。关键是在硬件设计中找到 "规律中的规律"——尽管 token 目标是动态的，但通信的物理操作模式是可预测和可硬件化的。

MoE 通信挑战

Dispatch 阶段

生产者 GPU 将 token 发送到拥有目标专家的消费 GPU。

  • • 每个 token 有动态目标（门控网络决定）
  • • 需要为每个目标 expert 分配输入 buffer
  • • token 数量 per expert 不可预测
  • • 传统方案：先 gather 路由信息，再发送数据

Combine 阶段

消费 GPU 将 expert 处理后的 token 输出发送回原始生产者 GPU。

  • • 目标地址已知（反向于 dispatch）
  • • 需要路由元数据来重建 token 原始顺序
  • • 非关键路径数据可使用 .nop 后缀标记
  • • 同样受 token imbalance 影响

Token 路由不规则性与 Pipeline Bubbles

MoE 的动态 token 路由导致以下关键挑战：

Token Imbalance

不同 expert 接收的 token 数量方差可达 0.05（标准差）。某些 expert 可能接收大量 token 而另一些几乎为空，导致通信负载严重不均。

Pipeline Bubbles

软件层需要等待路由信息确定后才能开始数据传输，计算 kernel 与通信 kernel 之间存在等待间隙，形成 pipeline bubble，降低 GPU 利用率。

地址解析延迟

每个 token 发送前需要在软件中查找目标 GPU 的基地址和偏移，增加 per-token 延迟，在大规模 MoE 模型中尤为显著。

依赖管理开销

需要等待所有 token 到达后才能开始 expert 计算。软件轮询方式消耗大量 GPU cycle，且无法精确感知数据到达。

设计哲学

MoE-Hub 采用 "目的地无关" (Destination-Agnostic) 设计理念：发送 token 时不需要知道目标地址，而是通过 MallocID 标识目标 expert，由硬件自动解析和分配地址。这一范式从根本上解耦了 "发送什么" 与 "发送到哪里"，消除了软件层的目标地址解析开销。

目的地无关范式 (Destination-Agnostic Paradigm)

核心思想

传统的 store 指令需要指定完整的目标内存地址。MoE-Hub 提出 st.rowsp (Row Store Packet) 指令，发送数据时只需指定一个逻辑标识符（MallocID），而非物理地址。硬件负责将 MallocID 解析为实际的物理地址，实现发送与目标地址的完全解耦。

解耦的意义

目的地无关范式将 通信意图（发送 token 到哪个 expert）与 通信执行（写入哪个物理地址）解耦。发送方不需要知道：(1) 目标 GPU 的基地址；(2) expert 输入 buffer 的偏移；(3) 当前写入位置。这些信息全部由消费侧的 AAU 和 RPM 在硬件中处理。这消除了传统方案中每个 token 都需要查找目标地址的软件开销。

地址分配单元 AAU (Address Allocation Unit)

功能概述

AAU 位于消费侧 GPU 的 Hub 中，负责自动为到达的 token 数据分配物理地址。它维护两个核心数据结构：RAT (Row Allocation Table) 和 APT (Allocation Pointer Table)，实现无软件参与的地址分配。

AAU 内部结构

RAT (Row Allocation Table)

RAT 是一个 16-bank 双端口 SRAM 缓存，存储 (MallocID, RowID) 到 LocalRowID 的映射。当 st.rowsp 指令到达时，AAU 查找 RAT 获取 LocalRowID，然后结合 RowOffset 计算物理地址。

RAT 条目结构 (Key-Value 映射)
Key
(MallocID, RowID)
Value
LocalRowID
查找: input (MallocID + RowID) → output LocalRowID
地址计算: Addr = BaseAddr + LocalRowID × RowSize + RowOffset

RAT 是 AAU 中最大的硬件组件，占 MoE-Hub 面积的绝大部分。

APT (Allocation Pointer Table)

APT 跟踪每个内存区域的下一个可用行 (row)。使用 原子指针 (atomic pointer) 实现并发分配：多个 st.rowsp 指令可以同时请求地址分配而不会冲突。

地址分配完整流程

1

接收 st.rowsp 指令: 从网络收到带有 (RowID, RowOffset, MallocID) 的写入请求。

2

RAT 查找: 使用 (MallocID, RowID) 作为键查找 RAT。

3

RAT 未命中 (Miss): 如果 RAT 中没有对应条目，使用 APT 分配新的 LocalRowID。APT 对 MallocID 对应的区域原子递增 next_row_ptr，并将新映射 (MallocID, RowID) -> LocalRowID 插入 RAT。如果 RAT 已满，按 FIFO 策略淘汰最旧条目。

4

物理地址计算: PhysicalAddr = BaseAddr + LocalRowID * RowSize + RowOffset

5

写入 HBM: 将数据写入计算得到的物理地址，通知 DAM 数据已到达。

AAU 的关键设计

AAU 的 16-bank 双端口 SRAM 设计确保了高并发地址分配能力。16 个 bank 允许同时处理 16 个独立的地址分配请求而不会产生 bank 冲突。FIFO 淘汰策略简单高效，适合 MoE 通信中 "写入后不再访问" 的访问模式。

运行时包管理器 RPM (Runtime Packet Manager)

功能概述

RPM 位于生产者侧 GPU 的 Hub 中，负责优化网络传输。它将多个小型写入请求合并为缓存行对齐 (cache-line-aligned) 的数据包，提高网络带宽利用率，并通过轮转调度 (round-robin) 管理多目标传输。

RPM 内部结构

Remote Write Buffer Pool

一组缓存行大小的缓冲区，临时存储待发送的写入数据。当 st.rowsp 指令发出时，数据首先写入 Buffer Pool 中的对应位置，而非立即发送。

Buffer 数量影响并发度：更多 buffer 允许更多并发写入。

缓存行对齐包合并

RPM 通过匹配 (RowID, RowOffset) 元组来合并来自多个 st.rowsp 请求的数据。属于同一 Row 且偏移连续的写入被合并到同一个缓存行大小的数据包中，减少网络包数量。

合并后发送一个完整缓存行而非多个小包。

包调度器

使用 轮转调度 (Round-Robin) 在多个目标 GPU 之间公平分配带宽。支持 消费者感知排序 (consumer-aware ordering)，确保接收端的数据到达顺序符合依赖要求。

兼顾公平性和数据依赖性。

包合并机制详解

在 MoE 场景中，单个 token 数据通常远小于一个缓存行（64 字节）。如果不合并，每个 token 都会产生一个独立网络包，导致大量包头部开销和低带宽利用率。

轮转调度与消费者感知排序

RPM 的包调度器在多个目标 GPU 之间使用轮转策略分配网络带宽，确保没有单个目标饿死。同时，它考虑消费侧的依赖需求：消费者感知排序 确保关键路径数据（非 .nop 标记的数据）优先发送，而非关键路径数据可以在带宽空闲时发送。

数据可用性管理器 DAM (Data Availability Manager)

功能概述

DAM 位于消费侧 GPU 的 Hub 中，负责跟踪数据到达状态并在所有依赖数据就绪时自动触发 expert 计算。它用 事件触发 机制替代了传统的 忙等待轮询，显著减少 GPU 资源浪费。

DAM 内部结构

依赖表 (CAM Structure)

DAM 使用 CAM (Content-Addressable Memory) 结构的依赖表来跟踪每个 expert 的数据到达状态。CAM 允许通过内容（而非地址）快速查找条目，适合动态 token 路由的不可预测模式。

事件触发 Dispatch vs 忙等待轮询

传统忙等待轮询
  • • GPU 持续轮询内存，检查数据是否到达
  • • 轮询消耗大量 GPU 周期和内存带宽
  • • 轮询间隔需要调优（太短浪费资源，太长增加延迟）
  • • 与计算 kernel 竞争资源
  • • 无法精确感知数据到达时刻

  • • 硬件自动跟踪数据到达状态

  • • GPU 无需轮询，可继续其他工作

  • • 精确的数据到达感知

通信工作流 (Communication Workflow)

Dispatch 阶段工作流

Dispatch 阶段将 token 从生产者 GPU 发送到拥有目标专家的消费 GPU。MoE-Hub 通过硬件加速消除了软件中介的多个步骤。

1

预分配 expert buffer: 调用 rowspMalloc 为每个 expert 预分配输入 buffer，获取 MallocID。

2

执行门控计算: GPU 运行门控网络，为每个 token 确定 Top-K 目标 expert。

3

发射 st.rowsp 指令: 对每个 token，发射 st.rowsp 指令将 token 数据发送出去。指令携带 (RowID, RowOffset, MallocID)，无需物理地址。

4

RPM 包合并: 生产者侧 RPM 将多个 st.rowsp 请求合并为缓存行对齐的数据包，通过轮转调度发送到网络。

5

AAU 地址分配: 消费侧 AAU 接收数据，通过 RAT/APT 分配物理地址，写入 HBM。

6

DAM 依赖跟踪: 消费侧 DAM 跟踪数据到达，当所有 token 到达后自动触发 expert 计算。

7

Expert 计算: GPU 启动 expert 计算 kernel，处理 token 数据。

Combine 阶段工作流

Combine 阶段将 expert 处理后的输出发送回原始生产者 GPU。与 dispatch 不同，combine 阶段的目标地址已知（反向于 dispatch），且部分数据（如 padding）可以使用 .nop 后缀标记为非关键路径。

.nop 后缀 告诉 DAM 不为此数据触发依赖事件 — 避免无意义的触发，减少不必要的等待

.nop 后缀的意义

.nop 后缀是 MoE-Hub 针对 combine 阶段优化的关键设计。Combine 阶段经常需要发送 padding 数据来对齐 token 布局，这些 padding 不需要触发任何依赖事件。标记为 .nop 后，DAM 忽略这些数据，避免不必要的依赖计数增加和提前触发。

Token 不平衡鲁棒性

MoE-Hub 在高 token 不平衡场景下（标准差最高达 0.05）仍保持显著的性能优势。这表明硬件加速方案对 MoE 的动态路由模式具有良好的适应性。

可扩展性

MoE-Hub 在 2-16 GPU 配置下展现出 近线性 FLOPS 扩展。这意味着随着 GPU 数量增加，计算效率不会因通信瓶颈而下降，证明硬件加速通信有效地解决了 MoE 的扩展问题。

96.8% 理想加速达成率

MoE-Hub 实现了理论理想加速比（计算与通信完全重叠）的 96.8%。这意味着剩余的 3.2% 延迟来自不可避免的物理限制（如网络 RTT、数据传输最小粒度等），而非架构设计缺陷。这一达成率显著优于所有基线方案。

面积开销分解

MoE-Hub 的总硬件面积为 0.49 mm²（在 TSMC 7nm 工艺下），仅占 H800 GPU 总芯片面积（约 814 mm²）的 0.06%。这是一个极小的面积投入。

模块 主要组件 面积占比
AAU 16-bank 双端口 SRAM (RAT) + APT + FIFO 主导 (最大)
RPM Remote Write Buffer Pool + 合并逻辑 + 调度器 中等
DAM CAM 依赖表 + TB 计数器 + 完成检测器 较小

RAT 是最大的硬件组件

RAT (Row Allocation Table) 作为 16-bank 双端口 SRAM，是 MoE-Hub 中面积最大的组件。这是因为 RAT 需要缓存大量 (MallocID, RowID) 到 LocalRowID 的映射，以支持高并发的地址分配。16 个 bank 的设计确保了并发地址分配时的 bank 冲突最小化。

性价比分析

仅用 0.06% 的芯片面积（相当于 H800 上不到一个 SM 的面积），MoE-Hub 就获得了 1.40x-3.08x 的每层加速和 1.21x-1.98x 的端到端加速。这意味着每平方毫米硬件面积带来的性能收益极高，是一次非常高效的硬件投资。

影响分析

GPU 架构设计新范式

MoE-Hub 证明了在 GPU Hub 中添加专用通信加速模块是可行且高效的。仅 0.06% 的面积开销就获得显著的通信加速，这为未来 GPU 架构设计提供了新方向：将更多重复性通信操作硬件化，而非依赖软件库优化。

MoE 推理/训练加速

随着 MoE 模型（Mixtral, Qwen-MoE, DeepSeek-MoE 等）的广泛应用，MoE-Hub 提供的硬件级通信加速可以直接提升这些模型的推理吞吐和训练效率。最高 1.98x 端到端加速意味着相同的硬件可以处理近两倍的 MoE 工作负载。

软件简化

目的地无关范式极大简化了 MoE 通信软件栈。开发者无需手动处理目标地址解析、buffer 分配、包合并等复杂操作，只需发射 st.rowsp 指令。这降低了 MoE 系统开发门槛，减少了软件 bug 和维护成本。

可扩展性突破

近线性的 FLOPS 扩展（2-16 GPU）意味着 MoE-Hub 解决了 MoE 模型在大规模分布式系统中的通信瓶颈。这为扩展到更多 GPU（32, 64, 128+）提供了可能，支持更大规模的 MoE 模型训练和推理。

与现有工作的互补

MoE-Hub 与 FlashOverlap（信号驱动重叠）、T3（软硬件协同跟踪）、C3（DMA 卸载）等工作互补。MoE-Hub 专注于 MoE 特有的动态 token 路由问题，提供了硬件级的目的地无关通信抽象。未来可与其他计算通信重叠技术结合，实现全栈优化。

工业落地潜力

MoE-Hub 的硬件开销极小（0.06% 芯片面积），且仅需一条新指令和 MMIO API 支持，对现有 GPU 架构改动有限。这使得 GPU 厂商（NVIDIA, AMD, 华为等）在下一代 GPU 中集成 MoE-Hub 类似功能成为可能，为 AI 加速芯片提供差异化竞争力。
