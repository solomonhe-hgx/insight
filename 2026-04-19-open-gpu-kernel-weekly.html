MMU / UVM

▼

Issue #1108: nvidia_uvm 重 I/O 下内核崩溃 严重Bug

用户在 RTX 4050 Laptop GPU + Arch Linux + 内核 6.18.22-1-lts + 驱动 595.58.03 环境下，报告 nvidia_uvm 模块在持续顺序I/O（tar压缩约30GB）或并行C++编译时一致性地触发内核恐慌。

临时方案：在操作前卸载 nvidia_uvm 模块可完全避免崩溃。

涉及代码路径：kernel-open/nvidia-uvm/ 核心UVM模块。

GitHub Issue #1108

Issue #1107: nvidia_vma_access 空指针崩溃 严重Bug

在驱动 595.58.03 + 内核 6.18.21（Debian 13）+ RTX A6000（8台受影响）上，mmap_context 被初始化为 NULL 后立即解引用，导致空指针崩溃。触发条件为打开 /proc/PID/mem 并读取 GPU 内存映射地址偏移。

涉及代码路径：kernel-open/nvidia/nv.c 中 nvidia_vma_access() 函数。

GitHub Issue #1107

Issue #1111: Blackwell GSP 固件停止响应 严重Bug

RTX Pro 6000（Blackwell, sm_120）在持续零间隙 llama.cpp 推理时，GSP 固件完全挂起，无任何恢复机制。这是 Blackwell 架构特有的稳定性问题。

涉及代码路径：src/nvidia/src/kernel/gpu/mmu/ 中的 GSP RPC 与 MMU fault 处理。

GitHub Issue #1111

Issue #1106: Arch Linux 主机完全冻结 Bug

Arch Linux + nvidia-open-dkms 595.58.03 在 Plex Docker GPU 工作负载下发生完全系统冻结。可能与 UVM 模块的内存映射处理有关。

GitHub Issue #1106

Issue #1033: 关机时内核恐慌（持续活跃） Bug

驱动 590.48.01 在关机时因 nvkms-kapi.h:struct 的不安全使用触发内核恐慌。社区成员提出禁用内核地址随机化作为临时修复。

GitHub Issue #1033

VRAM 耗尽时无系统 RAM 回退 社区反馈

当显存耗尽时，Linux 驱动不会像 Windows 那样回退到系统 RAM（缺少 GTT 功能），导致应用直接崩溃。影响 RTX 4060 Laptop、GTX 1050 Ti、GeForce 750 Ti、RTX 30 系列。官方尚无修复计划。

趋势展望

▼

Blackwell 稳定性是最大痛点

RTX Pro 6000 (sm_120) 在持续 AI 推理负载下 GSP 固件完全挂起，且 Blackwell GPU 必须依赖 open kernel modules（闭源不支持）。这意味着 NVIDIA 在 Blackwell 的开源驱动成熟度上仍有明显差距，尤其是 MMU fault 处理和 GSP RPC 资源管理方面。

内核 6.19 兼容性是关键风险

590 分支已宣布 EOL，但 595 分支在 6.19 内核上仍有构建问题。Ubuntu、Fedora、Debian 三大主流发行版均受影响，可能延迟用户升级最新内核的进度。NVIDIA 需要尽快发布 595 热修复版本。

Open Kernel 正在成为事实标准

Arch Linux 已默认切换到 open kernel modules，Blackwell 强制要求开源驱动，Canonical 计划集成 DOCA-OFED。开源驱动正在从"备选"变为"默认"，这对 NVIDIA 的代码质量和社区响应速度提出了更高要求。

NVLink/NVSwitch 代码沉寂需关注

作为数据中心 GPU 互联的核心模块，NVLink 和 NVSwitch 本周在开源仓库中零活动。考虑到 GTC 2026 已展示 Vera Rubin NVL72 和 NVLink Fusion，开源代码可能滞后于硬件发布，需关注后续是否会推送大规模驱动更新。

本周观察

MMU/UVM 稳定性危机叠加 Blackwell 固件问题，使 open-gpu-kernel-modules 在 AI 推理场景下的可靠性面临严峻考验。与此同时，NVLink/NVSwitch 开源代码的沉默与 GTC 2026 展示的 Rubin NVL72 形成鲜明对比——硬件已发布但驱动未同步，暗示下一波大规模开源更新可能即将到来。
