1. 概述：什么是 CUDA CFT 编程模型

CUDA Compute Fabric Transport (CUDA CFT) 是 PTX ISA 9.3 引入的全新编程模型，它允许 GPU 线程通过 NVLink 直接对远端 GPU 的内存执行读写和归约操作，无需 CPU 代理参与。这是 NVIDIA 在 Blackwell 架构 (sm_100) 上实现的 GPU-Resident Networking 核心能力。

传统的 GPU 间通信（如 IBGDA、GPUNetIO）虽然绕过了 CPU 的数据路径，但仍需 CPU 参与控制路径（QP 创建、内存注册等）。CFT 模型则更进一步：GPU 线程直接通过 PTX 指令发起 NVLink 上的 RDMA 操作，实现了完全的 GPU-Resident 通信。

CFT 模型的核心创新在于：

故障弹性 (Resilience)

与基于指针的 NVLink 操作（失败即致命）不同，Fabric 操作可以向应用报告失败，使应用能够做出响应 — 包括中止、清理、或重试。这是 CFT 模型引入的首要设计目标。

异步执行模型

所有 Fabric 操作都是异步的，在后台推进。线程发出操作后可以继续执行其他工作，通过 mbarrier 机制追踪完成状态。

PTX ISA 原文定义

9.7.10. Fabric Instructions

NVLink failures encountered by pointer-address-based memory operations
are fatal to the application.

The CUDA Compute Fabric Transport (CUDA CFT) programming model
introduces fabric operations that report failures directly to the
application, enabling them to be resilient over NVLink failures.

How to handle these failures is entirely up to the application and
may include aborting or cleanly tearing the process down, or
attempting to recover from transient failures by retrying the
operation.

来源：PTX ISA 9.3, Section 9.7.10

2.1 Logical Endpoint 类型

类型说明适用指令

2.2 NVSHMEM 中的 cft_handle 实现

// 来自 NVSHMEM: logical_endpoint_device.cuh
enum class le_fabric_handle_kind {
    Unicast,
    Multicast,
};

template <le_fabric_handle_kind K>
struct cft_handle {
    CUlogicalEndpointId id_;     // 32-bit LE ID
    le_fabric_handle_kind type_; // Unicast or Multicast

NVSHMEM 中 offset 的计算方式是堆地址相对于 heap_base 的偏移：

offset = (uintptr_t)heap_addr - (uintptr_t)nvshmemi_device_state_d.heap_base;

3. Fabric 操作的完整生命周期

PTX ISA 明确定义了 Fabric 操作从发起到完成的五个阶段：

1
Issue（发出）
线程执行一个或多个 fabric 指令（try_get/try_put/try_red/try_pullred），由 mbarrier.layout::v1 追踪。
2
Submit（提交）
发出线程必须在关联的 mbarrier phase 推进之前提交这些操作。否则行为未定义。使用 fabric.submit 显式提交。
3
Arrive & Expect-TX（到达与期望计数）
线程在 mbarrier 上执行 arrive-on 和 expect-tx 操作，确保屏障在所有 fabric 操作完成后推进。
4
Wait（等待）
线程阻塞等待 mbarrier phase 完成。已提交的 fabric 操作保证最终完成。
5
Inspect & Handle（检查与处理）
线程检查 predicate report 中的错误信息，并根据应用策略处理。report value 可能包含更详细的错误信息。
Thread 0 Thread 1 Thread N │ │ │ ├─ fabric.try_put ─┤ │ ├─ fabric.try_put ─┤─ fabric.try_get │ │ │ │ ├─ fabric.submit ──┤─ fabric.submit ─┤ │ │ │ ├─ mbarrier.arrive ┤─ mbarrier.arrive┤ ├─ expect_tx ──────┤─ expect_tx ─────┤ │ │ │ ├─ try_wait ───────┴─ try_wait ──────┤ │ (loop until rdy=1) │ │ │ │ ├─ check cnd_inv ──┴─ check cnd_inv ─┤ │ (error handling) │ └─────────────────┴──────────────────┘

关键约束：Submit 必须在 Barrier Phase 推进之前完成

PTX ISA 明确指出："For any thread to observe completion of fabric operations via an mbarrier object, the issuing thread is required to submit those operations before the barrier phase tracking these operations advances. Otherwise, the behavior is undefined."

这意味着如果线程在 submit 之前就让 mbarrier phase 推进了，后续等待行为是未定义的。

4. 六大 Fabric 指令详解

PTX ISA 9.3 定义了 6 条 Fabric 指令，分为 4 条数据传输指令和 2 条控制指令：

指令类型方向描述
try_get数据传输远端 → 本地 SMEM从 Fabric Handle 异步拷贝数据到本地共享内存
try_put数据传输本地 SMEM → 远端从本地共享内存异步拷贝数据到 Fabric Handle
try_red数据传输+归约本地 SMEM → 远端(归约)将本地数据与远端数据执行逐元素归约
try_pullred数据传输+归约远端多播 → 本地 SMEM从多播 Handle 拉取数据并执行跨 GPU 归约
submit控制-提交当前线程之前发出的 fabric 操作
wait控制-等待已提交 fabric 操作的本地 SMEM 读取完成

4.1 fabric.try_get — 异步远端读取

语法

fabric.try_get.async.shared::cta
    .mbarrier::complete_tx::bytes.mbarrier::report::fabric
    .relaxed.sys.b128
    [dst], [srcLeId, srcDataOff], size, [bar];
参数说明
dst目标地址，必须在 .shared::cta 空间，16 字节对齐
srcLeId源 Logical Endpoint ID (u32)，必须是 Unicast LE
srcDataOff源 LE 内资源偏移 (u64)，16 字节对齐
size拷贝字节数 (u32)，必须是 16 的倍数
barmbarrier 对象地址，必须用 layout::v1 初始化

语义

从远端 LE 的 [srcLeId, srcDataOff] 处异步读取 size 字节到本地 SMEM [dst]。操作通过 fabric-proxy 访问源 Handle，通过 async-proxy 写入目标 SMEM，通过 generic-proxy 更新 mbarrier。

完成计数

使用 complete_tx::bytes — 完成计数等于拷贝的字节数（与 try_put 的 16B 单位不同）。

内存序

对源 Handle 的访问是 128-bit 宽度的原子操作，具有 .relaxed.sys 内存序语义。

4.2 fabric.try_put — 异步远端写入

语法（三种变体）

// 变体 0：基本 mbarrier 完成追踪
fabric.try_put.async{.multimem}.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .relaxed.sys.b128
    [dstLeId, dstDataOff], [src], size, [bar];

// 变体 0 + cp_mask：子 16B 写入
fabric.try_put.async{.multimem}.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .cp_mask.relaxed.sys.b128
    [dstLeId, dstDataOff], [src], size, [bar], bytemask;

// 变体 1：带 counted completion
fabric.try_put.async{.multimem}.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .counted::bytes.relaxed.sys.b128
    [dstLeId, dstDataOff, dstCounterOff], [src], size, [bar];
参数说明
dstLeId目标 LE ID (u32)。无 .multimem 时必须为 Unicast LE；有 .multimem 时必须为 Multicast LE
dstDataOff目标 LE 内数据偏移 (u64)，16 字节对齐
dstCounterOff计数器偏移 (u64)，256 字节对齐（仅 counted 变体）
src源 SMEM 地址，16 字节对齐
bytemask16-bit 掩码，选择每个 16B chunk 中哪些字节被写入（仅 cp_mask 变体）

完成计数差异

使用 complete_tx::16B — 完成计数等于 size / 16，即每 16 字节计数 +1。这与 try_get 的 complete_tx::bytes（按字节计数）不同。

cp_mask 变体的用途

当传输大小不是 16 的整数倍时，CFT 模型要求以 16B 为单位传输。cp_mask 允许对最后一个不完整的 16B chunk 进行精确字节级写入。bytemask 的低位对应先写入的字节。

4.3 fabric.try_red — 异步远端归约写入

语法

fabric.try_red.async{.multimem}.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .relaxed.sys.<redOp>.<type>
    [dstLeId, dstDataOff], [src], size, [bar];

支持的归约操作与数据类型

归约类别操作数据类型
位运算.and, .xor, .or.b32, .b64
MinMax.min, .max.u32, .s32, .u64, .s64, .f16, .bf16
算术.add.u32, .u64, .f16, .bf16, .f32, .f64

语义

将本地 SMEM 中的 size 字节数据发送到远端 LE，与远端目标数组中的对应元素执行逐元素归约。归约操作由 .redOp 指定，元素类型由 .type 指定。

支持 .multimem 修饰符，此时目标必须是 Multicast LE，且发出操作的 GPU 必须是该多播组的成员。

4.4 fabric.try_pullred — 异步拉取归约

语法

fabric.try_pullred.async.multimem.shared::cta
    .mbarrier::complete_tx::bytes.mbarrier::report::fabric
    .<redOp>.<type>.sync.relaxed.sys
    [dst], [srcLeId, srcDataOff], size, [bar], imm-membermask;

支持的归约操作与数据类型

归约类别操作数据类型
位运算.and, .xor, .or.b32, .b64
MinMax.min, .max.u32, .s32, .u64, .s64, .bf16, .f16, .e4m3, .e5m2
算术加法.add.u32, .u64, .f16, .bf16, .f32
累加到 f16.add.acc::f16.e4m3, .e5m2
累加到 f32.add.acc::f32.f16, .bf16

语义

从 Multicast LE 指向的多个远端资源异步加载数据，并对所有加载结果执行逐元素归约，将最终结果存入本地 SMEM [dst]。

这是一个 NVLink SHARP 操作 — 归约在 NVSwitch 硬件中执行（in-network reduction），而非在 SM 上。

.sync 修饰符

.sync 修饰符要求发出该指令的线程等待 imm-membermask 指定的 warp lane 也用相同操作数执行该指令。当前唯一支持的 membermask 值是 0xFFFFFFFF（全 warp）。

架构支持差异

.add.acc::f16 和 .add.acc::f32 以及 .min/.max 对 .e4m3/.e5m2 类型的支持需要 sm_120a、sm_121a、sm_100f+ 或 sm_110f+。

4.5 fabric.submit — 操作提交

语法

fabric.submit{.op_restrict::fetching};

语义

提交当前线程之前发出的所有 fabric 操作。这是生命周期中的关键步骤 — 未提交的操作不会被 mbarrier 追踪完成。

变体效果
fabric.submit;提交所有先前 fabric 操作（try_get/try_put/try_red/try_pullred）
fabric.submit.op_restrict::fetching;仅提交 try_get 和 try_pullred 操作（"拉取"类操作）

对已经提交过的 fabric 操作没有影响（幂等性）。

4.6 fabric.wait — SMEM 读等待

语法

fabric.wait.sync_restrict::reads;

语义

等待已提交 fabric 操作对本地 SMEM 的读取完成。这允许应用在 fabric 操作整体完成之前安全地覆写被读取的 SMEM 区域。

这是一种部分完成机制 — 它不等待整个操作的完成，只等待"本地 SMEM 已被读取"这一特定事件。

双缓冲流水线意义

在 TMA + Fabric 的双缓冲流水线中，当 Buffer A 的数据正在通过 fabric 发送到远端时，Buffer B 可以通过 TMA 从远端加载新数据。fabric.wait 确保在覆写 Buffer A 之前，fabric 操作已经读完了 A 中的数据。

5. 三种完成追踪机制

PTX ISA 定义了三种不同的完成追踪机制，分别服务于不同的等待场景：

机制 1：mbarrier 完成追踪（完整完成）

通过 mbarrier.layout::v1 追踪整个 fabric 操作的完成。这是主要的完成追踪方式。

  • complete_tx::bytes — try_get / try_pullred 使用，按字节计数
  • complete_tx::16B — try_put / try_red 使用，按 16 字节计数

观察到完成意味着操作的所有内存访问和 Handle 资源访问都已排序。

机制 2：Fabric-read 完成追踪（部分完成）

通过 fabric.wait.sync_restrict::reads 追踪本地 SMEM 读取的完成。

适用于从 SMEM 读取数据的操作（try_put, try_red）。允许在操作整体完成之前覆写 SMEM。

注意：try_get 不读取 SMEM（它写入 SMEM），所以此机制对 try_get 不提供任何排序保证。

机制 3：Counted Completion（计数器完成，部分完成）

通过远端全局内存中的 8 字节计数器追踪目标数据访问的完成。

// 语法示例（try_put 带 counted completion）
fabric.try_put.async.counted::bytes.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .relaxed.sys.b128
    [dstLeId, dstDataOff, dstCounterOff], [src], size, [bar];
属性说明
计数器位置[dstLeId, dstCounterOff]，256 字节对齐
计数器大小8 字节 (64-bit)
递增粒度每访问 1 字节数据 +1（但实际更新粒度未指定）
更新方式系统级原子操作
等待方式程序轮询计数器达到最终值

对于 .multimem 操作的 counted completion，计数器仅追踪目标设备处的访问。

约束

.counted::bytes 与 .cp_mask 修饰符互斥。如果 LE 不支持 counted completion，行为未定义。

6. mbarrier layout::v1：Fabric 专用屏障

Fabric 操作要求使用 layout::v1 的 mbarrier 对象。这是 PTX ISA 9.3 新增的特性，相比 layout::v0 增加了关键的 payload report 能力。

layout::v0 vs layout::v1 对比

特性layout::v0layout::v1
引入版本PTX 7.0 (sm_80)PTX 9.3 (sm_90+)
Expected arrival count 范围1 ~ 220-11 ~ 29-1
Pending arrival count 范围0 ~ 220-10 ~ 29-1
tx-count 范围-(220-1) ~ 220-1-(220-1) ~ 220-1
Payload Report不支持支持 — 每个 primary phase 一个
Conditional Phase与 Primary Phase 同步推进可独立推进（通过 report-on 机制）
Fabric 操作支持不支持必须使用

6.1 Primary Phase vs Conditional Phase

layout::v0: Primary Phase ──────► 完成 ──┐ Conditional Phase ───► 完成 ──┘── 同步推进（同进同退） layout::v1: Primary Phase ──────► 完成 ──┐ ├── Primary 完成后检查 payload report Conditional Phase ───► 完成 ──┘ ├─ payload report = 0 → Conditional Phase 推进（成功） └─ payload report ≠ 0 → Conditional Phase 不推进（有错误）

Primary Phase 完成条件：

  • Pending arrival count 降至 0
  • tx-count 降至 0

Conditional Phase 完成条件（layout::v1）：

  • Pending arrival count 降至 0
  • tx-count 降至 0
  • Payload report 为 0（无错误报告）

6.2 Payload Report 机制

当 mbarrier 使用 layout::v1 时，每个 primary phase 关联一个 payload report，包含：

组件类型含义
reportPredicate (cnd_inv)1-bit predicate错误指示 — 非零表示有 fabric 操作失败
reportValueopaque value错误详情 — 可通过 cudaFabricOpErrorStatusGet/Count API 解码

报告语义

  • 操作成功：payload report 不被修改（predicate 和 value 保持不变/为零）
  • 操作失败：mbarrier report predicate 被置位，opaque report value 可能包含更多信息

一个类似 fabric.try_put 的操作可能发出多个请求，每个请求可能因不同原因失败。所有失败都通过 report-on 操作汇总到 mbarrier 的 payload report 中。

6.3 NVSHMEM 中的 handle_barrier_t 实现

struct handle_barrier_t {
    alignas(16) __mbarrier_t bar;

    // 初始化必须使用 layout::v1
    inline __device__ void init(int arvCnt) {
        asm volatile ("mbarrier.init.shared.layout::v1.b64 [%0], %1;"
            :: "l"(smem_addr), "r"(arvCnt) : "memory");
    }

    // expect_tx 以 16B 为单位（CFT_HANDLE_TX_SIZE = 16）
    inline __device__ uint64_t arrive_relaxed(uint32_t size_bytes) {
        uint32_t adjusted_size = ((size_bytes + 15) / 16) * 16;
        asm volatile(
            "mbarrier.arrive.expect_tx.relaxed.cta.shared::cta.b64 %0, [%1], %2;"
            : "=l"(state)
            : "l"(smem_addr), "r"(adjusted_size / 16));
        return state;
    }

    // 读取 rdy 和 cnd_inv 两个 predicate
    inline __device__ mbarrier_try_wait_rdy_cnd
    try_wait_token_rdy_cnd(uint64_t state) {
        asm volatile(
            "mbarrier.try_wait.phase_type::primary"
            ".acquire.cta.shared::cta.b64 "
            "p_rdy|p_cnd, [%2], %3;"
            ...);
        return {rdy, cnd_inv};
    }

    // 最终等待 + 错误检查
    inline __device__ bool try_wait_token(uint64_t state) {
        uint8_t err = 0;
        while(!try_wait_token_with_err(state, &err)) {}
        assert(err == 0);  // 所有 fabric 错误在此终止
        return true;
    }
};

7. Fence 与内存序：fabric-proxy 模型

CFT 编程模型引入了新的 fabric-proxy 概念。Fabric 操作涉及多个 proxy 之间的内存访问，需要专门的 fence 指令来建立跨 proxy 的内存序。

7.1 三种 Fabric Fence 指令

// 1. generic → fabric 方向（释放语义）
fence.proxy.generic::fabric.alias.release.sys;

// 2. fabric → generic 方向（获取语义）
fence.proxy.fabric::generic.alias.acquire.sys;

// 3. fabric → fabric 方向（内部排序）
fence.proxy.fabric::fabric.alias.sem_fabric.sys;
// sem_fabric = { .acquire, .release }

Proxy 分类

Proxy用途相关操作
generic-proxy常规内存访问（ld/st/atom/red）mbarrier 更新
async-proxy异步拷贝（TMA, cp.async）SMEM ↔ Global 数据移动
fabric-proxyCFT Handle 资源访问fabric.try_* 指令

7.2 Fabric 操作中的 Proxy 使用模式

fabric.try_put: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ fabric-proxy │ │ async-proxy │ │generic-proxy │ │ 访问 dst │ │ 读取 src │ │ 更新 mbar │ │ Handle资源 │ │ SMEM数据 │ │ SMEM对象 │ └──────────────┘ └──────────────┘ └──────────────┘ fabric.try_get: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ fabric-proxy │ │ async-proxy │ │generic-proxy │ │ 访问 src │ │ 写入 dst │ │ 更新 mbar │ │ Handle资源 │ │ SMEM数据 │ │ SMEM对象 │ └──────────────┘ └──────────────┘ └──────────────┘ 正确的 fence 序列确保跨 proxy 的内存序： fence.proxy.generic::fabric.alias.release.sys; // ... fabric operations ... fence.proxy.fabric::generic.alias.acquire.sys;

7.3 NVSHMEM 中的 Fence 使用模式

// 来自 NVSHMEM: put_signal 操作中的 fence 使用
// 1. 在 fabric put 之前：确保数据写入对 fabric-proxy 可见
fence.proxy.generic::fabric.alias.release.sys;

// 2. fabric.try_put 操作
fabric.try_put.async.shared::cta...

// 3. 在 fabric put 之后：确保 fabric 操作的结果对 generic-proxy 可见
fence.proxy.fabric::generic.alias.acquire.sys;

PTX ISA 9.3 新增

.to_proxykind::from_proxykind_fabric 修饰符是 PTX ISA 9.3 新增的，需要 sm_100 或更高架构。在此之前（PTX 7.5+），只支持 fence.proxy.alias 和 fence.proxy.async。

8. 错误报告与弹性处理

错误处理是 CFT 模型区别于传统 NVLink 操作的核心特性。PTX ISA 定义了完整的错误报告链路：

Fabric 操作失败 │ ▼ Report-on 操作 (mbarrier report-on) │ ▼ 更新 Payload Report ┌─────────────────────────┐ │ reportPredicate (cnd_inv)│ ← 1-bit 错误标志 │ reportValue (opaque) │ ← 详细错误信息 └─────────────────────────┘ │ ▼ mbarrier.try_wait.phase_type::primary 返回 waitComplete | reportPredicate {, reportValue} │ ▼ 应用检查 reportPredicate ┌────────────────────────────────┐ │ if (reportPredicate != 0) { │ │ // 有错误发生 │ │ // reportValue 包含详情 │ │ // 可通过 cudaFabricOpError │ │ // StatusGet/Count 解码 │ │ } │ └────────────────────────────────┘

PTX ISA 原文

"If the operation succeeds, the payload report (contents of the mbarrier object) is not modified. Otherwise, the operation fails, the mbarrier report predicate is set, and the opaque report value may contain more information, which may be decoded via the cudaFabricOpErrorStatusGet/Count APIs."

8.1 错误处理的设计哲学

传统 NVLink 操作

基于指针的 NVLink 内存操作遇到 NVLink 故障时直接致命。应用无法感知、无法处理、无法恢复。

→ 零弹性

CFT Fabric 操作

Fabric 操作将故障报告给应用。如何处理完全由应用决定：

  • 中止进程 (abort)
  • 优雅清理 (clean teardown)
  • 重试操作 (retry transient failures)

→ 完全弹性

8.2 Report-on 操作的语义

PTX ISA 定义了 report-on 操作作为 mbarrier 的一种通用机制：

"The report-on operation on an mbarrier object signals miscellaneous information regarding asynchronous operation. For example, such miscellaneous information can be about errors, warnings encountered; otherwise, it can as well be any extra information conveyed by asynchronous operation."

目前，能够执行 report-on 操作的异步操作包括：

  • fabric.try_get
  • fabric.try_put
  • fabric.try_red
  • fabric.try_pullred

未定义行为警告

PTX ISA 明确指出："The behavior of a program that combines fabric operations specifying report::fabric with other operations specifying different reporting mechanism is undefined."

即：同一个 mbarrier 不能同时追踪 fabric 操作（使用 report::fabric）和非 fabric 操作（使用其他 reporting mechanism）。

8.3 NVSHMEM 的实际处理策略

在 NVSHMEM 的实现中，错误处理采用了assert-fatal 策略：

// logical_endpoint_device.cuh
inline __device__ bool try_wait_token(uint64_t state) {
    uint8_t err = 0;
    while(!try_wait_token_with_err(state, &err)) {}
    assert(err == 0);  // 所有 fabric 错误在此终止
    return true;
}

虽然 PTX ISA 赋予了应用处理错误的完全自由，但 NVSHMEM 选择了最简单的策略 — 遇到任何 fabric 错误直接 assert 终止。这与 NVSHMEM 作为通信库的定位有关：通信错误通常意味着系统级问题，库层面难以做出合理的恢复决策。

10. 总结与架构洞察

CFT 编程模型的核心价值

CUDA Compute Fabric Transport 代表了 GPU 通信架构的根本性转变：

  1. 从 CPU-Proxy 到 GPU-Resident：通信操作完全在 GPU 线程中发起和完成，消除了 CPU 代理瓶颈
  2. 从 Fatal 到 Resilient：通过 Handle 抽象和 mbarrier report 机制，将 NVLink 故障从不可处理的硬件异常转变为可处理的软件事件
  3. 从同步到异步：所有操作异步执行，通过 mbarrier 实现细粒度的完成追踪和内存序保证
  4. 从裸指针到 Handle：CFT Handle 提供了间接寻址层，使硬件能够管理端点生命周期和故障隔离
