摘要

本专利披露了一种程序化多播技术 (programmatic multicast technique),其中一个处理器核心为其他多个核心获取信息。该技术通过让指定的 leader core 一次性从全局内存读取数据,然后通过硬件级多播机制同时分发给多个目标线程块,消除了重复的 L2 缓存读取。多播由跟踪电路 (tracking circuitry) 支持,该电路在接收自处理器核心的多播请求和可用内存之间进行接口。请求包携带目标标识符、共享内存偏移量和屏障同步信息,数据在路由到"分离点 (point of separation)"之前保持为单一事务,之后硬件复制数据并分发到各目标。

问题背景与技术挑战

GPU 扩展中的 L2 缓存带宽瓶颈

挑战 1: 强扩展 (Strong Scaling) 导致缓存带宽需求指数增长

当开发者缩小计算 tile 尺寸以实现更快的结果时 (强扩展),L2 缓存的带宽需求呈指数增长。多个线程块需要读取相同的全局内存数据 (如矩阵乘法的权重矩阵),导致相同的缓存行被重复读取,严重浪费 L2 缓存带宽。

挑战 2: 冗余读取模式 (Redundant Fetch Patterns)

在传统架构中,当 8 个 SM 需要相同的数据块时,会产生 8 次独立的 L2 缓存读取请求。这不仅浪费了 L2 带宽,还增加了 crossbar 互连的流量,导致拥塞和延迟增加。对于矩阵乘法等具有高度数据共享的计算模式,这个问题尤为严重。

挑战 3: 缺乏硬件级多播支持

传统 GPU 的内存子系统只支持单播 (unicast) 读取——每个请求对应一个响应。对于需要向多个计算引擎分发相同数据的场景 (如 Broadcast、AllGather),缺乏硬件级多播机制意味着必须通过软件循环多次读取,效率低下。

挑战 4: 跨引擎同步的开销

在多 SM 场景中,确保所有目标引擎在数据分发完成后同步开始计算是一个挑战。传统的内存屏障 (memory barrier) 无法感知异步多播事务的完成状态,需要额外的软件同步步骤。

来源: US12020035B2 Background, US20230289190A1

为什么这个问题越来越重要?

随着 GPU 向更多 SM 集成 (Hopper H100: 132 SM, Blackwell B200: 更多 chiplet),强扩展场景下的 L2 缓存带宽压力呈指数增长。在矩阵乘法、注意力计算等核心工作负载中,相同的权重/激活值被多个 SM 重复读取。US12020035B2 通过硬件级多播从根本上解决了这个问题——一次读取,多处分发,将 L2 带宽需求从 O(N) 降低到 O(1)。

核心设计理念: 将数据多播从"多次单播"转变为"一次读取 + 硬件分发"。Leader core 发起单次全局内存读取,L2 Request Coalescer 记录多播元数据,数据返回后通过 on-chip network 路由到"分离点",硬件在此复制数据并分发到所有目标 core,最终通过字节级计数器跟踪完成状态并触发同步。

传统方案 (多次单播)

• 8 个 SM 需要相同数据

• 8 次独立的 L2 缓存读取

• 8 次 crossbar 传输

• 8 次共享内存写入

L2 带宽消耗: 8x

Crossbar 流量: 8x

本专利方案 (硬件多播)

• 8 个 SM 需要相同数据

• 1 次 L2 缓存读取

• 1 次 crossbar 传输 (到分离点)

• 硬件在分离点复制为 8 份

L2 带宽消耗: 1x (节省 87.5%)

Crossbar 流量: 大幅减少

多播请求包结构

请求包字段详解

Leader core 生成的多播请求包包含完整的元数据,使接收方 core 在没有任何初始上下文的情况下也能正确处理数据。包结构如下:

字段 类型 作用 示例
Global Memory Address 64-bit 要读取的全局内存地址 0x7fff_1234_5678
Destination ID List Bit Vector / List 目标 core/thread block 标识符列表 位向量: 0b10110100
Destination Shared Memory Offset N-bit 数据写入各目标共享内存的偏移地址 各目标可能不同
CTA ID Mask Bit Mask 协作线程数组 (CTA) 的位掩码,指定哪些 CTA 参与 CGA 内的 CTA 子集
Barrier ID / Sync Entity N-bit 同步屏障标识符,用于跟踪多播完成 关联 MEMBAR.ASYNC
Source ID N-bit 发起多播的 leader core 标识符 SM ID + Core ID
Byte Count N-bit 多播数据的字节数,用于完成跟踪 如 128 bytes (cache line)
关键设计: 由于接收方 core 在收到数据之前没有关于多播的上下文信息,请求包必须携带所有处理元数据。这包括目标列表 (可用位向量紧凑表示)、共享内存写入偏移量、屏障同步标识符等。L2 Request Coalescer 提取并存储这些元数据到硬件跟踪结构中,然后在数据返回时重建分发列表。
来源: US12020035B2, US20230289190A1

L2 请求合并器 (LRC)

LRC 核心功能

L2 Request Coalescer (LRC) 是该专利的核心硬件组件,负责拦截多播请求、存储元数据、并在数据返回后重建分发列表。

阶段 1: 请求拦截与元数据提取

当 leader core 发出的多播请求包通过 crossbar 到达 L2 分区时,LRC 拦截该请求。LRC 从请求包中提取多播特定信息 (目标 ID 列表、共享内存偏移量、屏障 ID 等),并将其存储到硬件跟踪结构 (tracking structure) 中。

阶段 2: 转发标准读取请求

LRC 向 L2 缓存转发一个不带多播标签的标准读取请求。这意味着 L2 缓存本身不需要理解多播语义,只需处理普通的内存读取。这保持了 L2 缓存设计的简洁性,同时通过 LRC 添加了多播能力。

阶段 3: 数据返回与分发列表重建

当 L2 缓存返回数据 payload 时,LRC 将数据与跟踪结构中挂起的条目进行关联,重建分发列表。LRC 构建一个响应包,将检索到的数据与原始目标列表组合在一起。

跟踪结构 (Tracking Structure)

跟踪结构是一个硬件 CAM (Content-Addressable Memory) 或 SRAM 表,每条记录包含: 请求 ID、目标 ID 位向量、共享内存偏移量、屏障 ID、字节计数、状态 (pending/complete)。当多播请求到达时分配一条记录,当所有目标确认接收后释放该记录。

来源: US12020035B2

LRC 设计的巧妙之处

LRC 的关键创新是将多播语义与 L2 缓存解耦。L2 缓存不需要知道多播的存在,只需处理标准读取请求。多播的所有复杂性 (目标列表、分发、同步) 都由 LRC 处理。这种设计使得多播功能可以作为现有 L2 缓存的"附加层"实现,无需重新设计缓存架构。

数据分发机制

从单一事务到多分发的完整流程

1

Leader Core 发起请求

指定的 leader core 生成多播请求包,包含全局内存地址、目标 ID 位向量、共享内存偏移量、屏障 ID 等元数据。

2

LRC 提取并存储元数据

L2 Request Coalescer 拦截请求,将多播特定信息提取到硬件跟踪结构中。然后向 L2 缓存转发标准读取请求 (不带多播标签)。

3

L2 缓存返回数据

L2 缓存返回请求的数据 payload (如一个 128-byte cache line)。LRC 将数据与跟踪结构中的挂起条目关联,重建分发列表。

4

LRC 构建响应包

LRC 构建响应包,将检索到的数据与原始目标列表组合。该响应包作为单一事务进入 on-chip network。

5

到达"分离点 (Point of Separation)"

响应包通过 on-chip network 路由,在到达"分离点"之前保持为单一事务。在分离点,网络逻辑将数据包"复制为两个或多个包",将副本沿不同分支路由到各目标 core。这发生在物理网络交叉点,由 multidrop switch 硬件执行。

// 分离点示例 (4 个目标)
单一包进入 → [复制逻辑] → 4 个包输出
包1 → SM0 包2 → SM2 包3 → SM3 包4 → SM5
6

各目标 Core 接收数据

各 follower core 接收到数据,写入各自的共享内存 (根据请求包中指定的偏移量)。每个接收方递增字节计数器,并向 leader 发送确认信号。

7

同步完成

当所有目标都确认接收后,屏障计数器达到预期值,屏障清除。所有 follower core 通过 MEMBAR.ASYNC 感知多播完成,开始计算。

分离点 (Point of Separation) 的硬件实现

"分离点"是 on-chip network 中的物理网络节点,在此处多播包被复制并分发到不同分支。其关键特性:

  • 单一入口,多出口: 响应包从单一入口进入分离点,根据目标列表被复制为多个包,从不同出口路由出去
  • 硬件复制: 数据 payload 在硬件中被复制,无需软件干预。复制发生在网络层,而不是在数据返回 leader 后再分发
  • 多级分离: 对于多目标场景,可能存在多级分离点。例如,8 个目标可能需要 3 级二叉树分离 (1→2→4→8)
  • 与 crossbar 集成: 分离点与 GPU 的 crossbar 互连深度集成,利用 crossbar 的多播能力 (如果硬件支持)
来源: US12020035B2, US20230289190A1

信用仲裁机制 (Credit-Based Arbitration)

为什么需要信用仲裁?

多播包需要同时发送到多个目标。如果某个目标的接收缓冲区已满 (没有可用信用 credit),而网络仍然发送数据,会导致数据丢失或死锁。信用仲裁确保多播包只在所有目标都有可用信用时才被发送。

核心规则: Multidrop 路由逻辑在传输前为每个预期目标获取信用。如果任何目标缺乏可用信用,"数据包将被阻塞 (packet will get stalled)"。只有当所有目标都有可用信用时,多播包才被允许前进。

信用池 (Credit Pool)

每个网络节点 (SM、L2 分区等) 维护一个信用池,表示其接收缓冲区中可用的空间。信用数量取决于缓冲区大小和数据包大小。

  • 发送前检查: 检查所有目标的信用
  • 信用消耗: 发送后扣减相应信用
  • 信用恢复: 接收方处理数据后返回信用

多播优先级

专利指出,优先级仲裁通常 favor 多播请求高于单播流量,以确保"改进的多播包吞吐量"并防止队头阻塞 (head-of-line blocking)。

  • 多播优先: 减少多播包等待时间
  • 防止饿死: 避免单播流量阻塞多播
  • 公平性: 在多个多播请求间公平仲裁

信用仲裁的性能影响

信用仲裁确保多播包不会在目标缓冲区满时被发送,防止数据丢失。但这也意味着如果某个目标处理缓慢 (如正在进行密集计算),会阻塞整个多播事务。专利通过多播优先级仲裁缓解这个问题,但也暗示了在极端情况下可能需要的 flow control 策略。

来源: US12020035B2

同步机制 (MEMBAR.ASYNC)

字节级完成跟踪

多播完成跟踪依赖于与特定屏障地址关联的字节级计数器 (byte-level counters)。接收方在数据流进入时递增到达总数 (arrival totals),并向发起方发送确认信号。

数据接收
Follower core 接收数据
递增计数器
字节计数器 += 接收字节数
发送确认
ACK 信号返回 leader

MEMBAR.ASYNC: 异步内存屏障

问题: 标准内存屏障 (memory barrier) 忽略挂起的异步提取操作。这意味着在多播数据还在传输中时,标准屏障可能已经清除,导致线程在数据未就绪时就开始计算。
解决方案: MEMBAR.ASYNC

该系统采用 MEMBAR.ASYNC 显式等待所有完成确认。相位计数器 (phase counters) 跟踪未完成的操作,允许新的提取在交替相位中继续,而先前的事务在后台完成。

// Leader Core 流程
multicast_async(global_addr, dest_list, barrier_id);

// Follower Core 流程
MEMBAR.ASYNC(barrier_id); // 等待多播完成

// 屏障清除后,数据已在共享内存中
compute

相位计数器 (Phase Counter) 机制

相位计数器允许系统跟踪多播事务的完成状态,同时允许新的事务启动:

  • 相位 0: 第一批多播事务在相位 0 中启动,相位计数器记录未完成的事务数
  • 相位 1: 当相位 0 的事务还在完成中时,新的事务可以在相位 1 中启动
  • 交替等待: MEMBAR.ASYNC(phase) 等待特定相位的所有事务完成,而不阻塞其他相位的新事务
  • 无 stall 流水线: 通过相位交替,系统可以实现无 stall 的多播流水线,提高吞吐量
来源: US12020035B2, PTX ISA (MEMBAR)

协作组阵列 (CGA) 模型

CGA 模型概述

该专利的多播架构构建在协作组阵列 (Cooperative Group Array, CGA) 范式之上。CGA 确保"所有 CTA (协作线程数组) 被启动到相同的硬件组织级别",通过保证空间邻近性 (spatial proximity) 和并发调度 (concurrent scheduling),实现高效的跨核心通信、统一共享内存池和芯片上的确定性执行。

空间邻近性

CGA 的所有 CTA 被调度到相同的 GPU 处理集群 (GPC) 或相邻的 SM。这确保了多播数据可以在低延迟的 on-chip network 中传输,而不需要跨 chiplet 或跨 GPU。

并发调度

硬件保证 CGA 中的所有 CTA 同时被调度和执行。这确保了多播的 follower core 在 leader 发起多播时已经就绪,避免了调度延迟。

统一共享内存

CGA 内的 CTA 可以访问统一的共享内存池。多播数据被写入各 CTA 的共享内存 (根据请求包中指定的偏移量),实现 CTA 间的低延迟数据共享。

CGA 与多播的协同

CGA 模型为该专利的多播机制提供了理想的执行环境:

  • 可预测的延迟: 由于 CTA 在空间上邻近且并发执行,多播数据从 leader 到 follower 的传输延迟是可预测的,不需要复杂的超时和重传机制
  • 共享内存池: CGA 内的 CTA 可以通过共享内存直接读写数据,多播只需将数据写入共享内存,无需额外的全局内存访问
  • 确定性执行: CGA 确保所有 CTA 在相同的时间点到达同步点,简化了多播完成跟踪和屏障同步
  • 硬件支持: NVIDIA GPU 对 CGA 的硬件支持 (如 cluster.sync()) 与多播机制协同工作,提供端到端的跨 CTA 同步
来源: US12020035B2, CN116775265A (CGA), NVIDIA Cooperative Groups 博客

对集合通信的支持

原生支持 Broadcast 和 AllGather 模式

虽然专利文本中没有明确命名 "AllGather" 和 "Broadcast",但该架构原生支持这些广播式分发模式。通过让 "leader thread" 请求多个 "follower threads" 所需的数据,系统高效地将相同 payload 分发到整个 "cooperative group array",与 allgather 风格操作的要求匹配,且无需冗余的网络遍历。

集合通信操作 传统实现 本专利硬件加速实现
Broadcast Root 进程发送 N-1 次,或通过二叉树 (log N 步) Leader 单次读取,硬件多播一次分发到所有目标,O(1) L2 带宽
AllGather Ring AllGather: N-1 步,每步发送/接收 chunk 每个 leader 多播自己的 chunk,CGA 内一次多播完成分发
Reduce-Scatter Ring Reduce: N-1 步,每步 reduce + 转发 多播 + 硬件 reduce (如果 LRC 支持),减少中间步骤
All-to-All 每个进程发送 N 个 chunk,共 N² 次发送 每个 leader 多播到目标子集,减少总发送次数

Broadcast 加速示例

场景: 一个权重矩阵需要广播到 CGA 中的 8 个 CTA。

传统方案: 8 次 L2 读取 + 8 次 crossbar 传输

专利方案: 1 次 L2 读取 + 1 次 crossbar (到分离点) + 硬件复制 × 8

L2 带宽节省: 87.5%

AllGather 加速示例

场景: 8 个 CTA 各自有一个数据 chunk 需要分发给其他 7 个。

传统 Ring AllGather: 7 步,每步发送 + 接收

专利方案: 每个 CTA 的 leader 多播自己的 chunk,CGA 内一次多播完成

通信步数: 从 7 步减少到并发多播

对 NCCL 的意义

该专利为 NCCL 的 Broadcast 和 AllGather 操作提供了硬件级加速基础。在 GB200 NVL72 等大规模系统中,多播是集合通信的关键原语。通过硬件多播,NCCL 可以显著减少跨 NVLink 的重复数据传输,提升集合通信的吞吐量。未来 NCCL 版本可能直接利用该专利的多播机制,实现更高效的集合通信。

核心权利要求解读

根据专利公开文本,核心权利要求涵盖以下技术方案:

权利要求 1: 程序化多播方法

一种用于跨多个计算引擎的程序化控制数据多播的方法,包括:由第一处理器核心生成多播请求,该请求包含全局内存地址、目标标识符列表和同步实体;通过互连网络将多播请求发送到请求合并器;由合并器执行单次内存读取;以及通过互连网络将数据多播到多个目标计算引擎,其中数据在到达分离点之前作为单一事务传输,之后被复制并分发到各目标。

权利要求 2: L2 请求合并器与跟踪结构

一种 L2 请求合并器,包括:用于从多播请求中提取多播特定信息的电路;用于将多播信息存储到硬件跟踪结构的逻辑;用于向缓存转发标准读取请求的接口;以及用于在数据返回时重建分发列表并构建响应包的电路。

权利要求 3: 分离点与硬件复制

一种 on-chip network 中的分离点,包括:用于接收包含数据和目标列表的单一响应包的入口;用于根据目标列表将数据包复制为多个包的复制逻辑;以及用于将各复制包路由到不同目标核心的多个出口。

权利要求 4: 信用仲裁机制

一种多播信用仲裁方法,包括:在传输多播包之前,为每个预期目标获取信用;如果任何目标缺乏可用信用,则阻塞多播包;当所有目标都有可用信用时,允许播包传输;以及优先处理多播请求优于单播流量。

权利要求 5: MEMBAR.ASYNC 同步

一种多播同步方法,包括:使用字节级计数器跟踪多播数据完成状态;接收方在数据到达时递增计数器并发送确认;通过 MEMBAR.ASYNC 显式等待所有多播完成确认;以及使用相位计数器跟踪未完成的多播事务,允许交替相位中的新事务。

权利要求 6: CGA 集成

一种在协作组阵列 (CGA) 中执行多播的方法,包括:将 CGA 的所有 CTA 调度到相同的硬件组织级别;保证空间邻近性和并发调度;通过多播将数据分发到 CGA 内各 CTA 的共享内存;以及使用 CGA 同步原语确保多播完成后各 CTA 开始计算。

影响分析与未来展望

对 L2 缓存带宽的影响

该专利将多播场景下的 L2 缓存带宽需求从 O(N) 降低到 O(1)。在 8-SM 的 Broadcast 场景中,L2 带宽节省 87.5%。对于 Hopper H100 (132 SM) 和 Blackwell B200 (更多 chiplet),这种节省更为显著,直接缓解了强扩展场景下的 L2 带宽瓶颈。

对 NCCL 集合通信的影响

该专利为 NCCL 的 Broadcast、AllGather 等操作提供了硬件级加速基础。通过硬件多播,NCCL 可以减少跨 NVLink 的重复数据传输,提升集合通信吞吐量。未来 NCCL 版本可能直接利用该专利的多播机制。

对 LLM 训练的影响

在 LLM 训练中,注意力计算和矩阵乘法涉及大量数据共享。硬件多播可以将权重矩阵、激活值等共享数据一次分发到多个 SM,减少 L2 带宽压力和 crossbar 拥塞,提升训练吞吐量。

对矩阵乘法的影响

矩阵乘法 (GEMM) 是该专利的重点优化场景。在 GEMM 中,相同的权重矩阵被多个 SM 的 tile 读取。硬件多播将权重矩阵的读取从 N 次减少到 1 次,显著降低 L2 带宽需求,提升 GEMM 吞吐量。

对 GPU 架构设计的影响

该专利揭示了 NVIDIA GPU 架构中 L2 缓存与多播硬件的协同设计思路。LRC 作为 L2 缓存的"附加层",无需重新设计缓存即可添加多播能力。这种设计可能被集成到 Hopper、Blackwell 及后续架构中。

专利护城河

该专利由 NVIDIA GPU 架构核心团队 (Apoorv Parle, Jack Choquette 等) 发明,保护了 NVIDIA 在硬件级多播的核心技术。竞争对手 (AMD、华为) 需要开发替代的多播方案或获得授权,构成了重要的专利壁垒。

总结

US12020035B2 是 NVIDIA 在 GPU 硬件级数据多播领域的核心专利,通过 L2 Request Coalescer、分离点硬件复制、信用仲裁和 MEMBAR.ASYNC 同步,将多播场景下的 L2 带宽需求从 O(N) 降低到 O(1)。该专利与 CGA 模型深度集成,确保跨 SM 的多播数据分发和同步的一致性。

在 LLM 训练、矩阵乘法、集合通信等场景中,该专利的多播机制可以显著减少 L2 带宽压力和 crossbar 拥塞,提升整体吞吐量。随着 GPU 向更多 SM 集成 (Hopper 132 SM、Blackwell 更多 chiplet),该专利的多播机制将变得越来越重要。

对于竞争对手,该专利构成了硬件级多播的专利壁垒。对于学术界和开源社区,该专利揭示了 NVIDIA GPU 多播机制的底层原理,有助于优化分布式训练和集合通信库。
