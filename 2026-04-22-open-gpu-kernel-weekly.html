NVLink Fusion 扩展至 Marvell 定制 XPU

2026-03 GTC

创新点: Marvell 将 NVLink Fusion 集成到定制 XPU 设计中，使第三方加速器能够参与 NVLink scale-up 拓扑，这是 NVLink 生态系统的重大扩展。

来源: NVIDIA Newsroom

详细技术方案

NVIDIA 与 Marvell 合作，将后者整合到 "NVIDIA AI factory 和 AI-RAN 生态系统"。Marvell 将提供 "定制 XPUs 和 NVLink Fusion 兼容的 scale-up 网络"，创建 "与 NVIDIA 系统完全兼容的异构 AI 基础设施"。这意味着 NVLink 不再仅限于 NVIDIA 自研 GPU，而是向第三方加速器开放，可能从根本上改变 AI 加速器竞争格局。

GTC 2026: Vera Rubin 全栈平台

2026-03

创新点: Vera Rubin 平台采用 NVIDIA Vera CPU (88核 Olym Arm) + Rubin GPU (288GB HBM) + NVLink-C2C 相干互连，七芯片全栈平台设计。

NVSwitch 机架级 GPU 互连

GB300 NVL72 平台 — 260 TB/s NVSwitch 带宽

2026 GTC

创新点: GB300 NVL72 集成 72 颗 Blackwell GPU，通过 NVLink Switch fabric 互连，提供 260 TB/s 聚合 GPU 带宽，代表当前机架级 GPU 互连的最高水平。

来源: NVIDIA NVLink & NVSwitch

架构详情

GB300 NVL72 是 NVIDIA 当前的旗舰机架级系统，72 颗 Blackwell GPU 通过 NVLink Switch 全互联。NVSwitch 在交换机级别实现 NCCL 集合通信 (AllReduce/AllGather/Broadcast) 的硬件卸载，大幅降低 LLM 推理和训练的通信延迟。HGX 平台支持共享 NVSwitch GPU 透传虚拟化，允许多租户安全共享互连资源。

Vera Rubin 平台采用 NVLink-C2C

2026 GTC

创新点: Vera Rubin 全栈计算平台通过 NVLink-C2C 连接 Vera CPU 和 Rubin GPU，扩展了 Grace-Blackwell C2C 模型到下一代平台，实现 Vera CPU 与 Rubin GPU 之间的相干内存访问。

来源: GTC 2026: Live Updates | NVIDIA NVLink-C2C

C2C 架构详情

NVLink-C2C 提供 "1 TB/s 内存带宽" 在 superchip 内部，同时提供 "比 PCIe Gen 6 PHY 高 6 倍能效和 3.5 倍面积效率"。支持 "AMBA CHI 或 Compute Express Link (CXL)" 协议。采用 "世界级 SerDes 和 Link 设计技术"，跨 "多芯片模块 (MCM) 和硅 interposer 或晶圆级连接"。实现 "高带宽、相干数据传输" 并支持 "处理器间原子操作" 用于快速同步。部署于 GB200/GB300 NVL72、Vera Rubin NVL72 和 DGX Spark (GB10)。

#1110 — uvm: fix NULL pointer dereference in uvm_hmm_unregister_gpu

2026-04-16

Bug Fix Open PR
来源: PR #1110

技术详情

该 PR 修复了 uvm_hmm_unregister_gpu 函数中的 NULL 指针解引用问题。在 HMM 管理的 GPU 内存注销流程中，如果特定条件未满足，可能导致空指针访问。这是 UVM 模块内存管理路径的关键稳定性修复。

#1088 — fix(uvm): check pRmApi->Control status in nvGpuOpsSetPageDirectory

2026-03-30

Bug Fix Open PR MMU
来源: PR #1088

技术详情

nvGpuOpsSetPageDirectory 负责管理 GPU 页目录设置，是 MMU 页表配置的核心函数。缺少对 pRmApi->Control 返回值的检查可能导致页表配置失败但不报错，进而引发后续的内存访问异常或数据损坏。

技术趋势分析

开源模块稳定性差距扩大

本周数据显示闭源驱动与开源 GPU 内核模块之间的稳定性差距正在扩大。多个关键 bug 仅在开源模块中出现，用户确认相同版本的闭源驱动不存在这些问题。PR #1110, #1096, #1088 均针对 UVM/MMU 稳定性，表明 NVIDIA 正在积极修复，但 bug 报告率超过合并率。

需要关注

Linux 6.19+ 前向兼容性

PR #1015 (已合并) 解决了 Linux 6.19+ 中 HMM 和 PMM API 变更，展示 NVIDIA 对上游内核变更保持前向兼容的承诺。这对在前沿发行版中的采用至关重要。

积极进展

GTC 2026 硬件路线图影响

Vera Rubin 平台 (Vera CPU + Rubin GPU 通过 NVLink-C2C) 和 GB300 NVL72 量产信号表明，开源内核模块将需要支持这些新架构。当前主要目标 Ampere 及以后，但完整 Vera Rubin 支持需要新的 SoC 定义和互连初始化代码。

长期挑战

短期趋势 (2026 Q2)

  • PR #1110 (UVM HMM NULL 指针) 预计将合入下一版本
  • 595.x 稳定分支可能包含 UVM 稳定性修复
  • RHEL9 签名开源模块结束 tech preview，进入正式仓库
  • Blackwell 消费级 GPU 开源支持逐步完善 (Issue #1111)
  • Linux 6.19+ 兼容性修复进入生产分支
  • Nova DRM 驱动 (Rust) 持续进展，可能与官方开源模块互补

中期趋势 (2026-2028)

  • Vera Rubin NVL72 量产部署，开源模块需支持 Vera SoC
  • Marvell NVLink Fusion XPU 可能需要开源驱动适配
  • Feynman 架构 + LP40 LPU 开源支持
  • 异构 NVLink 拓扑 (NVIDIA GPU + Marvell XPU) 开源驱动支持
  • 企业级采用加速 (RHEL9 签名模块降低部署门槛)
  • 开源/闭源驱动稳定性差距逐步缩小

本周核心洞察

1. UVM/MMU 是本周最关键模块

3 个 open PRs (#1110, #1088, #1096) 和 3 个 open Issues (#1108, #1107, #1115) 集中在 UVM 和内存管理路径。内核恐慌 (#1108) 特别值得关注，因为它在闭源驱动中不存在，暗示开源模块回归。建议密切关注 PR #1110 的合并进展。

2. 开源模块与闭源驱动的稳定性差距需正视

用户明确报告相同驱动版本下，闭源版本稳定而开源版本崩溃。这对评估开源模块用于生产工作负载是重要警示。NVIDIA 需要加速开源模块的 QA 流程。

3. GTC 2026 硬件路线图为开源模块提出新挑战

Vera Rubin 平台和 Marvell NVLink Fusion 合作意味着开源内核模块最终需要支持 Vera SoC、异构 NVLink 拓扑和第三方 XPU。这些架构变更需要大量的 SoC 定义和互连初始化代码。

4. 企业级采用加速在即

Red Hat 合作签名开源模块解决了 Secure Boot 这一企业部署关键障碍。一旦 tech preview 结束并进入 NVIDIA Compute 正式仓库，企业采用开源模块的速度预计将显著提升。
