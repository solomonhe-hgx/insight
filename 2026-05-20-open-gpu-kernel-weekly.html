Issue #1148 — RTX 3090 Ti S3 周期挂起: NVLink 单边关闭导致 virtualBar2.pCpuMapping NULL

2026-05-17 新 Issue

新 Bug: Ampere RTX 3090 Ti 在无 Peer 连接环境下，每次 S3 睡眠触发 NVLink 单边关闭序列。多次睡眠后 virtualBar2[GPU_GFID_PF].pCpuMapping 在恢复时为 NULL，导致内存管理单元无法操作，挂起序列完全停止。

New Bug NVLink / Suspend / MMU
来源: Issue #1148

技术详情

该 Issue 报告了一个间歇性 S3 唤醒冻结问题。环境：Arch Linux, Kernel 7.0.3-arch1-2, 驱动 595.71.05。由于显卡无 Peer 连接，每次睡眠都回退到逐个禁用 NVLink 的序列。报告者提出假设：重复执行此单边关闭序列正在逐渐损害内部内存管理结构。关键失败点：virtualBar2[GPU_GFID_PF].pCpuMapping 在恢复时为 NULL。没有此 aperture，GPU MMU 无法操作。日志显示多个分配例程的断言失败，无法保留虚拟地址范围。技术假设：虚拟地址分配器在单边关闭阶段存在慢速资源泄漏，经过多个睡眠周期后分配器耗尽内部 aperture 可用空间。

Issue #1152 — 新增 NVreg_AllowP2PWithoutCoherentMapping 实验性模块参数 (2026-05-18)

2026-05-18 新 Issue

新功能提案: 用户提议新增 NVreg_AllowP2PWithoutCoherentMapping 实验性模块参数，用于允许在无相干映射支持下启用 GPU P2P 内存访问。此功能与 DMA-BUF 和 P2P 内存路径相关，可能解决 Issue #1130 中 dmabuf 跨设备导入的部分场景。

Feature Request DMA-BUF / P2P
来源: Issue #1152

技术详情

Issue #1152 提议为 NVIDIA 开源内核模块添加一个新的实验性模块参数 NVreg_AllowP2PWithoutCoherentMapping。该参数将允许在系统不支持 CPU 相干映射的情况下，仍然启用 GPU 之间的 P2P 内存访问。这与 DMA-BUF 和 P2P 内存路径直接相关。在当前开源模块中，P2P 内存访问依赖 CPU 相干映射，这在某些平台（如部分 ARM 系统或特定主板配置）上可能不可用。如果实现，此功能可以为 Issue #1130 (Vulkan dmabuf 无法被 mlx5 导入) 提供一种可能的变通路径。涉及模块参数注册代码，可能在 kernel-open/nvidia/nvidia.c 中的模块参数定义区域。

New Bug RC Watchdog / Linux 7.0
来源: Issue #1153

技术详情

该 Issue 报告了 Turing GPU 在 Linux 7.0.x 内核上的持续性 GPU 初始化失败。环境：OpenSUSE Tumbleweed, 内核 7.0.5 或 7.0.7, 驱动 595.58.03 和 595.71.05, DKMS 构建和预签名包均受影响。同一硬件在 6.19.12 内核上无错误。关键断言：驱动初始化约 1.3 秒后触发 "GPPut < WATCHDOG_GPFIFO_ENTRIES @ kernel_rc_watchdog.c:1549"。随后 RC Watchdog 超时: "GPU is probably locked! Notify Timeout Seconds: 7"。nvidia-smi -L 初始成功，但完整诊断挂起。用户排除了驱动版本、安装方法、Secure Boot、GSP 固件、硬件退化等因素。引导配置包含 nvidia.NVreg_EnableGpuFirmware=1。理论分析：内核 7.0.x 引入的架构变化（内存映射、中断路由或 PCIe 时序）与 Turing 命令通道冲突。涉及文件路径：kernel-open/nvidia/kernel_rc_watchdog.c。

  • Blackwell 桌面 GPU 驱动稳定性 (#1151, #1045) 影响 RTX 5080/5090 用户
  • GSP 固件交互路径 (#1145, #1064) 需要系统性修复
  • 595.71.05 之后的下一版本需关注上述 Issues 修复情况

中期趋势 (2026-2028)

  • Rubin NVL72 全面投产后，开源驱动对新硬件 (NVLink 6 / NVSwitch 6 / C2C 2nd) 的适配成为关键
  • Linux 内核主线与 NVIDIA 开源驱动的兼容性需要持续协调
  • PCIe 6.0 作为 NVLink 替代方案的生态位扩展 (Astera Scorpio X + NVLink Fusion)
  • 跨设备 DMA 互操作 (GPU dmabuf → NIC) 需求增长，开源模块需补齐兼容性
  • GSP 固件架构向开源模块的适配是长期稳定性基础
  • DRM/KMS 完全取代传统 X 驱动路径，modeset=1 成为默认

本周核心洞察

1. Linux 7.0.x 兼容性危机 — 本周最重要的系统性趋势

本周 6 个新 Issues 中有 3 个直接与 Linux 7.0.x 内核相关：#1153 (Turing RC Watchdog 断言在 7.0.5/7.0.7)、#1148 (RTX 3090 Ti S3 挂起在 7.0.3)、#1117 (Blackwell s2idle 在 Linux 7.0)。Linux 7.0.x 的架构变化（内存映射、中断路由、PCIe 时序或电源管理 API）对 NVIDIA 开源驱动产生了系统性影响。建议用户在升级 7.0.x 内核前评估风险，或暂缓升级至 6.19.x 系列。

2. UVM 稳定性问题清单持续延长 — 开源模块信任危机加深

关键 Issue #1108 (UVM 内核恐慌) 已开放超过 5 周无任何修复动作。加上 #1130 (dmabuf)、#1151 (Blackwell Xid 79)、#1117 (s2idle)、#1064 (GSP 心跳)、#1030 (MMU Fault)、#1045 (Xid 62/45→119→154)，UVM 和内存管理子系统的稳定性问题清单正在显著延长。在 Arch Linux 默认采用开源模块、用户基数急剧扩大的背景下，NVIDIA 需要加速修复节奏，否则可能实质性影响开源驱动的采用信心。

3. Issue #1152 (P2PWithoutCoherentMapping) — 本周唯一的功能提案

本周仓库无任何代码提交、PR 合并或版本发布，唯一的代码级动态是 Issue #1152 提出的实验性模块参数提案。该参数允许在无 CPU 相干映射支持下启用 GPU P2P 访问，可能为 #1130 (Vulkan dmabuf → mlx5) 提供变通方案。如果 NVIDIA 采纳并实现，将是开源模块在跨设备 DMA 互操作能力上的重要补充。

4. NVLink 状态管理问题 — #1148 和 #1150 揭示深层缺陷

本周 2 个新 Issues 涉及 NVLink：#1148 揭示了 NVLink 单边关闭序列在多次 S3 循环后导致 virtualBar2.pCpuMapping NULL 的资源泄漏问题；#1150 报告 NVML 无法检索 NVLink 信息（所有链路非活动），重启后暂时恢复。这两个问题指向 NVLink 链路状态管理和电源管理路径的深层缺陷，可能影响所有使用 NVLink 的多 GPU 系统。

5. Blackwell 桌面 GPU 驱动仍不成熟 — RTX 5080 问题累积

RTX 5080 (GB203) 作为最新消费级 Blackwell GPU，在 Linux 开源驱动上的问题持续累积：#1151 (随机 Xid 79 "GPU fallen off bus"，Windows 上也复现，隔离为硅片或 GSP 微码缺陷)、#1117 (s2idle 恢复挂起)、#1045 (间歇性 Xid 62/45→119→154)。这些问题表明 Blackwell 桌面 GPU 在 Linux 上的支持仍需大量适配工作。
