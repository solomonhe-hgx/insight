摘要

本专利披露了一种自同步远程内存操作 (self-synchronizing remote memory operations) 技术,用于多处理器系统中的高效远程内存访问。该系统将远程 store 操作和同步元数据合并为单一操作单元,在传输过程中减少互连流量。在接近目标处理器的"一致性点 (point of consistency)",组件将合并操作分离,确保 store 操作在同步更新之前执行。这种设计消除了传统方案中独立的同步消息、确认和原子更新步骤,显著降低了跨处理器互连的延迟。多个同步更新可以被合并 (coalesced) 为单次原子更新,进一步优化带宽。

问题背景与技术挑战

传统远程内存操作的局限性

挑战 1: 独立的同步消息增加延迟

传统方法中,远程内存操作需要分离的同步消息、确认 (acknowledgements) 和原子更新步骤。例如,RDMA 写操作完成后,需要额外的原子操作来更新同步标志,消费者再轮询该标志。这增加了 2-3 次额外的互连事务,在低性能互连 (如跨节点 InfiniBand) 上延迟显著。

挑战 2: 互连流量冗余

每次远程写操作后跟随独立的同步更新,导致互连流量翻倍。在多处理器系统中,当多个处理单元频繁执行远程内存操作时,冗余的同步消息会消耗大量互连带宽,成为性能瓶颈。

挑战 3: 缺乏硬件级操作排序

传统方案依赖软件确保数据写入在同步更新之前完成。这需要额外的内存围栏或屏障操作,增加了软件复杂性和执行延迟。硬件层面缺乏保证 store 和 sync 操作排序的机制。

挑战 4: 同步更新无法合并

当多个处理单元向同一个同步对象发送更新时,传统方案需要逐个处理这些更新,每个更新都是一次独立的原子操作。这导致同步对象成为热点,串行化访问进一步增加延迟。

来源: US12105960B2 Background

为什么这个问题在多节点 GPU 训练中尤为关键?

在大规模分布式 LLM 训练中,梯度同步通过 NCCL over InfiniBand/RDMA 执行。每次梯度 AllReduce 完成后,需要更新同步标志通知下一轮训练开始。传统方案中,这需要额外的 RDMA 原子操作和轮询。US12105960B2 通过自同步操作将梯度写入和同步更新合并为一次 RDMA 事务,显著降低多节点训练的通信延迟。

核心设计理念: 将远程 store 操作和同步元数据合并为单一操作单元,在传输中减少互连流量。在接近目标的"一致性点",hub 组件将合并操作分离,强制排序 (store 先于 sync)。多个同步更新可以被合并为单次原子更新,进一步优化带宽。目标处理器通过轮询同步对象并在达到目标值后加载数据。

传统方案 (分离操作)

• 步骤 1: 远程 store 操作 (写入数据)

• 步骤 2: 独立同步消息 (更新标志)

• 步骤 3: 确认消息 (ack)

• 步骤 4: 原子更新 (同步对象)

互连事务: 4 次

延迟: 高 (多轮往返)

本专利方案 (自同步)

• 步骤 1: 合并操作 (store + sync 元数据)

• 步骤 2: 单次传输到一致性点

• 步骤 3: Hub 自动分离并排序

• 步骤 4: 同步更新可合并

互连事务: 1-2 次 (节省 50-75%)

延迟: 低 (单次往返)

操作合并机制 (Merge)

Store + Sync 合并流程

处理单元生成远程内存操作时,将 store 操作和同步操作合并 (merge) 为单一操作单元。合并后的操作包含:

字段 类型 作用
Store Payload Data 要写入远程内存的数据
Store Address Virtual/Physical 数据存储的目标地址
Sync Object Address Virtual/Physical 同步对象的地址 (如信号量、屏障)
Sync Update Value N-bit 同步对象的更新值 (如递增量、目标值)
Operation Type Enum 标识为自同步操作 (vs 标准操作)
Coalesce ID N-bit 用于合并多个同步更新的标识符
关键创新: 合并操作在源处理器端生成,通过互连网络作为单一事务传输。这消除了传统方案中独立的同步消息,将互连流量减少 50-75%。合并操作在接近目标的一致性点被分离,确保 store 在 sync 之前执行。

// 硬件自动合并 store + sync 为单一事务
// 在一致性点自动分离并排序

一致性点分离 (Point of Consistency Split)

Hub 组件的分离逻辑

在接近目标处理器的"一致性点 (point of consistency)",一个 hub 组件负责将合并的操作分离为独立的 store 和 sync 操作,并强制排序。

阶段 1: 接收合并操作

Hub 接收到来自互连网络的合并操作,解析操作类型、store payload、store 地址、同步对象地址和同步更新值。

阶段 2: 分离为独立操作

Hub 将合并操作分离为两个独立的内部操作:(1) store 操作: 将数据写入目标内存地址;(2) sync 操作: 更新同步对象。这两个操作在 hub 内部被标记为有序对。

阶段 3: 强制排序 (Store 先于 Sync)

Hub "将 store 操作序列排在同步更新之前"。这通过内部队列和依赖检查实现:sync 操作被阻塞,直到 store 操作完成并被确认。这确保了消费者在同步对象更新后能看到完整的数据。

阶段 4: 执行与确认

Store 操作完成后,hub 执行 sync 操作 (可能与其他 sync 更新合并)。完成后,目标处理器可以通过轮询同步对象感知操作完成。

为什么一致性点接近目标很重要?

一致性点越接近目标处理器,store 和 sync 操作之间的排序保证越强。如果一致性点在源端,store 和 sync 可能在互连中经历不同的路由延迟,导致到达目标的顺序不确定。将一致性点放在目标端 (或接近目标的 hub),可以确保 store 在 sync 之前到达目标内存,消费者在 sync 更新后一定能看到数据。

来源: US12105960B2

MMU 集成与地址转换

MMU 在远程内存操作中的角色

专利中描述了 "MMU 320 包含一组页表条目 (PTEs),用于将虚拟地址映射到物理地址"。MMU 负责地址转换并将操作转发到高速 hub。

虚拟地址到物理地址转换

当处理单元发起自同步远程内存操作时,操作中包含的是虚拟地址。MMU 查找页表条目 (PTE),将虚拟地址转换为目标物理地址。这支持远程内存访问的统一虚拟寻址 (UVA)。

转发到高速 Hub

地址转换完成后,MMU 将操作转发到适当的高速 hub (可能是本地 L2 缓存、互连 hub 或网络接口)。Hub 负责后续的合并操作分离和排序。

PTE 中的远程内存标记

专利暗示 PTE 可能包含标记,指示该页面是否支持远程内存操作和自同步语义。这允许 MMU 在地址转换时识别操作类型并路由到正确的处理路径。

来源: US12105960B2

同步更新合并 (Update Coalescing)

合并机制详解

核心创新: 专利指出 "对同一对象的多个同步更新可以被合并 (coalesced) 为单次原子更新"。这消除了传统方案中逐个处理同步更新的串行化瓶颈。

传统方案 (逐个处理)

• 8 个处理单元向同一同步对象发送更新

• 8 次独立的原子操作

• 每次原子操作需要互连事务

• 串行化访问同步对象

互连事务: 8 次

延迟: 8 × 原子操作延迟

本专利方案 (合并处理)

• 8 个处理单元向同一同步对象发送更新

• Hub 合并为 1 次原子操作

• Coalesce ID 标识可合并的更新

• 单次原子更新完成所有计数

互连事务: 1 次 (节省 87.5%)

延迟: 1 × 原子操作延迟

合并的条件与实现

  • 相同同步对象: 只有指向同一同步对象地址的更新才能被合并
  • Coalesce ID: 合并操作中包含 Coalesce ID,hub 使用该 ID 识别可合并的更新组
  • 时间窗口: Hub 在特定时间窗口内收集同一 Coalesce ID 的更新,然后执行合并后的原子操作
  • 合并逻辑: 合并可以是简单的累加 (如 8 个 +1 合并为 +8),也可以是更复杂的聚合 (如设置最大值)
  • 原子性保证: 合并后的原子操作保证所有被合并的更新在逻辑上同时生效,消费者看到最终的聚合值
来源: US12105960B2

操作降级机制 (Demotion)

何时触发降级?

降级条件: 专利指出,如果 store 操作和同步操作针对"不同的虚拟内存页面 (different virtual memory pages)",该操作将被降级 (demoted) 为标准的非同步类型。
为什么需要降级?

当 store 和 sync 针对不同内存页面时,它们可能经过不同的 MMU 转换路径、到达不同的物理内存控制器。在这种情况下,hub 无法保证 store 和 sync 的排序,因此必须降级为标准操作,由软件层确保正确的同步顺序。

// 正常情况: store 和 sync 在同一页面 → 自同步
self_sync_write(page_A_addr, data, page_A_addr + offset, sync_value)
// → Hub 可以分离并排序,保持自同步语义

// 降级情况: store 和 sync 在不同页面 → 降级为标准操作
self_sync_write(page_A_addr, data, page_B_addr, sync_value)
// → Hub 检测到不同页面,降级为标准 rdma_write
// → 软件需要额外的内存围栏确保排序

降级机制的设计智慧

降级机制体现了该专利的务实设计:不试图在所有场景下强制自同步语义,而是在无法保证排序时优雅地降级。这避免了复杂的全局排序协议,同时保证了正确性。软件层可以通过将数据和同步对象分配到同一内存页面来避免降级,从而获得自同步的性能优势。

来源: US12105960B2

与 GPUDirect RDMA 的关联

GPUDirect RDMA 架构基础

GPUDirect RDMA 是 NVIDIA 与 Mellanox 联合研发的技术,实现 GPU 显存与 RDMA 网卡之间的直接 DMA 传输,无需 CPU 介入。该技术是 US12105960B2 自同步远程内存操作的重要应用场景。

PCI BAR 映射与地址可见性

GPU 通过 PCI BAR 从 BIOS/OS 请求物理地址窗口,将其绑定到部分 VRAM。同一 Root Complex 上的 RDMA 网卡通过访问该 BAR 到达 GPU 物理地址。这为 RDMA 网卡提供了直接访问 GPU 显存的路径。

地址转换与内存锁定 (Pinning)

NVIDIA 内核驱动使用 nvidia_p2p_get_pages() 验证 GPU 虚拟地址 (如来自 cudaMalloc),将其转换为绑定到 BAR 的物理地址,并存储到 nvidia_p2p_page_table。该锁定页表被传递给 RDMA 驱动以配置其 DMA 引擎。

统一虚拟寻址 (UVA)

CUDA 将应用程序的虚拟地址空间划分为 CUDA 管理的 (GPU、CPU、FREE) 和 OS 管理的区域。这种布局允许驱动快速识别有效的 GPU 地址,避免错误映射 CPU 内存。

来源: GPUDirect RDMA 解析, GPUDirect RDMA 演进

自同步操作在 GPUDirect RDMA 中的应用

US12105960B2 的自同步远程内存操作可以直接应用于 GPUDirect RDMA 场景:

多节点 GPU 梯度同步

在分布式 LLM 训练中,梯度通过 NCCL over InfiniBand/RDMA 在节点间同步。自同步操作将梯度写入和同步标志更新合并为一次 RDMA 事务,减少 AllReduce 后的同步延迟。

GPU 显存直接 RDMA 访问

RDMA 网卡直接访问 GPU 显存 (VRAM),无需 CPU staging buffer。自同步操作确保数据写入 GPU 显存后,同步标志自动更新,通知 GPU kernel 开始计算。

零拷贝 (Zero-Copy) 优化

GPUDirect RDMA 1.0 引入了共享主机缓冲区,GPUDirect Async (CUDA 8.0) 进一步优化了控制路径。自同步操作进一步减少了控制路径的 CPU 参与,实现真正的 GPU-centric 网络。

同步机制

开发者可以使用 cuPointerSetAttribute 与 CU_POINTER_ATTRIBUTE_SYNC_MEMOPS 标志确保内存操作与 RDMA 引擎同步。自同步操作在硬件层面提供了更强的排序保证。

对 NCCL over InfiniBand 的影响

NCCL over InfiniBand 是多节点 LLM 训练的核心通信路径。传统方案中,每次 RDMA 写操作后需要独立的原子操作更新同步标志。US12105960B2 通过自同步操作将这两步合并,减少 NCCL AllReduce 的尾部延迟 (tail latency)。对于万亿参数模型的跨节点训练,这可能带来显著的吞吐量提升。

核心权利要求解读

根据专利公开文本,核心权利要求涵盖以下技术方案:

权利要求 1: 自同步远程内存操作方法

一种在多处理器系统中执行自同步远程内存操作的方法,包括:由处理单元生成 store 操作和同步操作;将 store 操作和同步操作合并为单一操作单元;通过互连网络传输合并操作;在接近目标处理器的一致性点,由 hub 组件将合并操作分离为独立的 store 和同步操作;以及强制 store 操作在同步更新之前执行。

权利要求 2: 操作合并与传输优化

一种操作合并方法,包括:将远程 store 数据和同步元数据组合为单一事务;通过互连网络传输该单一事务,减少互连流量;以及在目标端分离事务并恢复独立的 store 和同步操作。

权利要求 3: 同步更新合并

一种同步更新合并方法,包括:接收指向同一同步对象的多个同步更新;使用 Coalesce ID 识别可合并的更新组;在时间窗口内收集更新;以及将多个更新合并为单次原子更新。

权利要求 4: MMU 集成与地址转换

一种 MMU 集成方法,包括:MMU 包含页表条目 (PTEs) 用于将虚拟地址映射到物理地址;支持远程内存访问的统一虚拟寻址 (UVA);以及将转换后的操作转发到高速 hub。

权利要求 5: 操作降级机制

一种操作降级方法,包括:检测 store 操作和同步操作是否针对不同的虚拟内存页面;如果针对不同页面,将自同步操作降级为标准的非同步类型;以及由软件层确保正确的同步顺序。

权利要求 6: 目标轮询与数据加载

一种目标端同步方法,包括:目标处理器轮询同步对象;当同步对象达到目标值时,确认数据已就绪;以及从远程内存加载数据。该方法支持异步通信模型,消费者在数据就绪后立即开始处理。

影响分析与未来展望

对 RDMA 通信延迟的影响

该专利将 RDMA 写操作后的同步延迟从 4 次互连事务减少到 1-2 次,节省 50-75%。在 InfiniBand 跨节点场景中,这可能降低尾部延迟 (tail latency) 数百纳秒,对 NCCL AllReduce 性能有直接影响。

对 NCCL over InfiniBand 的影响

自同步操作为 NCCL over InfiniBand 提供了硬件级优化基础。通过合并梯度写入和同步更新,NCCL 可以减少 AllReduce 后的同步开销,提升多节点训练的吞吐量。未来 NCCL 版本可能直接利用该专利的自同步机制。

对大规模 LLM 训练的影响

在万亿参数模型的跨节点训练中,梯度同步是主要的通信瓶颈。自同步操作减少每次梯度 AllReduce 的同步延迟,对于 1000+ GPU 的训练集群,累积的延迟节省可能达到秒级,显著缩短训练时间。

对 GPU 架构设计的影响

该专利揭示了 NVIDIA GPU 架构中 MMU、DMA 控制器和互连 hub 的协同设计思路。自同步机制可以作为现有 RDMA 栈的"附加层"实现,无需重新设计网络协议。这种设计可能被集成到 Hopper、Blackwell 及后续架构中。

更新合并的带宽优化

同步更新合并机制将多处理器的原子操作从 O(N) 减少到 O(1)。在大规模并行场景中 (如 72-GPU GB200 NVL72),这可能显著减少同步对象的热点竞争,提升整体系统的并发性能。

专利护城河

该专利由 NVIDIA GPU 架构核心团队 (Olivier Giroux 等) 发明,保护了 NVIDIA 在自同步远程内存操作的核心技术。竞争对手需要开发替代的同步方案或获得授权,构成了重要的专利壁垒。这也是 NVIDIA 通过 Mellanox 收购获得的技术协同。

总结

US12105960B2 是 NVIDIA 在多处理器系统远程内存访问领域的核心专利,通过自同步操作将 store 和 sync 合并为单一事务,在一致性点分离并强制排序,将互连流量减少 50-75%,显著降低 RDMA 通信延迟。该专利与 MMU 深度集成,支持统一虚拟寻址和远程内存访问的地址转换。

在 GPUDirect RDMA 和 NCCL over InfiniBand 场景中,该专利的自同步机制可以直接应用于多节点 LLM 训练的梯度同步,减少跨节点通信的尾部延迟。更新合并机制进一步优化了同步对象的热点竞争,将 O(N) 的原子操作减少到 O(1)。

对于竞争对手,该专利构成了自同步远程内存操作的专利壁垒。对于学术界和开源社区,该专利揭示了 NVIDIA GPU 远程内存同步机制的底层原理,有助于优化分布式训练框架和集合通信库。
