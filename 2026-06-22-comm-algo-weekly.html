Roadmap Issue #2090
设计原理
NCCL Q2 2026 Roadmap 核心方向详细设计原理
核心问题：NCCL v2.30+ 需要在保持向后兼容的同时引入多项架构级改进：Device API 需要从同步阻塞模型迁移到异步非阻塞；CUDA Graph 需要支持通信操作的捕获和重放；Zero-SM collectives 需要完全不占用 SM 的集合操作。
设计原理：(1) 非阻塞 GIN：当前 gin.get 是阻塞操作——GPU kernel 必须等待远端数据到达。非阻塞版本将引入 completion-based 语义，允许 GPU 在等待数据到达期间继续执行其他计算。(2) Zero-SM AllGather：使用 RMA CPU proxy + Copy Engine 实现集合操作，完全不启动 GPU SM kernel。数据搬运由 Copy Engine（CE）完成，释放的 SM 全部用于用户计算。这在通信-计算并发场景下可提升 15-30% 的 SM 利用率。(3) Cost model 重构：当前的算法选择（ring vs tree vs NVLS）基于简单的消息大小阈值。新的 cost model 将考虑 NVLink 拓扑、NIC 布局、消息大小、并发操作数量等多维度因素，实现自动最优算法选择。
适用场景：(1) 大规模训练（100k+ GPU）——非阻塞 GIN 和 cost model 重构直接提升大规模扩展效率；(2) 推理服务——Zero-SM AllGather 允许通信和推理计算完全并行；(3) CUDA Graph 部署——通信操作被捕获到 Graph 中减少 launch 开销。

PR #665
设计原理
PP send/recv 多 QP 负载均衡详细设计原理
核心问题：InfiniBand 中每个 QP 绑定到一个物理端口。当 PP send/recv 只使用单个 QP 时，所有通信流量被限制在一条物理链路上——即使节点有多个 RDMA NIC（如 8-port IB 配置）。在 bonding 模式下，多端口本应提供聚合带宽，但单 QP 导致只有 1/N 的端口被利用。
设计原理：(1) 多 QP 创建：为每对通信 rank 创建多个 QP（每个 RDMA NIC 一个），而非之前的单个 QP。(2) 流量分散策略：数据按 block 或 message 级别轮询分发到不同 QP——每个 QP 通过独立的 NIC 端口发送。(3) 完成排序：多 QP 可能导致消息乱序到达。PP send/recv 是点对点操作，接收方需要确保消息按正确顺序处理——通常通过 per-QP sequence number 或消息级别 tag 实现。
适用场景：多 rail RDMA 集群（如 8-port IB 的 DGX H100/H200）——PP 通信带宽可提升接近 N 倍（N = NIC 数量）。对 Pipeline Parallelism 的 bubble reduction 有直接帮助。

Issue #668: V2 barrier 设计 vs V1 双缓冲机制讨论

Issue #668
设计原理
V2 Barrier vs 双缓冲设计对比详细设计原理
核心问题：DeepEP V1 使用 double-buffer 机制——维护两组 buffer 交替使用，一组用于当前通信，另一组用于上一通信的清理/同步。这种方式简单但存在限制：(1) buffer 数量固定为 2，无法流水线化超过 2 个阶段；(2) 双缓冲的切换点隐含了全局同步语义，增加了延迟。
设计原理：V2 将同步语义从 buffer 管理中解耦，改为显式的 pre-sync barrier（通信前同步）和 post-sync barrier（通信后同步）。优势：(1) 灵活的流水线：barrier 可以独立于 buffer 数量放置，支持任意深度的流水线；(2) 更精确的同步范围：pre-sync 只等待前一个阶段的特定操作完成，而非全局 double-buffer 切换；(3) TMA 兼容性：Hopper/Blackwell 的 TMA 使用 mbarrier 硬件原语，显式 barrier 与 TMA 的 mbarrier 机制天然对齐。(4) 代价：显式 barrier 需要额外的 GPU 指令（mbarrier.arrive/wait），在低消息数场景可能略慢于 double-buffer 的隐式同步。
适用场景：V2 barrier 设计在高并发、深度流水线场景（如 MoE 推理的多层 dispatch/combine）中优势明显；V1 double-buffer 在简单场景下实现更简洁。

设计原理
Multicast Handle 基础设施详细设计原理
核心问题：NVSHMEM 3.7.0 引入了 "logical endpoint" 概念作为 fabric-PTX unicast 通信的抽象。3.8.0 正在将这一抽象扩展到 multicast 操作——reduce、pull-reduce、fcollect 等集合操作都需要 multicast handle 来管理 multicast group 的生命周期。之前的实现直接使用 IB multicast group，缺乏统一的资源管理和错误处理。
设计原理：(1) Multicast Handle 抽象：将 IB multicast group 的创建、成员管理、消息路由封装为统一的 handle 接口。不同传输层（IBRC、IBGDA、NVLS）可以提供各自的 handle 实现，但上层 API 使用统一的 handle 接口。(2) Pull-reduce handle：在 reduce 操作中，多个 PE 向一个目标 PE 发送数据并进行归约。Pull-reduce 模式下，目标 PE 主动从源 PE "拉取"数据并执行归约——适合源 PE 数量多的场景，避免目标 PE 成为瓶颈。(3) NVLS fcollect：NVLS (NVLink SHARP) 是 NVIDIA 的硬件加速归约机制。fcollect 是全收集操作——所有 PE 的数据被归约后广播到所有 PE。将 NVLS fcollect 切换到 multicast handle 使得 NVLS 和 IB multicast 可以共享同一套上层逻辑。(4) 逻辑端点生命周期：multicast logical endpoint 管理从创建、加入组、通信、到退出的完整生命周期——类似 TCP socket 的 connect/send/close 但面向 multicast。
适用场景：(1) 大规模 AllReduce——multicast handle 允许 IB switch 级别的硬件归约（IB SHARP），减少网络流量；(2) Broadcast/ReduceScatter——multicast 天然适合一对多数据分发；(3) 混合传输——NVLS (节点内) + IB multicast (跨节点) 通过统一 handle 接口无缝切换。

设计原理
NIXL AMD ROCm 支持 + 存储后端扩展详细设计原理
核心问题：NIXL 之前仅支持 NVIDIA CUDA 平台——使用 CUDA runtime API 进行内存注册、GPUDirect RDMA 进行数据传输。但 Meta 等大规模用户同时部署 NVIDIA 和 AMD GPU，需要一个跨平台的通信库。同时，推理场景需要高效的 KV cache offloading 到存储系统，需要原生的存储后端支持。
设计原理：(1) ROCm/HIP 适配层：NIXL 的内存注册和数据传输接口通过 HAL (Hardware Abstraction Layer) 抽象。CUDA 路径使用 cudaMalloc/cudaMemcpy，ROCm 路径使用 hipMalloc/hipMemcpy。HAL 在编译时根据 __HIP_PLATFORM_AMD__ 宏选择对应实现。(2) DDN Infinia 后端：DDN Infinia 是高性能 NVMe-over-Fabric 存储系统。NIXL 通过 plugin 架构集成——注册存储后端为一种 "transfer engine"，KV cache 可以直接通过 RDMA 写入/读取 Infinia 存储节点，绕过文件系统层。(3) Path-based 文件注册：之前 NIXL 需要通过 fd (file descriptor) 注册文件——在多进程环境中 fd 传递复杂。Path-based 模式允许每个进程独立通过文件路径注册，简化部署。(4) GIL 释放：makeXferReq 是 CPU 密集操作（构造 RDMA work request），之前持有 Python GIL 导致多线程 Python 程序无法并行。释放 GIL 后，多个 Python 线程可以同时创建传输请求。
适用场景：(1) 混合 GPU 集群——NVIDIA + AMD GPU 共存的数据中心统一使用 NIXL；(2) KV cache offloading——通过 DDN Infinia 后端将推理 KV cache 高速写入存储；(3) 多租户推理服务——Python GIL 释放允许高并发请求处理。

torchcomms
设计原理
GPU Runtime Seam + 多厂商抽象层详细设计原理
核心问题：torchcomms 之前深度绑定 NVIDIA CUDA 生态——使用 NVML 查询 GPU 拓扑、CUDA runtime 管理内存、NCCL 执行集合操作。Meta 正在大规模部署 AMD MI300X/MI400 GPU，需要一个能在 NVIDIA 和 AMD GPU 上运行同一套通信代码的架构。
设计原理：(1) GPU Runtime Seam：通过 oss_gpu_cpp_library 抽象层，将 GPU 厂商特定的 API（NVML vs AMD SMI、CUDA vs HIP）封装在统一接口后面。编译时通过 OSS allowlist 选择具体实现——NVIDIA 构建使用 NVML，AMD 构建使用 SMI (System Management Interface)。(2) NVML Factory 模式：GPU 拓扑发现使用 Factory 模式——NVML factory 在 NVIDIA 平台上调用 nvmlDeviceGetTopologyNeighbors，AMD no-op stub 返回默认拓扑。这允许同一套拓扑发现代码在不同平台上编译和运行。(3) AMD RDMA/GPUDirect：AMD 的 RDMA 实现基于 ROCm RDMA verbs（与 IB verbs 兼容），GPUDirect 通过 ROCm P2P 内存映射实现。torchcomms 的传输层通过 HAL 接口调用这些原语，不直接依赖 CUDA。(4) 战略意义：Meta 不再将通信库与 GPU 厂商绑定——这意味着 torchcomms 可以同时在 NVIDIA 和 AMD GPU 集群上部署，降低多厂商环境下的维护成本。
适用场景：(1) Meta 混合集群——同时运行 NVIDIA H100 和 AMD MI300X 的数据中心；(2) 迁移路径——从 NVIDIA 向 AMD 渐进迁移时保持通信层不变；(3) 开源社区——其他使用 AMD GPU 的组织可以直接使用 torchcomms。

设计原理
CSV/JSON NCCL 算法调优器详细设计原理
核心问题：NCCL 的集合操作算法选择（ring vs tree vs NVLS vs collnet）目前基于硬编码的消息大小阈值。但最优算法取决于多维度因素：消息大小、GPU 数量、NVLink 拓扑、NIC 布局、网络拥塞程度。Meta 的 100k+ GPU 集群需要数据驱动的、可配置的算法选择策略。
设计原理：调优器读取 CSV/JSON 配置文件，其中包含不同配置（消息大小范围 x GPU 数量 x 拓扑类型）下的最优算法选择。配置文件可以通过自动化 benchmark 生成——在部署时运行一组基准测试，测量每种算法的实际性能，生成最优配置。运行时，调优器根据当前操作参数查找配置表选择算法。这比硬编码阈值更精确，且可以随硬件升级更新配置而不需要修改代码。
适用场景：(1) 大规模训练——不同训练阶段（prefill vs decode）使用不同配置；(2) 硬件升级——新 GPU 或 NIC 部署后更新配置文件即可；(3) A/B 测试——对比不同算法选择的性能影响。

设计原理
RMA Rendezvous 协议详细设计原理
核心问题：UCX 的 RMA (Remote Memory Access) 操作对小消息使用 eager 协议（直接发送数据），但对大消息（>阈值）eager 协议会占用大量接收方 buffer。Rendezvous 协议通过 "握手-确认" 机制解决这一问题——先发送控制消息（包含源地址和 rkey），接收方主动拉取数据。
设计原理：(1) Rendezvous GET：发送方将数据留在本地 buffer，发送控制消息（CTS, Clear-To-Send）给接收方。接收方使用 RDMA READ 从发送方拉取数据。优势：接收方控制拉取速率，避免 buffer overflow。(2) Rendezvous PUT：发送方发送控制消息，接收方分配目标 buffer 并返回地址。发送方使用 RDMA WRITE 推送数据。优势：接收方可以选择最优的目标内存位置（如 NUMA-local memory）。(3) 与之前实现的差异：之前 UCX 只有 tag-matching 层面的 rendezvous，RMA 层面缺少完整的 rendezvous 协议。新增的实现支持 rkey exchange 和 multi-lane 传输。
适用场景：(1) 大消息 RMA——超过 64KB 的 GPU 内存传输；(2) 多 lane 传输——利用多个 NIC 端口并行传输大消息；(3) 拥塞控制——rendezvous 协议天然支持反压（backpressure），避免网络拥塞。

设计原理
WarpSpeed 自动调优 + gfx1250 Shader 预绑定详细设计原理
核心问题：RCCL 的集合操作有多种 kernel 实现（不同 block size、unroll factor、算法变体），运行时选择最优 kernel 对性能至关重要。之前使用简单的启发式规则（消息大小 + GPU 数量），但在不同 GPU 架构（gfx942 vs gfx1250）和网络配置下，最优 kernel 差异很大。
设计原理：(1) WarpSpeed 自动调优：在首次运行时或初始化阶段，对当前硬件配置运行一组 micro-benchmark，测量每种 kernel 变体的实际性能。结果缓存到配置文件，后续操作直接查表选择。这类似于 NCCL 的 autotune 但更系统化。(2) gfx1250 shader 预绑定：RDNA4 (gfx1250) 的 shader 编译时间较长——如果在运行时编译会导致首次 kernel launch 延迟数百毫秒。预绑定将 shader 二进制直接嵌入到 RCCL 库中，消除运行时编译。(3) HSA 直接链接：之前 RCCL 通过 dlopen 动态加载 HSA runtime——每次 API 调用需要间接寻址。直接链接将 HSA 符号在链接时解析，减少函数调用开销。
适用场景：(1) 新一代 AMD GPU——gfx1250 (RDNA4) 的首次部署需要预绑定 shader 避免冷启动延迟；(2) 性能敏感场景——WarpSpeed 自动调优可以在特定拓扑下找到比默认启发式快 10-20% 的 kernel；(3) 生产环境——A/B 性能回归 CI 确保每次代码变更不引入性能退化。
