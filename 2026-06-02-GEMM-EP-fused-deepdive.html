GEMM-EP Fused 三大技术分类

分类 1: 计算-通信 Overlap（流水线）

Dispatch/Combine 与 GEMM 在不同 CUDA stream 或 thread block 中并行执行。GPU 同时进行计算和网络传输。

  • 代表：MegaScale-MoE, Comet, Lancet
  • 核心：tile-level barrier + double buffer
  • 延迟隐藏率：86.5% (Comet)

分类 2: 算子 Fused（单一 kernel）

将 Dispatch + GEMM + SwiGLU + Combine 融合为单个 CUDA kernel，消除 kernel launch 和中间 buffer 开销。

  • 代表：DeepGEMM Mega MoE, SonicMoE, ClusterFusion
  • 核心：FP8 tensor core + NVLink symmetric buffer
  • 峰值性能：1550 TFLOPS (H800)

分类 3: 通信感知优化（算法层）

在 MoE 路由、token 放置、expert 分组等算法层面优化，减少通信量或提升通信效率。

  • 代表：Occult, X-MoE, NCCL EP
  • 核心：expert placement + redundancy bypass
  • all-to-all 延迟减少：50.7% (X-MoE)

MegaScale-MoE: 大规模通信高效 MoE 训练

arxiv 2505.11432 | 字节跳动

核心: 实现两类 Overlap 内核：all-gather+scatter+GroupedGEMM 融合和 all-to-all+GroupedGEMM 融合。使用 tile-level barrier 实现通信与计算的细粒度流水线，迭代时间降低 7.1%-12.9%。

论文原文 PDF 设计原理
MegaScale-MoE GEMM-EP 融合详细设计原理
核心问题：MoE 训练中，每层需要执行 Dispatch（token 路由到 expert）→ GEMM（expert 计算）→ Combine（结果聚合）三个操作。传统实现中这三个操作串行执行：Dispatch 完成所有 token 传输后才开始 GEMM，GEMM 完成所有 expert 计算后才开始 Combine。这导致 GPU 的 Tensor Core 和网络 NIC 无法同时工作，通信时间完全浪费。
设计原理：(1) Tile-level Barrier：将输入 tensor 切分为多个 tile（如 128x128 的小块），每个 tile 独立管理通信和计算状态。GPU 使用 warp-level barrier 同步——当一个 tile 的通信完成后，立即触发该 tile 的计算，无需等待其他 tile。这实现了 tile 粒度的流水线。(2) All-gather+Scatter+GroupedGEMM 融合：传统 GroupedGEMM 需要先将所有 expert 的输入 all-gather 到每个 GPU，然后 scatter 到各自的 expert。MegaScale 将 all-gather 的每个 slice 与 scatter+GEMM 融合——每次 all-gather 接收一个 slice 后，立即对已接收的数据执行 scatter 和 GEMM，无需等待完整 all-gather。(3) Token 重排序：为 GroupedGEMM 重新排列 token 顺序，最小化跨 rank 的依赖——同一 GPU 上的 token 优先计算，跨 GPU 的 token 在通信完成后计算。这减少了 barrier 等待时间。
性能数字：在 352B MoE 模型上，相比 Megatron-LM 基线，迭代时间降低 7.1%-12.9%；combined operator（all-gather+scatter+GEMM）加速 1.2-4.7x；在 128 GPU 上达到 1.88x 整体加速。
适用场景：(1) 大规模 MoE 训练——128+ GPU 的集群效果最佳；(2) 高通信带宽场景——NVLink/IB 带宽充足时 overlap 效果更好；(3) 深 MoE 模型——层数越多，每层节省的时间累积越大。

NCCL EP: 统一 Expert Parallel 通信 API

arxiv 2603.13606 | NVIDIA

核心: NVIDIA 推出 NCCL EP 统一 API，融合 DeepEP（低延迟 LL）和 Hybrid-EP（高吞吐 HT）内核。通过 staged execution（send_only=1）实现 GEMM 与 dispatch/combine 的 overlap。内存占用比 DeepEP 降低 ~14x。

论文原文 中文解读 设计原理
NCCL EP GEMM-EP 融合详细设计原理
核心问题：MoE 训练/推理中，dispatch 和 combine 是两个 all-to-all 通信操作。传统 NCCL 的 ncclAllToAll 是同步阻塞的——通信完成前无法执行 GEMM。DeepEP 虽然提供了非阻塞的 dispatch/combine，但依赖 NVSHMEM 后端（v1）或 NCCL GIN（v2），与标准 NCCL API 不兼容。NCCL EP 的目标是在 NCCL 生态内提供统一的 EP API。
设计原理：(1) 双层内核架构：NCCL EP 融合两种内核设计——LL（Low-Latency）模式使用 full all-to-all mesh + double-buffering，通过 NCCL GIN 实现 GPU-initiated RDMA；HT（High-Throughput）模式使用 hierarchical aggregation（先 NVLink 域内聚合，再 inter-node RDMA），采用 warp-specialized persistent kernel（不同 warp 负责 load/network/store）。(2) Staged Execution：API 层面，ncclEpDispatch 支持 send_only=1 标志——先发起网络传输（send），然后 GPU 立即执行 GEMM（计算已到达的 token），最后再执行 receive。这使得 GEMM 与网络传输完全 overlap。(3) Handle 复用：训练场景中，同一个 dispatch handle 在 forward 和 backward 之间复用——forward 的 routing 信息（topk_idx, token 分布）被缓存，backward 直接使用，避免重新计算 routing。
GEMM-EP Overlap 流水线：LL 模式：dispatch(send_only=1) → GEMM(local tokens) → dispatch(complete) → GEMM(remote tokens) → combine。HT 模式：micro-batch 流水线——dispatch(micro-batch-0) → combine(micro-batch-0) || dispatch(micro-batch-1) → combine(micro-batch-1)。
性能数字：LL Dispatch：多节点（2-8）持平或超过 DeepEP；单节点略低约 4%。LL Combine：1-4 节点落后 DeepEP 5-12%，8 节点收敛。vLLM 集成：端到端吞吐比 DeepEP+NVSHMEM 低 7-10%。内存：LL 内核设计实现内存占用降低 ~14x。
适用场景：(1) 训练——NCCL EP 的 handle 复用在训练中效果显著，避免每次 forward/backward 重新 routing；(2) 推理——micro-batch 流水线适合 decode 阶段的小 batch 推理；(3) 可移植性——替换 NVSHMEM/IBGDA 后端为 NCCL GIN，跨平台兼容性更好。

Comet: 细粒度计算-通信重叠

arxiv 2502.19811 | 清华大学

核心: 使用 shared tensor decomposition 实现细粒度 GEMM-通信重叠。Layer0 按 source rank 重排 token，Layer1 按 column-wise 计算，隐藏 86.5% 通信延迟。自适应 thread block 分配，layer 加速 1.96x，端到端加速 1.71x。

论文原文 设计原理
Comet GEMM-通信重叠详细设计原理
核心问题：传统 MoE 的 GEMM-通信 overlap 是粗粒度的——整个 dispatch 完成后再开始 GEMM，或整个 GEMM 完成后再开始 combine。这导致 overlap 效率低下：(1) dispatch 的尾延迟（straggler）阻塞整个 GEMM；(2) GEMM 的最后一个 token 计算完成后才触发 combine。
设计原理：(1) Shared Tensor Decomposition：将 MoE 的输入 tensor 按 source rank（token 来自哪个 GPU）和 expert（token 要去哪个 expert）两个维度分解为多个 sub-tensor。每个 sub-tensor 独立管理通信和计算状态。(2) Layer0 优化（Dispatch 侧）：按 source rank 重排 token——先处理已到达的 token（来自本地 GPU），后处理未到达的 token（来自远程 GPU）。这样 GEMM 可以先计算本地 token，远程 token 的通信与本地计算 overlap。(3) Layer1 优化（Combine 侧）：按 column-wise 计算——GEMM 的每个输出列独立，先完成的列立即触发对应的 combine 传输，无需等待整行计算完成。(4) 自适应 Thread Block 分配：根据每个 expert 的 token 数量动态分配 CUDA thread block——token 多的 expert 分配更多 thread block，避免 thread block 饥饿。
性能数字：隐藏 86.5% 通信延迟；layer 级加速 1.96x；端到端训练加速 1.71x。
适用场景：(1) 训练场景——forward 和 backward 都需要 dispatch/combine，Comet 的两层优化同时适用；(2) 高通信延迟环境——inter-node 通信延迟高时，细粒度 overlap 收益更大；(3) 不均匀 token 分布——MoE 的 token 分布天然不均匀，自适应 thread block 分配减少负载不均衡。

Hybrid-EP: NVIDIA 混合 Expert Parallel 通信

NVIDIA Blog | 2025

核心: Hybrid-EP 将 GEMM 与 EP 通信融合，使用 CUDA block 实现计算-通信 overlap。在 DeepSeek-V3 上实现 514% 吞吐提升，仅用 416 SM 即可饱和带宽。在 DGX Hopper 和 Grace Blackwell 上验证。

NVIDIA Blog 设计原理
Hybrid-EP GEMM-EP 融合详细设计原理
核心问题：传统 MoE EP 通信使用独立的 NCCL all-to-all kernel，完全占用 SM（通常 64-128 SM），导致 GEMM 无法同时进行。在 DeepSeek-V3 这样的大规模 MoE（256 experts, top-8）中，每层两次 all-to-all 的通信时间占总时间的 40-60%。
设计原理：(1) Warp-Specialized Persistent Kernel：Hybrid-EP 使用 persistent kernel 设计——一个长期运行的 CUDA kernel 占据固定数量的 SM（如 416 个 SM 中的 64 个），内部使用 warp specialization：部分 warp 负责从 shared memory 加载 token（load warp），部分 warp 负责 RDMA 传输（network warp），部分 warp 负责写入结果（store warp）。这种设计使通信和计算在同一 kernel 内 overlap。(2) Hierarchical Aggregation：在 NVLink 域内先做 local aggregation（多个 GPU 的 token 在域内聚合），减少 inter-node 传输量。例如 8 GPU 节点中，先将 8 个 GPU 的 token 通过 NVLink 聚合到 2 个 GPU 上，再通过 IB 发送到其他节点——IB 传输量减少 4x。(3) CUDA Block 级 Overlap：将 GEMM 按 CUDA block 切分，每个 CUDA block 的 GEMM 完成后立即触发对应的通信——而非整个 GEMM 完成后才触发。这实现了 block 粒度的流水线。
性能数字：DeepSeek-V3 吞吐提升 514%；仅 416 SM 即可饱和网络带宽；DGX Hopper 和 Grace Blackwell 双平台验证。
适用场景：(1) 大规模 MoE 训练——256+ experts 的场景；(2) 多节点训练——inter-node 通信是瓶颈时，hierarchical aggregation 效果显著；(3) Blackwell 架构——Grace Blackwell 的 NVLink 5.0 使 hierarchical aggregation 更高效。

SonicMoE: Blackwell IO 与 Tile-aware 优化

tridao.me | 2026

核心: SonicMoE 利用 Blackwell 的 TMEM double-buffering 隐藏 IO/compute 延迟，将 gather 操作融合进 GEMM 减少 24% HBM 流量。使用 CLC schedulers 和 2CTA MMA，在 B300 上比 DeepGEMM 高 54-35% TFLOPS。

SonicMoE Blog OpenReview 设计原理
SonicMoE GEMM-EP 融合详细设计原理
核心问题：Blackwell 架构引入了 TMEM（Tensor Memory）——一种专用的 SRAM 用于存储 GEMM 的中间结果。但 TMEM 的容量有限，当 MoE 的 expert 数量多（如 256 experts）时，每个 expert 的权重矩阵需要分多次加载到 TMEM，导致 IO 瓶颈。传统 GroupedGEMM 在每个 expert 的 GEMM 之间没有 overlap IO 和 compute。
设计原理：(1) TMEM Double-Buffering：使用两个 TMEM buffer——buffer A 正在被 tensor core 计算时，buffer B 同时在加载下一个 expert 的权重。这隐藏了 HBM → TMEM 的加载延迟。(2) Gather-Fused GEMM：传统流程是 dispatch → gather（将 token 从各 GPU 收集到本地）→ GEMM。SonicMoE 将 gather 融合进 GEMM 的 prologue——在加载 A 矩阵（token）时直接通过 NVLink 从远程 GPU gather，而非先全部 gather 到本地再计算。这减少 24% HBM 流量（因为 gather 的数据直接进 shared memory 而非 HBM）。(3) CLC Scheduler + 2CTA MMA：Blackwell 的 CLC（Cluster Level Compute）scheduler 允许一个 CTA cluster 内的多个 CTA 协同工作。SonicMoE 使用 2CTA MMA——两个 CTA 共同完成一个 GEMM tile，提升 tensor core 利用率。(4) Epilogue 优化：对于 dH（hidden 维度梯度）等重型 epilogue，利用 Blackwell 的 double-buffering 隐藏 epilogue 延迟。
性能数字：B300 上比 DeepGEMM 高 54-35% TFLOPS；HBM 流量减少 24%；TFLOPS 仅下降 11%（vs 理论峰值）。
适用场景：(1) Blackwell GPU——B200/B300 专属优化，Hopper 架构不适用；(2) 大规模 MoE 推理——256 experts 的场景，IO 是主要瓶颈；(3) FP4/FP8 精度——SonicMoE 支持 Blackwell 的 NVFP4 精度。

ClusterFusion: 集群级 Collective 算子融合

NeurIPS 2025

核心: 通过 ClusterReduce/Gather 将 QKV、Attention、Output 的 GEMM 与 all-reduce/all-gather 融合为单一算子。利用 H100 的 DSMEM（Distributed Shared Memory）实现集群级通信。加速 1.61x vs SOTA。

论文原文 GitHub 设计原理
ClusterFusion GEMM-Collective 融合详细设计原理
核心问题：传统 Transformer 中，QKV projection → Attention → Output projection 之间需要 all-reduce（TP）或 all-gather（SP）通信。每次通信都是独立的 kernel launch，引入 kernel launch overhead（~5μs）和全局内存往返（HBM write then read）。在 decode 阶段，由于 batch size 小，GEMM 的算术强度低，通信时间占比高达 30-50%。
设计原理：(1) ClusterReduce：将 all-reduce 融合进 GEMM 的 epilogue——GEMM 的每个输出 tile 计算完成后，立即通过 DSMEM 与其他 GPU 的对应 tile 做 reduce，而非写入 HBM 后再 all-reduce。这消除了 HBM 的中间写入。(2) ClusterGather：将 all-gather 融合进 GEMM 的 prologue——GEMM 在加载输入时直接通过 DSMEM 从其他 GPU gather 需要的 slice，而非先 all-gather 到本地 HBM。(3) DSMEM 利用：H100 的 NVLink 支持 GPU 之间的 distributed shared memory（DSMEM）——一个 GPU 的 shared memory 可以被另一个 GPU 直接访问。ClusterFusion 利用 DSMEM 实现 GPU 之间的细粒度通信，避免了 HBM 的中间存储。
适用场景：(1) LLM 推理——decode 阶段 batch size 小，GEMM 算术强度低，融合收益大；(2) Tensor Parallel 训练——TP 中的 all-reduce 可以通过 ClusterReduce 融合；(3) MLA（Multi-head Latent Attention）——DeepSeek 的 MLA 需要特殊的 all-gather 模式，ClusterFusion 支持 MLA。

X-MoE: 可扩展 MoE 训练通信优化

arxiv 2508.13337

核心: 三大技术：Padding-Free Token 存储消除 zero-padding 通信；Redundancy-Bypassing Dispatch 仅发送 pilot token 跨节点再本地复制；Sequence-Sharded MoE Blocks 将序列分片减少 dispatch/combine 内存。All-to-all 延迟降低 50.7%，吞吐提升 1.42x。

论文原文 GitHub 设计原理
X-MoE 通信优化详细设计原理
核心问题：传统 MoE 系统中，token 需要 padding 到固定大小（为了对齐 GEMM 的 tile size），导致大量 zero-padding 通信（发送无用的 0 数据）。同时，dispatch 需要将所有 token 通过 all-to-all 发送到目标 expert 所在的 GPU，即使某些 token 的目标 expert 在本地 GPU 上已有副本。
设计原理：(1) Padding-Free Token (PFT) 存储：使用 cross-platform Triton kernel 实现变长 token 存储——token 之间无 padding，紧密排列。通信时只发送有效 token 数据，消除 zero-padding 的带宽浪费。(2) Redundancy-Bypassing Dispatch (RBD)：在 MoE 中，top-k routing 意味着每个 token 被发送到 k 个 expert。如果多个 expert 在同一 GPU 上（如 EP size < total experts），传统方案会将 token 复制 k 份分别发送。RBD 只发送 1 份 "pilot token" 到目标 GPU，目标 GPU 收到后在本地复制 k 份——跨节点通信量减少 k/x（x 为本地 expert 数）。(3) Sequence-Sharded MoE Blocks (SSMB)：在 MoE 层内将序列维度分片到多个 GPU——每个 GPU 只处理序列的 1/TP_size 部分。这减少了 dispatch/combine 的内存占用（因为每个 GPU 只需要存储部分 token 的 routing 信息）。
性能数字：在 1024 AMD MI250X GPU 上扩展到 545B 参数——比先前系统大 10x。All-to-all 延迟降低 50.7%；RBD 减少 52.5% dispatch 时间；吞吐提升 1.42x。
适用场景：(1) 超大规模 MoE——500B+ 参数，1000+ GPU；(2) AMD GPU 集群——X-MoE 基于 Triton，跨平台兼容 AMD/Intel GPU；(3) 高 EP size——256+ experts 的场景，RBD 的冗余 bypass 效果显著。

Lancet: 全图计算-通信重叠

MLSys 2024

核心: Lancet 将 overlap 范围从单个 MoE 层扩展到整个计算图——非 MoE 计算（如 attention、FFN）与 MoE 的 all-to-all 通信重叠。前向传播中启用非 MoE 计算与 dispatch/combine overlap，反向传播中启用 gradient 计算与通信 overlap。

论文原文 PDF 设计原理
Lancet 全图 Overlap 详细设计原理
核心问题：之前的 GEMM-EP overlap 工作（如 Megatron-LM）仅在单个 MoE 层内做 overlap——dispatch 与 GEMM 重叠。但 MoE 层之外还有大量计算（attention、residual add、layernorm）可以与 all-to-all 通信重叠。传统方案中这些计算与 all-to-all 是串行的。
设计原理：(1) Whole-Graph Scheduling：Lancet 分析整个训练计算图，识别出所有可以与 all-to-all 通信重叠的计算节点。在 MoE 层中，dispatch 可以与上一层的 attention 计算重叠；combine 可以与下一层的 pre-computation 重叠。(2) Dependency-Aware Overlap：Lancet 使用细粒度的 dependency analysis——不是整个 all-to-all 与整个 attention 重叠，而是将 all-to-all 切分为多个 micro-batch，每个 micro-batch 的通信与对应的 attention slice 重叠。这确保了数据依赖的正确性。(3) Backward Overlap：反向传播中，gradient 计算（如 weight gradient、data gradient）可以与 dispatch/combine 重叠。Lancet 将 weight gradient 计算分为多个 stage，每个 stage 与通信 stage 流水线执行。
适用场景：(1) 深层 MoE 模型——层数越多，非 MoE 计算越多，可重叠的资源越多；(2) 大 batch 训练——batch 越大，attention 等计算的时间越长，与通信重叠的空间越大；(3) 混合并行——TP+EP 场景中，TP 的 all-reduce 也可以与 EP 的 all-to-all 重叠。

Megatron Core: 可扩展 MoE 训练系统

arxiv 2603.07685 | NVIDIA

核心: Megatron-Core 使用 DeepEP 和 HybridEP 最大化 EP 带宽，通过"计算-通信 Overlap"隐藏 all-to-all 延迟——在路由其他 token 的同时计算本地 token。在 GB200/GB300 上 DeepSeek-V3-685B 达到 1233 TFLOPS/GPU，Qwen3-235B 达到 974 TFLOPS/GPU。

论文原文 NeMo Skill 设计原理
Megatron Core GEMM-EP Overlap 详细设计原理
核心问题：在生产级 MoE 训练中，需要同时考虑 TP、PP、EP、CP 多种并行维度的通信重叠。如何在不影响正确性的前提下最大化 overlap 是一个工程挑战。
设计原理：(1) DeepEP 后端集成：Megatron-Core 通过 apply_flex_dispatcher_backend(cfg.model, moe_flex_dispatcher_backend="deepep") 启用 DeepEP 后端。DeepEP 提供非阻塞的 dispatch/combine，与 GEMM 在不同 CUDA stream 上并行。(2) Delay Wgrad Compute：配置 cfg.comm_overlap.delay_wgrad_compute = True——在 backward 中，延迟 weight gradient 的计算，先完成 all-to-all 通信，然后同时执行 wgrad 计算和通信。这避免了 wgrad 与 all-to-all 争抢 SM 资源。(3) Token Dispatcher 类型：支持 alltoall（标准）和 deepep/hybrid-ep（高性能）两种 dispatcher。alltoall 模式使用 ncclAllToAll，deepep 模式使用 DeepEP 的非阻塞 API。
性能数字：GB200/GB300 上 DeepSeek-V3-685B 达到 1233 TFLOPS/GPU；Qwen3-235B 达到 974 TFLOPS/GPU。
配置约束：需要 expert_model_parallel_size > 1；支持 BF16/FP16；PyTorch >= 2.6.0；shared expert overlap 必须为 False。

Triton Persistent Cache-Aware Grouped GEMM

PyTorch Blog | 2025

核心: 使用 persistent kernel + grouped launch ordering 优化 MoE GroupedGEMM。Device-side TMA descriptor 创建支持动态 expert routing。端到端训练加速 2.62x（DeepSeek-V3-16B on 8xH100），L2 cache hit 率提升 60%。

PyTorch Blog 设计原理
Triton Persistent Grouped GEMM 详细设计原理
核心问题：传统 GroupedGEMM 为每个 expert 启动一个独立的 GEMM kernel，引入大量 kernel launch overhead。同时，由于 MoE 的 token 路由是动态的（运行时才知道每个 expert 分到多少 token），传统的 static-shape kernel 无法高效处理变长输入。
设计原理：(1) Persistent Kernel：设置 grid size = NUM_SMS（H100 上 132 个 SM），每个 SM 运行一个 Triton program。Program 在 loop 中处理多个 tile——for tile_id in tl.range(...)，避免了 wave quantization 和 launch overhead。(2) Grouped Launch Ordering：按 column-major 分组（如 GROUP_SIZE_M=2）安排 tile 处理顺序——C(0,0) → C(1,0) → C(0,1) → C(1,1)。这使相邻 SM 访问的 A 矩阵行在 L2 cache 中保持 hot，L2 hit 率提升 60%。(3) Device-Side TMA Descriptor：对于动态 expert routing，kernel 在 device 端创建 TMA descriptor——每个 SM 在预分配的 workspace 中声明一个 slot，使用 tl.extra.cuda.experimental_device_tensormap_create2d 创建指向特定 expert 权重的 tensor map。这避免了 host-side 的同步开销。(4) 通信重叠：虽然本文主要关注 GEMM kernel 优化，但 persistent kernel 的设计天然支持通信重叠——kernel 在 loop 中处理 tile 时，可以在 tile 间隙插入通信操作（如 all-gather 的下一个 slice）。
性能数字：端到端训练加速 2.62x（DeepSeek-V3-16B on 8xH100）；microbenchmark 加速 1.50x（vs baseline Triton Group GEMM）；L2 cache hit 率提升 60%。
适用场景：(1) FSDP2 训练——FSDP2 中 expert 权重需要 all-gather，persistent kernel 的 cache-aware 设计减少 all-gather 延迟；(2) 动态 routing——运行时才知道 expert 分配的场景；(3) H100/Blackwell——TMA 和 persistent kernel 依赖 Hopper+ 架构特性。

NCCL EP 实现详细设计原理

DeepEP GEMM-EP 融合详细设计原理
核心问题：DeepEP V1 依赖 NVSHMEM 作为底层 RMA 传输层，但 NVSHMEM 的重量级 PGAS 运行时引入显著启动开销（>100ms）和 SM 占用（8-16 SMs）。V2 转向 NCCL Gin 后端实现零 SM 占用。

DeepGEMM Mega MoE 详细设计原理
核心问题：MoE 推理中，每次 forward 需要执行 dispatch → GEMM-1 → SwiGLU → GEMM-2 → combine 五个操作。传统实现每个操作独立 kernel launch，引入 5 次 kernel launch overhead（~25μs）和中间 buffer 的 HBM 读写。
设计原理：(1) Mega MoE 融合：将五个操作融合为单一 kernel。Dispatch 通过 NVLink symmetric memory 执行——GPU 直接从远程 GPU 读取 token（而非 all-to-all 传输），读取完成后立即执行 GEMM-1。SwiGLU 的激活值直接在 register file 中计算（不写回 shared memory）。GEMM-2 的输出直接通过 NVLink 写回源 GPU（combine）。(2) FP8 Tensor Core：使用 FP8 精度（E4M3/E5M2）执行 GEMM，相比 BF16 吞吐提升 2x。SM90 使用 FP32 scaling；SM100 使用 packed UE8M0 scaling。(3) JIT 编译：kernel 根据实际的 M（token 数）、N（expert 维度）、K（hidden 维度）在运行时 JIT 编译为特化的 PTX，避免通用 kernel 的分支和 padding 开销。
性能数字：H800 峰值 1550 TFLOPS（FP8）；Mega MoE 融合减少 ~80% kernel launch overhead。

vLLM Docs 设计原理
vLLM DBO 详细设计原理
核心问题：MoE 推理中，单个 batch 的 all-to-all 通信与 GEMM 无法完全 overlap——因为 GEMM 需要等待所有 token 到达后才能开始。DBO 通过将 batch 切分为两个 micro-batch，实现交错执行。
设计原理：(1) Dual Batch Overlap：将 batch 切分为 micro-batch-0 和 micro-batch-1。执行顺序：dispatch(mb-0) → dispatch(mb-1) || GEMM(mb-0) → combine(mb-0) || GEMM(mb-1) → combine(mb-1)。dispatch(mb-1) 与 GEMM(mb-0) 重叠；combine(mb-0) 与 GEMM(mb-1) 重叠。(2) CUDA Stream 管理：vLLM 使用 3 个 CUDA stream——compute stream（GEMM）、comm stream（all-to-all）、copy stream（H2D/D2D）。三个 stream 并行执行，通过 CUDA event 同步依赖。
适用场景：(1) 大 batch 推理——batch size >= 64 时 DBO 效果最佳；(2) 长序列推理——长序列的 token 多，micro-batch 切分更细，overlap 效果更好；(3) DeepEP/HybridEP 后端——需要非阻塞的 dispatch/combine API。

GEMM-EP Fused 核心技术详解

技术 1: Tile-Level Barrier 同步

核心思想：将输入 tensor 切分为多个 tile（如 128x128），每个 tile 独立管理通信和计算状态。GPU 使用 warp-level barrier 同步——当一个 tile 的通信完成后，立即触发该 tile 的计算，无需等待其他 tile。

代表方案：MegaScale-MoE、Comet。

性能影响：实现 tile 粒度的流水线，通信延迟隐藏率可达 86.5%（Comet）。

// Pseudo-code: Tile-level barrier for (tile_id = 0; tile_id < NUM_TILES; tile_id++) { // 1. 通信此 tile 的数据 nccl_send_recv(tile_buffer[tile_id], ...); // 2. Warp-level barrier - 等待此 tile 通信完成 __syncwarp(); // 3. 立即计算此 tile 的 GEMM gemm_tile(tile_buffer[tile_id], weight[tile_id], ...); // 4. 通信结果（combine） nccl_send_recv(output_buffer[tile_id], ...); }

技术 2: Staged Execution (send_only)

核心思想：将 dispatch 操作拆分为 send 和 receive 两个阶段。send_only=1 时仅发起网络传输（send），GPU 立即执行 GEMM（计算已到达的 local token），然后执行 receive 等待远程 token 到达。

代表方案：NCCL EP、DeepEP、NIXL。

流水线：dispatch(send) → GEMM(local) → dispatch(recv) → GEMM(remote) → combine

// Pseudo-code: Staged execution // Phase 1: 发起网络传输（不等待） ncclEpDispatch(tokens, handle, send_only=1); // Phase 2: GPU 计算 local tokens gemm(local_tokens, expert_weights, ...); // Phase 3: 等待远程 token 到达 ncclEpDispatch(tokens, handle, send_only=0); // Phase 4: GPU 计算 remote tokens gemm(remote_tokens, expert_weights, ...); // Phase 5: Combine 结果 ncclEpCombine(output, handle);

技术 3: Hierarchical Aggregation

核心思想：在 NVLink 域内先做 local aggregation（多个 GPU 的 token 在域内聚合），减少 inter-node 传输量。例如 8 GPU 节点中，先将 8 个 GPU 的 token 通过 NVLink 聚合到 2 个 GPU 上，再通过 IB 发送到其他节点——IB 传输量减少 4x。

代表方案：Hybrid-EP、NCCL EP HT 模式。

性能影响：inter-node 通信量减少 N/2（N 为 NVLink 域内 GPU 数），但引入 NVLink 聚合开销（~2μs）。

技术 4: DSMEM (Distributed Shared Memory)

核心思想：利用 H100 的 NVLink DSMEM 特性——一个 GPU 的 shared memory 可以被另一个 GPU 直接访问。GEMM 的 prologue/epilogue 直接通过 DSMEM 从远程 GPU 加载/存储数据，避免 HBM 的中间写入。

代表方案：ClusterFusion（ClusterReduce/ClusterGather）、SonicMoE（Gather-Fused GEMM）。

性能影响：消除 1 次 HBM 写入 + 1 次 HBM 读取，减少 ~30% HBM 流量。但需要 NVLink 连接（仅适用于节点内）。

技术 5: Warp-Specialized Persistent Kernel

核心思想：使用 persistent kernel 设计——一个长期运行的 CUDA kernel 占据固定 SM，内部使用 warp specialization：load warp 负责从 shared memory 加载 token，network warp 负责 RDMA 传输，store warp 负责写入结果。三种 warp 在不同 tile 上并行执行。

代表方案：Hybrid-EP、NCCL EP HT 模式、Triton Persistent GEMM。

性能影响：消除 kernel launch overhead（~5μs per launch），实现 load/network/store 三级流水线。SM 利用率提升 2-3x。

技术分类分布

趋势展望

趋势 1: 从 Overlap 到 Fusion

2024 年的主流方案是 Overlap（MegaScale-MoE、Comet、Lancet）——通信和计算在不同 stream 上并行。2025-2026 年转向 Fusion（DeepGEMM Mega MoE、SonicMoE、ClusterFusion）——通信和计算融合为单一 kernel，消除 kernel launch 和中间 buffer。Fusion 方案的性能更好（减少 ~80% kernel launch overhead），但实现复杂度更高（需要硬件特性如 TMEM、DSMEM）。

趋势 2: 硬件特性驱动

Blackwell 的 TMEM double-buffering、H100 的 DSMEM、NVLink 5.0 的带宽提升正在成为 GEMM-EP fusion 的关键使能技术。SonicMoE 的 gather-fused GEMM 依赖 NVLink 5.0 的带宽；ClusterFusion 的 ClusterReduce 依赖 DSMEM。未来的 fusion 方案将更加依赖特定硬件特性——跨平台兼容性挑战增大。

趋势 3: NCCL EP 统一生态

NCCL EP 正在成为 MoE 通信的统一 API——它融合 DeepEP（LL）和 Hybrid-EP（HT）两种内核，提供统一的 C/Python API。这结束了此前 DeepEP、 NVSHMEM、NCCL all-to-all 三种方案并存的混乱局面。预计 2026 年下半年 NCCL EP 将成为 MoE 训练/推理的标准后端。

总体观察

GEMM-EP fused 技术正从"通信与计算的粗粒度 overlap"快速演进为"单一 kernel 内的细粒度融合"。三大驱动力：(1) 硬件使能——Blackwell TMEM、H100 DSMEM、NVLink 5.0 提供了融合的物理基础；(2) 算法创新——tile-level barrier、warp-specialized persistent kernel、staged execution 提供了融合的软件基础；(3) 生态统一——NCCL EP 的统一 API 结束了碎片化，降低了采用门槛。预计 2026 年底，GEMM-EP fusion 将成为 MoE 训练/推理的标配技术。
