摘要

本专利披露了一种新型同步系统,用于同步处理器和内存之间的数据交换。该系统引入了 "store + arrive" (STS+Arrive) 组合指令,将数据写入接收缓冲区的同时更新同步屏障,实现"光速"延迟 (约 0.5 往返时间) 的 GPU 间数据同步。通过双屏障架构——同步功能分布在两个屏障上 (一个位于消费者侧,一个位于生产者侧),消除了远程轮询开销。该系统与协作组阵列 (CGA) 模型深度集成,确保并发线程执行的一致性语义,内存围栏"在所有互连路径上复制"以保证数据可见性。

技术背景与问题陈述

传统同步机制的局限性

问题 1: 传统 GPU 同步依赖软件轮询

传统并行处理器 (如早期 NVIDIA Pascal 和 Volta 架构) 依赖有限的硬件屏障或软件轮询机制来实现线程同步。在多 GPU 场景下,远程轮询 (remote polling) 需要跨越 PCIe 或 NVLink 链路反复读取标志位,产生大量不必要的互连流量,显著增加延迟。

问题 2: 数据写入与同步信号分离

传统方案中,生产者需要先执行数据写入 (store),再执行独立的同步信号更新 (flag update)。这两个步骤之间存在竞态窗口,消费者可能在数据尚未完全写入时就被唤醒,导致数据不一致。同时,两步操作增加了总体延迟。

问题 3: 跨 SM/跨处理器同步的扩展性问题

随着 GPU 规模扩大 (如 Hopper H100 的 132 个 SM, Blackwell B200 的更多 chiplet),跨 SM 和跨 GPU 的同步需要可扩展的机制。传统的 bulk-synchronous (整体同步) 模型要求所有线程在 barrier 处等待,造成严重的资源浪费。

问题 4: 内存围栏 (Memory Fence) 的高开销

NCCL 的 Simple 模式通过内存围栏强制执行排序,但围栏操作会暂停整个 SM 的指令发射,直到所有之前的写操作被确认。这在小消息场景下造成显著的延迟尖峰 (latency spikes)。

核心设计理念: 将数据写入和同步信号更新合并为单一原子操作 (STS+Arrive),并通过双屏障架构消除远程轮询。同步硬件直接集成到数据移动引擎 (DMA、TMAU、多播单元),实现"光速"延迟的 GPU 间通信。

创新 1: STS+Arrive 组合指令

单一的 STS+Arrive 指令同时完成数据写入和同步信号更新,消除传统方案中独立的 flag 更新步骤。该指令在硬件层面保证数据写入完成后才更新屏障计数器,避免竞态条件。

创新 2: 双屏障架构

同步功能分布在两个屏障上——一个位于消费者侧,一个位于生产者侧。生产者在本地完成屏障更新,消费者在本地等待屏障清除,完全消除了跨互连连线的远程轮询开销。

创新 3: CGA 模型集成

协作组阵列 (CGA) 模型确保并发线程执行的一致性语义。内存围栏"在所有互连路径上复制",保证数据在 NVLink、PCIe 等所有路径上的可见性,支持 Multi-Instance GPU (MIG) 独立同步。

创新 4: 硬件级同步单元

专用同步单元 (Synchronization Unit) 集成在 GPU 硬件中,包含本地缓存 (barrier cache)、等待缓冲区 (try-wait buffer) 和数据路径电路。屏障数据结构存储在内存中,但通过硬件缓存加速访问,减少对主存的流量。

STS+Arrive 指令详解

指令语义与执行流程

STS+Arrive 指令 是该专利的核心创新。它将传统的两个独立操作——数据写入 (store) 和屏障到达 (barrier arrive)——合并为单一原子指令。
1

数据写入阶段

生产者线程执行 STS+Arrive 指令,将数据写入目标接收缓冲区。数据写入通过 GPU 的 store 指令路径执行,确保数据到达目标内存位置。

STS+Arrive [dest_addr], data, barrier_addr
// 原子操作: 写入数据 + 更新屏障
2

硬件级排序保证

硬件保证数据写入在屏障计数器更新之前完成。这通过内部的 store buffer 和 barrier update queue 实现,确保消费者在屏障清除时能看到完整的数据。

3

屏障计数器更新

数据写入完成后,硬件自动递增屏障的 arrival counter。当 arrival counter 达到 expected threshold 时,屏障清除,等待的消费者线程被唤醒。

4

消费者本地等待

消费者线程在本地执行 barrier wait,等待屏障清除。由于屏障状态在消费者侧本地缓存,消费者无需远程轮询生产者,大幅降低延迟。

为什么 STS+Arrive 能降低延迟?

传统方案需要两次独立的互连事务:一次数据写入,一次标志更新。STS+Arrive 将这两次事务合并为一次,减少了约 50% 的互连流量。更重要的是,它消除了数据写入和标志更新之间的竞态窗口——硬件保证数据先写入、后更新屏障,消费者无需额外的验证步骤。

来源: US20230315655A1, CUDA C++ Programming Guide

双屏障架构深度解析

屏障数据结构

屏障对象存储在内存中,包含以下关键字段:

字段 类型 作用 更新时机
phase_indicator 1 bit 相位标记,每次屏障清除时翻转 屏障清除时翻转
arrival_counter N bits 已到达线程计数器 每次 arrive() 调用时递减
transaction_counter M bits 已完成事务计数器 每次 DMA/TMAU 事务完成时递增
expected_arrivals N bits 期望到达的线程数 屏障初始化时设定
lock_flag 1 bit 并发访问锁 读写屏障时加锁
来源: US20230315655A1, US20230289242A1 (Transaction Barrier)

生产者侧屏障

生产者执行 STS+Arrive 指令后,本地屏障的 arrival counter 递减。当所有预期生产者都到达时,生产者侧屏障清除,触发数据发送。

// 生产者代码
STS+Arrive [recv_buf], data, prod_barrier
barrier.arrive(prod_barrier)
// 数据已写入,同步信号已更新

消费者侧屏障

消费者在本地执行 barrier wait,等待屏障清除。当生产者侧的数据到达且所有同步条件满足时,消费者侧屏障清除,线程被唤醒。

// 消费者代码
barrier.wait(cons_barrier)
// 屏障清除,数据已就绪
consume(recv_buf)

双屏障 vs 单屏障的性能优势

在单屏障方案中,消费者需要远程轮询生产者的屏障状态,每次轮询都产生一次跨互连连线的读事务。在 NVLink 上,这可能增加数百纳秒的延迟。双屏障方案中,消费者只需等待本地屏障,生产者通过硬件机制异步更新消费者侧屏障状态。这消除了轮询开销,使同步延迟降低到约 0.5 往返时间 (RTT)。

协作组阵列 (CGA) 模型

CGA 模型概述

协作组阵列 (Cooperative Group Array, CGA) 是 NVIDIA 在 Volta 架构引入的编程模型扩展,允许线程组在多个 GPU 或 GPU 实例之间协调执行。在本专利中,CGA 与 STS+Arrive 指令和双屏障架构深度集成,提供跨设备的同步语义。

关键语义保证: CGA 中的所有线程对 STS+Arrive 指令的执行是一致的。当一个线程执行了 STS+Arrive,所有其他 CGA 线程都能看到屏障状态的更新。内存围栏"在所有互连路径上复制",确保数据在 NVLink、PCIe、InfiniBand 等所有路径上的可见性。

单 GPU CGA

单个 GPU 内的多个 SM 组成 CGA,通过分布式共享内存 (DSMEM) 和片上屏障硬件实现同步。适用于 MIG 场景,多个 GPU 实例独立执行。

多 GPU CGA

多个 GPU 通过 NVLink 互连组成 CGA,通过 NVLink P2P 直接内存访问和硬件屏障复制实现同步。支持 GB200 NVL72 等 72-GPU 系统。

跨节点 CGA

跨多节点的 GPU 通过 InfiniBand/RDMA 组成 CGA,通过自同步远程内存操作 (US12105960B2) 实现无锁 RDMA 通信和同步。

内存围栏复制机制

专利中的关键创新是内存围栏在所有互连路径上的复制。当 CGA 线程执行内存围栏时,硬件确保:

  • NVLink 路径: 所有通过 NVLink 的写操作被刷新,确保远程 GPU 能看到数据
  • PCIe 路径: 所有通过 PCIe 的写操作被刷新,确保 CPU 和其他设备能看到数据
  • InfiniBand 路径: 所有通过 RDMA 的写操作被刷新,确保远程节点能看到数据
  • 一致性保证: 围栏操作在所有路径上完成后,屏障状态才更新,确保消费者看到的屏障状态与数据可见性一致
来源: US20230315655A1, CUDA Cooperative Groups

Barrier Cache (屏障缓存)

屏障数据结构存储在内存中,但同步单元维护一个本地缓存。当线程执行 arrive() 或 wait() 时,首先访问本地缓存,减少对主存的访问流量。缓存通过写回 (write-back) 策略与主存保持一致。

Try-Wait Buffer (等待缓冲区)

当线程执行 barrier wait 但屏障尚未清除时,线程 ID 被存储到 try-wait buffer 中,而不是执行忙轮询 (busy polling)。当硬件检测到屏障清除时,直接向 try-wait buffer 中的线程发送唤醒信号。这显著降低了互连流量和功耗。

Update Coalescer (更新合并器)

当多个远程 SM 或 GPU 向同一个屏障发送更新时,更新合并器将这些更新合并为单个内存写事务。例如,如果 8 个远程 SM 都向同一个屏障发送 arrive 更新,合并器将它们合并为一次 barrier counter 更新,大幅减少内存带宽消耗。

与异步数据移动引擎的集成

同步单元与 GPU 中的异步数据移动引擎深度集成,包括:

DMA 引擎

直接内存访问 (DMA) 引擎完成数据传输后,自动向同步单元发送更新包,包含屏障地址和数据量指标。同步单元据此更新 transaction counter。

TMAU (Tensor Memory Accelerator)

Hopper H100 引入的 TMAU 负责张量数据的异步加载。TMAU 完成加载后,通过硬件信号更新关联的屏障,允许计算线程在数据就绪后立即开始。

多播硬件

程序化多播硬件 (US12020035B2) 完成数据分发后,向所有目标计算引擎发送屏障更新,确保所有接收者在数据到达后能同步开始计算。

来源: US20230289242A1, NVIDIA Hopper Architecture In-Depth

Object Fence 语义

专利引入了 "object fence" 语义,确保与屏障关联的所有内存操作在屏障状态转换之前变得可见。这比传统的内存围栏更精确——它只等待与特定屏障相关的操作完成,而不是所有未完成的写操作。这减少了不必要的等待,进一步降低延迟。

与 NCCL 的关联分析

NCCL 传输模式与该专利的关系

根据 arXiv:2507.04786v3 对 NCCL 的深度分析,NCCL 使用三种传输模式,每种模式都与该专利的同步机制相关:

NCCL 模式 同步机制 与该专利的关联
Simple 内存围栏 (高开销) 专利的 STS+Arrive 指令可以替代 Simple 模式中的内存围栏,减少 ordering 开销。双屏障架构消除远程轮询。
LL (Low Latency) 基于标志的同步 (4B 数据 + 4B 标志) STS+Arrive 指令本质上是 LL 模式的硬件级优化——将数据和标志的两次写入合并为一次原子操作。本地屏障缓存消除了对 host-memory buffer 的依赖。
LL128 128B 原子写入 (120B 数据 + 8B 标志) CGA 模型与 LL128 的 128B 粒度对齐,确保跨 GPU 的同步一致性。内存围栏复制保证所有路径上的数据可见性。

NCCL 2.28 中的直接体现

在 NCCL 2.28 中,该专利的技术有了直接体现:

GPU-initiated Networking

GPU 可直接发起网络通信,无需 CPU 介入。这正是 STS+Arrive 指令的扩展——GPU 在写入数据后自动触发网络传输和同步更新。

Copy Engine Collectives

利用 GPU 的 copy engine 执行集合通信,与计算核心并行执行。Copy engine 与同步单元的集成 (DMA/TMAU 集成) 使这成为可能。

新设备 API

提供设备端的通信原语,允许 kernel 内部直接调用集合通信。这些原语底层使用 CGA 模型和 STS+Arrive 语义。

通信-计算融合

允许在单个 kernel 内完成通信和计算,减少 kernel 启动开销。双屏障架构确保通信和计算的正确排序。

NCCL Proxy 机制与专利的关系

NCCL 使用 CPU 端的 proxy thread 管理网络操作和 RDMA 传输。该专利的硬件级同步机制为未来 NCCL 版本的"无 proxy"模式提供了基础——当 GPU 可以直接处理同步时,CPU proxy 的参与可以最小化,进一步降低延迟。

核心权利要求解读

根据专利公开文本,核心权利要求涵盖以下技术方案:

权利要求 1: STS+Arrive 组合指令

一种用于处理器间数据同步的方法,包括:在执行单元执行组合指令,该指令同时 (a) 将数据写入接收缓冲区和 (b) 更新与接收缓冲区关联的同步屏障。该组合指令在硬件层面保证数据写入先于屏障更新完成。

权利要求 2: 双屏障架构

一种同步系统,包括:生产者侧屏障对象和消费者侧屏障对象。生产者侧屏障在数据写入完成后更新,消费者侧屏障在生产者侧屏障更新后清除。消费者在本地等待消费者侧屏障,无需远程轮询生产者状态。

权利要求 3: CGA 内存围栏复制

一种内存围栏方法,包括:在协作组阵列 (CGA) 中,将内存围栏操作复制到所有互连路径 (包括 NVLink、PCIe、InfiniBand),确保数据在所有路径上的可见性在屏障状态更新之前完成。

权利要求 4: MIG 支持

一种多实例 GPU 同步方法,包括:允许多个 GPU 实例独立执行同步操作,每个实例维护独立的屏障状态,通过硬件级隔离确保实例间的同步不互相干扰。

权利要求 5: 同步单元硬件

一种硬件同步单元,包括:屏障缓存 (barrier cache)、等待缓冲区 (try-wait buffer)、数据路径电路 (datapath circuitry) 和更新合并器 (update coalescer)。该单元支持 arrive() 和 wait() 分离原语,以及异步数据移动引擎的自动屏障更新。

影响分析与未来展望

对 NCCL 集合通信的影响

该专利是 NCCL 底层同步原语的硬件级优化,对 AllReduce、AllGather、Reduce-Scatter 等集合通信操作有直接影响。STS+Arrive 指令和双屏障架构可降低集体通信的 tail latency 约 20-30%,对万亿参数模型的多节点训练至关重要。

对 GB200 NVL72 的影响

在 72-GPU 的 GB200 NVL72 系统中,CGA 模型和内存围栏复制确保跨 NVLink mesh 的同步一致性。双屏障架构消除了跨 die 的远程轮询,对减少 NVLink 互连的流量开销有重要意义。

对 LLM 训练的影响

在万亿参数模型的分布式训练中,梯度同步是主要的通信瓶颈。STS+Arrive 指令使梯度 AllReduce 的同步延迟降低到约 0.5 RTT,显著提升训练吞吐量。CGA 模型确保跨节点同步的一致性。

对 LLM 推理的影响

在多 GPU 推理场景中,KV cache 的跨 GPU 同步是关键延迟来源。双屏障架构使消费者 GPU 在本地等待屏障清除,无需远程轮询,降低推理服务的 tail latency。

对 NCCL 2.28+ 的影响

NCCL 2.28 的 Copy Engine Collectives 和新设备 API 直接利用了该专利的同步机制。未来 NCCL 版本可能进一步减少 CPU proxy 的参与,实现完全由 GPU 硬件管理的集合通信。

专利护城河

该专利由 NVIDIA GPU 架构核心人物 Olivier Giroux 和 Jack Choquette 发明,保护了 NVIDIA 在硬件级同步原语的核心技术。AMD RCCL 和华为 HCCL 需要绕过该专利的权利要求范围,或开发替代的同步方案。

总结

US20230315655A1 是 NVIDIA 在 GPU 间快速数据同步领域的核心专利,通过 STS+Arrive 组合指令和双屏障架构,将 GPU 间同步延迟降低到约 0.5 RTT,同时消除了远程轮询开销。该专利与 CGA 模型深度集成,确保跨 NVLink、PCIe、InfiniBand 等所有互连路径的同步一致性。

在 NCCL 2.28 中,该专利的技术已经有了直接体现 (GPU-initiated networking、Copy Engine Collectives、新设备 API)。随着 NVIDIA GPU 向更多 chiplet 和更大规模系统发展,该专利的同步机制将变得越来越重要。

对于竞争对手 (AMD、华为、Intel),该专利构成了硬件级同步原语的专利壁垒,需要开发替代方案或获得授权。对于学术界和开源社区,该专利揭示了 NVIDIA GPU 同步机制的底层原理,有助于优化分布式训练和推理框架。
