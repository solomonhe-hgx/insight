NVLink Fusion 生态扩展: Marvell / Qualcomm / Fujitsu 加入

2026

核心内容: Marvell 正式加入 NVLink Fusion 生态;Qualcomm 和 Fujitsu 作为首批合作伙伴,支持第三方 ASIC 接入 NVLink 内存端口。

技术方案

NVLink Fusion 将 NVLink 内存端口(而非完整协议栈)授权给合作伙伴,使第三方 ASIC 能够通过 NVLink 直接访问 NVIDIA GPU 内存。这是一种"半开放"策略:保持 NVLink 核心协议控制,同时扩展生态兼容性。Marvell 的加入意味着定制芯片设计可以原生接入 NVLink fabric。

#1254 — 内核内存无限制增长导致 OOM (610.43.03)

严重 | 2026-07-21

问题描述: 在 v610.43.03 驱动上,二级页表 (secondary page tables) 消耗数十 GB 内核内存,最终导致 OOM Kill。

详细分析

该 Issue 报告在 v610.43.03 驱动上,UVM 驱动的二级页表内存消耗无上限增长,从正常水平飙升至数十 GB,最终触发 Linux OOM Killer。这是一个回归性 Bug,可能与 r580 分支引入的页表分配变更有关。影响所有使用 Managed Memory (UVM) 的工作负载。

#1244 — folio_mark_dirty 内核 Oops (UVM/HMM 路径)

高 | 2026-07-16

问题描述: UVM/HMM 内存迁移路径中触发 folio_mark_dirty 内核 Oops,陈旧 PTE 导致 Linux 7.1 内核崩溃。

详细分析

在 UVM 通过 HMM (Heterogeneous Memory Management) 进行页面迁移时,出现陈旧 PTE (Page Table Entry) 导致的 folio_mark_dirty 内核 Oops。该问题在 Linux 7.1 内核上复现,涉及 UVM 页面故障处理与内核内存管理子系统的交互。可能与 GPU 侧页表失效后 CPU 侧仍持有旧 PTE 引用有关。

#1245 — nv_mem_pool list_del 损坏 (RTX 5090 Mobile)

中 | 2026-07-16

问题描述: RTX 5090 Mobile (Blackwell) 上 nv_mem_pool 内存池出现 list_del 链表损坏。

详细分析

在 Blackwell 架构的 RTX 5090 Mobile 上,nv_mem_pool 内存池管理出现 list_del corruption。这通常意味着并发访问内存池时存在竞态条件,或者已释放的内存条目被重复操作。该问题在 Blackwell 移动平台上首次报告,可能与新的内存管理硬件特性有关。

#1253 — GPUDirect RDMA BAR1 读取失败 (Intel Arrow Lake)

中 | 2026-07-21

问题描述: Intel Arrow Lake 平台上 GPUDirect RDMA 的 BAR1 读取失败,NVIDIA 确认为硬件限制。

详细分析

该 Issue 报告在 Intel Arrow Lake 平台上,GPUDirect RDMA 通过 BAR1 (Base Address Register 1) 进行内存读取时失败。NVIDIA 已确认这是 Intel Arrow Lake 的硬件限制,非驱动 Bug。这影响在 Arrow Lake 平台上使用 GPUDirect RDMA 的用户,需要等待 Intel 侧硬件修复或使用替代方案。

PR #1234 — vzalloc 修复超大页表分配 (r580 回归)

PR | 2026-07-08

修复内容: 使用 vzalloc 替代 vmalloc 处理超大页表分配,修复 r580 分支引入的回归问题,已在 CXL 512 GiB 环境验证。

技术方案

当 GPU 页表规模超过 vmalloc 的连续内存限制时,分配失败导致驱动崩溃。该 PR 将分配方式从 vmalloc 改为 vzalloc (允许非连续物理内存),解决超大内存配置(如 CXL 512 GiB)下的页表分配问题。此回归始于 r580 驱动分支。

#1208 — Xid 31 MMU FAULT_PDE (RTX 5090 CUDA JIT)

2026-06-17

问题描述: RTX 5090 上 CUDA JIT 编译后触发 Xid 31 (GPU MMU Error: FAULT_PDE),页表目录项缺失。

详细分析

CUDA JIT 编译产生大量内存分配请求,导致 GPU MMU 在查找页表时出现 PDE (Page Directory Entry) 缺失,触发 Xid 31 错误。该问题在 Blackwell 消费级 GPU (RTX 5090) 上复现,可能与 Blackwell 新的 GMMU (GPU MMU) 硬件版本对大内存工作负载的处理有关。

#1223 — nvidia_uvm HMM fault-service 驱逐死锁 (Blackwell)

2026-07-01

问题描述: Blackwell 架构上 nvidia_uvm 的 HMM fault-service 路径出现驱逐死锁。

详细分析

在 Blackwell GPU 上使用 HMM (Heterogeneous Memory Management) 时,fault-service 线程在处理内存页面故障时进入死锁状态。驱逐 (eviction) 路径与故障服务路径之间存在锁顺序问题。该问题在 Blackwell 架构上特有,可能与新的内存管理硬件特性交互有关。

Blackwell MMU/UVM 稳定性挑战

本周 7 个新 Issue 中 4 个涉及 Blackwell 架构 (#1245, #1223, #1208, #1254),表明新一代硬件在 MMU/UVM 层面仍面临稳定性挑战,需要驱动侧持续修复。

NVLink-C2C 产品页已明确支持 CXL 协议,结合 GTC 2026 发布的 CMX 内存存储平台,CXL 正在成为 GPU 内存扩展的重要补充路径。

  • Blackwell MMU/UVM 问题 (#1223, #1245) 需要架构特定的修复

  • HMM/UVM 在 Linux 内核中的持续演进将影响驱动设计

核心洞察

1. MMU/UVM 是当前开源驱动最活跃的痛点

本周 7 个新 Issue 全部集中在 MMU/UVM 模块,涵盖页表内存泄漏、HMM 死锁、链表损坏、MMU Fault 等多种问题类型。这表明随着 Blackwell 硬件普及和 Linux 内核快速迭代,内存管理子系统的稳定性是开源驱动面临的最大挑战。

2. NVLink 6 生态已全面铺开

NVIDIA 官方博客详细披露 NVLink 6 技术细节,产品页完成 6 代规格对比更新,NVLink Fusion 生态扩展至 Marvell/Qualcomm/Fujitsu。NVLink 从封闭走向"半开放",是 NVIDIA 互连策略的重要转变。

3. Blackwell 驱动成熟度仍需提升

本周报告的 Issue 中,#1245 (RTX 5090 Mobile)、#1223 (Blackwell HMM 死锁)、#1208 (RTX 5090 MMU FAULT) 均指向 Blackwell 架构,表明新一代硬件在开源驱动中的支持仍在快速迭代中。用户在生产环境部署 Blackwell 时应关注这些已知问题。

4. 仓库活跃度与社区参与度存在落差

尽管 GitHub 上有 17.2k Stars 和活跃的 Issue 报告,但本周无新 Commits 合入 main 分支,也无新 Release。社区论坛讨论主要集中在消费级 Bug (Xid 109、Wayland HDR),而非互连或内存管理等核心技术话题。开源驱动的代码贡献和审查流程可能需要更多社区参与。
