第六代 NVLink — Rubin 平台 3.6 TB/s 带宽

2026-04 官方

创新点: NVIDIA 官网更新 Rubin 平台 NVLink 第六代规范，每 GPU 带宽翻倍至 3.6 TB/s，72 GPU 机架级聚合带宽达 260 TB/s。引入控制面弹性 (control plane resilience) 和 Switch Tray 热插拔功能。

来源: NVIDIA NVLink & NVSwitch 官方

详细技术方案

第六代 NVLink 专为 Rubin 平台优化，相比 GB200/GB300 的第五代 NVLink 带宽翻倍 (1.8 TB/s → 3.6 TB/s)。新增两大可靠性功能：(1) 控制面弹性 (control plane resilience) — NVLink 控制面通信的故障恢复机制，确保在链路降级时保持基本互连功能；(2) Switch Tray 热插拔 — 允许在机架运行期间更换 NVLink Switch 托盘，大幅提升数据中心运维灵活性。NVLink Fusion 继续扩展至半定制处理器和第三方 ASIC 集成。

2026-04-28

更新: open-gpu-kernel-modules 发布 595.71.05，本周唯一代码提交。对应 GeForce Game Ready Driver 595.71，含 KMS 默认启用、GPU 重置功能等。

版本追踪

DRM/KMS Preview
来源: NVIDIA Developer Forums | Phoronix

DRM 色彩管线 API

该预览驱动实现了 "DRM Per-Plane Color Pipeline API"，使桌面环境能够 "加速色彩处理（尤其是 HDR）"，这些处理通常在软件中完成。作为一个 "处方式 API"，它将硬件渲染块直接暴露给 Wayland 合成器，允许在硬件和软件处理路径之间无缝切换。基于 open-gpu-kernel-modules 595.58.03 回port。规范已合入 Linux v6.19 上游，此 NVIDIA 预览版本专为 DRM/KMS 测试设计， intentionally breaks compatibility with accompanying X driver。正式集成等待未来稳定版。

第六代 NVLink Switch — Switch Tray 热插拔

2026-04 官方

创新点: NVIDIA 官网更新第六代 NVLink Switch 规范，新增 Switch Tray 热插拔功能和控制面弹性机制，允许在机架运行期间更换交换机托盘，是数据中心运维的重大改进。

来源: NVIDIA NVLink & NVSwitch 官方

架构详情

第六代 NVLink Switch 面向 Rubin 平台，在保持 72 GPU 全互联架构和 260 TB/s 聚合带宽的同时，新增两大关键可靠性功能。Switch Tray 热插拔是数据中心运维的重大突破 — 此前更换 NVLink Switch 需要停机。控制面弹性确保在 NVLink 链路降级或部分故障时，控制面通信仍能维持基本互连功能，减少级联故障风险。这些功能与 Tesla 580.159.03 中的 NVSwitch 虚拟化支持（"full visibility of GPUs and NVSwitches to guest VMs"）形成互补。

NVLink-C2C 向第三方定制硅开放

2026-04 官方

创新点: NVIDIA 官网更新确认 NVLink-C2C 支持第三方定制硅集成，通过 chiplet 设计实现 "高带宽、相干连接"。支持 Arm AMBA CHI 或 CXL 协议，为跨供应商互操作性奠定基础。

来源: NVIDIA NVLink-C2C 官方

C2C 架构详情

NVLink-C2C 从 Grace-Blackwell 的 CPU-GPU 互连扩展为通用 chiplet 互连框架。关键特性：(1) 支持 Arm AMBA CHI 或 CXL 协议，确保跨供应商互操作性；(2) 高级 SerDes 和封装技术，支持标准 PCB、MCM 和晶圆级连接；(3) 处理器间原子操作用于快速共享数据同步；(4) 比 PCIe Gen 6 PHY 高 6 倍能效和 3.5 倍面积效率。集成产品包括 Vera Rubin NVL72、GB300 NVL72、GB200 NVL72、Grace Superchip (144 核) 和 DGX Spark (GB10)。与 NVLink Fusion 计划协同，为超大规模 AI 部署提供半定制系统设计能力。

Tesla 580.159 引入 CDMM — Coherent Driver-Based Memory Management

2026-04-28

新功能: R580 驱动为 GB200 平台引入 CDMM，将内存管理从操作系统转移到驱动层。防止容器编排工具中的容量报告不准确，但禁用直接存储访问并需要更新版 GDRCopy。

New Feature Tesla 580.159
来源: Tesla 580.159.03 Release Notes

技术详情

CDMM (Coherent Driver-Based Memory Management) 是 R580 驱动为 GB200 平台引入的全新内存管理机制。核心变化：内存控制从操作系统转移到驱动层。优势：防止容器编排工具 (如 Kubernetes) 中 GPU 内存容量报告不准确的问题。限制：启用 CDMM 后禁用直接存储访问 (direct storage access)，需要更新版 GDRCopy 软件。这一架构变更对数据中心 GPU 内存管理有深远影响，代表了 NVIDIA 对 GPU 内存管理统一化的持续推进。

Arch Linux 主 NVIDIA 驱动包默认使用开源内核模块

2026-04 重大新闻

里程碑: Arch Linux 将其主 NVIDIA 驱动包 (nvidia, nvidia-lts, nvidia-dkms) 切换为默认使用开源内核模块。这是主要 Linux 发行版首次默认采用 NVIDIA 开源 GPU 内核模块。对于 Blackwell (GBXXX) 及更新硬件，nvidia-open 系列包被上游推荐。

Milestone Distribution
来源: Phoronix | ArchWiki NVIDIA

发行版变更详情

Arch Linux 将其 NVIDIA 驱动包从闭源内核模块切换为开源版本，这是 Linux 发行版 NVIDIA 支持的历史性转折点。nvidia-open, nvidia-open-lts, nvidia-open-dkms 成为主包。需要注意的是，开源内核模块不能启用 D3 电源管理（节能状态），这是当前的已知限制。对于 Turing GPU，GSP 固件导致 "次优电源管理"；某些含 Ampere GPU 的笔记本电脑出现 "完全失败"，受影响用户应切换回闭源替代方案并应用 NVreg_EnableGpuFirmware=0 参数。

技术趋势分析

Arch Linux 默认采用开源模块 — 历史性转折

本周最大新闻是 Arch Linux 将其主 NVIDIA 驱动包切换为默认使用开源内核模块。这是主要 Linux 发行版的首次，标志着 NVIDIA 开源 GPU 驱动从实验性到生产就绪的重大转折。然而，电源管理 (D3 节能) 和 Turing/Ampere 笔记本兼容性问题仍需解决。

重大进展

DRM/KMS 生产就绪

595.71 使 modeset=1 默认启用，加上 DRM 色彩管线预览驱动，表明 NVIDIA 正在将开源内核模块的 DRM/KMS 支持推向生产就绪。这对于 Wayland 桌面环境和 HDR 支持至关重要。内核模式设置默认启用也允许通过 nvidia-smi 重置空闲 GPU。

积极进展

CDMM — 数据中心内存管理架构变更

Tesla 580.159 引入的 CDMM (Coherent Driver-Based Memory Management) 将 GPU 内存管理从操作系统转移到驱动层。这是 GB200 平台的关键架构变更，解决了容器编排中的容量报告问题，但牺牲了直接存储访问。这一趋势反映了 NVIDIA 对数据中心 GPU 内存管理的统一化持续推进。

架构变革

本周核心洞察

1. Arch Linux 默认采用开源模块是历史性转折点

主要 Linux 发行版首次将 NVIDIA 开源 GPU 内核模块作为默认驱动。这一变更的影响将深远：更多发行版预计跟进，用户基数扩大将加速 bug 发现和修复。但需注意当前的已知限制：无 D3 电源管理，Turing/Ampere 笔记本兼容性问题。

2. DRM/KMS 生产就绪 — NVIDIA 图形栈的范式转变

modeset=1 默认启用和 DRM 色彩管线预览驱动标志着 NVIDIA 开源驱动在 Linux 图形栈中的定位根本转变。从 "需要闭源 X 驱动" 到 "完全基于 DRM/KMS"，这一转变将使 NVIDIA GPU 在 Wayland 桌面环境中获得一等公民地位。

3. CDMM — 数据中心 GPU 内存管理的架构统一

Tesla 580.159 引入的 CDMM 将 GPU 内存管理从操作系统转移到驱动层。虽然解决了容器编排中的容量报告问题，但牺牲了直接存储访问。这一权衡反映了 NVIDIA 在数据中心场景下的优先级选择：一致性和可管理性优于灵活性。

4. UVM 稳定性与 Linux 7.0 兼容性仍是关键挑战

Issue #1108 (UVM 内核恐慌) 自 4 月中旬以来仍开放，#1117 (Linux 7.0 s2idle) 和 #1125 (GSP 心跳) 是本周新增问题。在 Arch Linux 默认采用的背景下，这些稳定性问题的影响面将显著扩大，NVIDIA 需要加速修复节奏。

5. 第六代 NVLink 为 Rubin 平台设定新的互连标准

3.6 TB/s 每 GPU 带宽和 Switch Tray 热插拔功能代表机架级互连的下一代标准。虽然开源内核模块目前对这些高级互连功能的支持有限（主要在固件和闭源驱动层面处理），但随着 Rubin 平台的推进，开源模块将需要逐步增加对这些功能的支持。
