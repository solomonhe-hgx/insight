摘要

Mixture-of-Experts (MoE) 模型已成为构建大规模语言模型的主流范式,但其 expert 并行化引入了大量动态且不规则的 GPU 间通信。现有的交换机方案 (如 NVSwitch) 和通信协议 (如 NCCL) 针对静态通信模式设计,无法有效处理 MoE 中 dispatch/combine 阶段的变化目标 (varying targets) 和非对称寻址 (asymmetric addressing),导致冗余数据传输和严重的带宽浪费。本文提出 DySHARP (Dynamic Switch-based Hardware Accelerated Routing and Processing)——一个动态交换机内计算框架,通过扩展静态交换机能力以支持不规则工作负载,将通信原语与通信感知调度深度融合。DySHARP 设计了动态多内存寻址 (NVLS 扩展)、交换机多播/归约指令、以及以 token 为中心的内核融合管线,在 32-GPU GH200 系统仿真中实现了近半流量削减和最高 4.25x 的 MoE 层加速,硬件开销仅占 NVSwitch 芯片面积的 0.1% 以下。

研究动机

MoE Expert 并行的 GPU 间通信瓶颈

MoE 模型通过稀疏激活 (每个 token 仅路由到少数 experts) 实现了计算效率的大幅提升。然而,这种稀疏性也带来了独特的通信挑战:

动机 1: MoE Expert 并行化的跨 GPU 通信瓶颈

当 MoE 模型的 experts 分布在多个 GPU 上时,每个 token 需要通过 all-to-all 通信被 dispatch 到正确的 expert GPU,计算后再通过 all-to-all 通信将结果 combine 回原始 GPU。这种跨 GPU 的 dispatch/combine 通信量极大,成为 MoE 训练和推理的主要瓶颈。

动机 2: 动态通信与不规则模式

MoE 的 routing 是动态的——每个 token 根据 gating 网络的结果被分配到不同的 experts。这导致每个 token 的目标 GPU 集合不同 (varying targets),通信模式在不同 mini-batch 甚至不同 step 之间剧烈变化。传统的静态通信协议无法有效处理这种不规则性。

动机 3: 静态协议的局限性

现有的交换机方案 (如 NVSwitch) 和通信库 (如 NCCL) 针对静态的、可预测的通信模式进行了优化。它们使用预定义的广播域、固定的路由表和静态的内存映射。面对 MoE 的动态 routing,这些方案需要退化为多次独立传输,导致严重的冗余数据传输和带宽浪费。

来源: arXiv:2605.05607 Introduction

为什么这篇论文值得关注?

MoE 已经成为万亿参数 LLM (如 DeepSeek-V3、Mixtral、Qwen-MoE) 的核心架构。随着模型规模持续扩大,MoE 的通信开销占比不断上升——在大规模训练中,通信时间可能占总时间的 40% 以上。DySHARP 是首个从硬件-架构-运行时协同设计角度系统性解决 MoE 动态通信问题的工作。它不是简单地优化软件层面的通信库,而是扩展了交换机硬件本身的能力,使其能够原生支持动态、不规则的通信模式。这种"让交换机变智能"的思路代表了未来 GPU 互连架构的重要方向。

Dispatch 阶段: 变化目标 (Varying Targets)

每个 token 根据 gating 网络的 softmax 输出,被路由到 top-k 个 experts (k 通常为 1-8)。由于 experts 分布在不同的 GPU 上,同一个 GPU 上的不同 token 可能需要发送到完全不同的目标 GPU 集合。这种varying targets 特性意味着每个 token 的通信目标都是动态变化的,无法用静态路由表预定义。

非对称寻址 (Asymmetric Addressing)

在 dispatch 阶段,GPU A 需要将 token 发送到 GPU B 的特定 expert 位置,但在 combine 阶段,GPU B 需要将计算结果返回到 GPU A 的不同位置 (对应原始 token 的存储位置)。两个阶段的内存地址映射完全不对称——dispatch 的目标地址在 expert GPU 上,combine 的目标地址在源 GPU 上。这种asymmetric addressing 使得静态的内存映射方案完全失效。

冗余数据传输

现有方案 (如 NCCL all-to-all) 无法识别哪些 token 去往同一个目标 GPU,因此需要为每个 token 独立传输完整的地址信息和元数据。在极端情况下,多个 token 去往同一 GPU 时,仍然需要分别传输,造成严重的冗余数据传输。

来源: arXiv:2605.05607 Section 2: MoE Communication Challenges

为什么现有交换机方案失败?

核心问题: 现有的交换机解决方案 (如 NVSwitch) 失败的根本原因是 "varying targets and asymmetric addressing"——即每个 token 的目标 GPU 集合动态变化,且 dispatch/combine 两阶段的地址映射完全不对称。静态交换机无法适应这种动态性和非对称性。
现有方案 处理 MoE 动态通信的方式 根本缺陷
NCCL All-to-All 每个 GPU 向所有其他 GPU 发送数据块 无视 varying targets,传输全量数据,大量冗余
NVSwitch 静态组播 预定义多播组,固定目标集合 无法适应动态变化的目标 GPU 集合
软件层 Token 打包 在 GPU 内核中合并去往同一 GPU 的 token 增加内核开销,无法解决 asymmetric addressing

DySHARP 架构

DySHARP = Dynamic Switch-based Hardware Accelerated Routing and Processing

DySHARP 是一个动态交换机内计算框架,通过扩展静态交换机 (如 NVSwitch) 的能力,使其原生支持 MoE 等不规则工作负载。它将通信原语与通信感知调度深度融合,在交换机层面实现动态路由、多播、归约和内存寻址。

DySHARP 四层架构设计
第 4 层: 通信感知调度 (Communication-Aware Scheduling)
并发执行管线,平衡双向网络流量,动态处理不规则通信模式
第 3 层: Token-Centric 内核融合
深度融合 dispatch-computation-combine 管线,降低数据移动开销
第 2 层: 交换机多播与归约
dispatch 阶段多播分发,combine 阶段部分结果求和归约
第 1 层: 动态多内存寻址 (Dynamic Multimem Addressing)
NVLS 动态扩展,代数索引到布局索引的翻译,片上表和缓存
核心设计理念: 扩展静态交换机能力

DySHARP 不是构建全新的交换机,而是在现有静态交换机 (如 NVSwitch) 的基础上,添加对不规则工作负载的支持。这包括: (1) 动态内存寻址能力,支持运行时变化的目标地址映射; (2) 交换机内计算能力,支持多播和归约操作; (3) 紧凑的分组格式,减少元数据开销。这种渐进式设计确保了与现有硬件的兼容性,同时将硬件开销控制在极低水平 (NVSwitch die area 的 0.1% 以下)。

通信原语与通信感知调度的融合

DySHARP 的另一个关键设计是将底层通信原语 (多播、归约、寻址) 与上层的通信感知调度协同设计。调度器不仅决定何时发送数据,还了解底层硬件的能力,从而做出更优的调度决策。例如,调度器可以并发执行 dispatch 和 combine 管线,利用双向网络的对称性平衡流量。

来源: arXiv:2605.05607 Section 3: DySHARP Architecture Overview

动态多内存寻址

NVLS 的动态扩展

DySHARP 对 NVIDIA 的 NVLS (NVLink Switch) 进行了动态扩展,使其能够支持 MoE 所需的运行时变化的地址映射。这一扩展涉及 ISA (指令集架构)、微架构和运行时三个层面的协同设计。

ISA / 微架构 / 运行时协同设计

DySHARP 在三个层面进行了协同设计:

  • ISA 层: 新增指令支持动态地址翻译,允许在运行时指定代数索引 (expert_id, offset) 而非固定的物理地址
  • 微架构层: 在交换机内部部署片上转换表和缓存,实现低延迟的地址翻译
  • 运行时层: 动态更新转换表,反映每个 mini-batch 的最新 token-to-expert 映射关系
紧凑寻址: 代数索引到布局索引的翻译

关键创新在于使用代数索引 (algebraic index) 而非完整的物理地址。代数索引仅包含 expert_id 和 token 在 expert 内的偏移量,非常紧凑。硬件内存管理器通过查询片上转换表 (on-chip translation table) 和缓存,将代数索引翻译为实际的布局索引 (GPU_id + GPU 内内存地址)。这种紧凑寻址大幅降低了每个 token 的元数据开销。

片上转换表与缓存

转换表存储在交换机芯片的 SRAM 中,支持快速查找。缓存层 (类似 TLB) 存储最近使用的映射关系,命中时可实现单周期翻译。由于 MoE 的 token-to-expert 映射在同一个 mini-batch 内相对稳定,缓存命中率很高。

来源: arXiv:2605.05607 Section 4: Dynamic Multimem Addressing

紧凑寻址的意义

紧凑寻址 (compact addressing) 的设计维持了更高的有效载荷效率 (payload efficiency)。传统的物理地址方案需要为每个 token 携带完整的 GPU ID 和内存地址 (可能 8-12 字节),而代数索引方案只需携带 expert_id 和偏移量 (2-4 字节)。在高吞吐的 MoE 通信中,这意味着每个数据包的 payload 占比显著提升,交换机带宽被更有效地利用。

交换机多播与归约

Dispatch 多播 + Combine 归约

DySHARP 在交换机层面实现了两个关键的原语操作,分别对应 MoE 的 dispatch 和 combine 阶段:

Dispatch 阶段: 多播分发

多播机制

在 dispatch 阶段,当一个 token 需要被发送到多个 expert GPU (top-k routing) 时,DySHARP 交换机执行硬件级多播 (hardware multicast)。交换机从源 GPU 接收一次数据,然后在交换机内部复制并同时发送到 k 个目标 GPU。这避免了源 GPU 发送 k 次相同数据的冗余。

Combine 阶段: 部分结果归约

归约机制

在 combine 阶段,来自多个 expert GPU 的同一 token 的部分结果需要在交换机中执行求和归约 (sum reduction)。DySHARP 交换机支持在交换机内部直接对 partial results 进行元素级求和,然后将归约后的完整结果发送回源 GPU。这避免了源 GPU 接收多个 partial results 后在本地进行归约的额外开销。

自定义多播与归约指令

DySHARP 定义了专用的交换机指令,支持多播和归约操作。这些指令被集成到交换机的控制平面中,可以通过 GPU 侧的驱动直接调用。指令格式经过优化,使用极少的元数据即可表达复杂的通信模式。

分组携带一个地址 + 紧凑目标列表

关键设计: 每个分组仅携带一个源地址和紧凑的目标 GPU 列表,而不是为每个目标单独打包。交换机解析分组后,根据紧凑目标列表在内部复制数据并路由到多个目标。这大幅减少了网络上的元数据流量,提高了有效载荷占比。

以 Token 为中心的内核融合

深度融合 Dispatch-Computation-Combine 管线

DySHARP 不仅优化了通信层面,还重新设计了 GPU 内核的执行模式,实现了 dispatch、expert 计算和 combine 三阶段的深度融合 (deep fusion)。

深度管线融合

传统 MoE 实现中,dispatch、computation 和 combine 是三个独立的阶段,每个阶段完成后需要将中间结果写回显存,下一阶段再从显存读取。DySHARP 将这三个阶段融合为一个连续的管线:token 到达 expert GPU 后直接开始计算,计算结果立即进入 combine 管线。这种融合消除了中间显存读写,大幅降低了延迟。

双向流量平衡

在 MoE 中,dispatch 阶段的流量方向 (源 GPU -> expert GPU) 与 combine 阶段的方向 (expert GPU -> 源 GPU) 相反。DySHARP 通过并发执行 dispatch 和 combine 管线,使双向流量在时间上重叠,从而平衡了双向网络流量 (balances directional traffic reductions)。这避免了单向流量饱和导致的带宽浪费。

数据移动减少带来真实执行收益

单纯减少网络流量不一定转化为加速——如果 GPU 侧的数据重组开销很大,节省的网络时间可能被抵消。DySHARP 的 token-centric 设计确保了减少的数据移动确实转化为端到端的执行时间减少:紧凑寻址减少了元数据解析时间,交换机内归约减少了 GPU 侧的后处理时间。

来源: arXiv:2605.05607 Section 6: Token-Centric Kernel Fusion

为什么"减少流量"本身不够?

论文的一个重要发现是:仅减少流量本身不足以带来加速,因为单向带宽会变成不平衡状态 (directional bandwidth becomes unbalanced)。dispatch 阶段减少流量后,如果 combine 阶段仍然使用传统方式,网络的一个方向可能空闲而另一个方向过载。DySHARP 的解决方案是并发执行管线,使双向流量在时间上平衡——这是一个关键的系统设计洞察。

通信感知调度

并发执行平衡双向网络流量

DySHARP 的调度器是系统设计的"大脑"——它了解底层硬件的所有能力,并据此做出最优的调度决策,处理动态通信与不规则模式。

并发执行: 平衡双向网络流量

DySHARP 调度器的核心策略是并发执行 dispatch 和 combine 管线。当第一批 token 完成 dispatch 并到达 expert GPU 后,调度器立即启动 combine 管线,而不是等待所有 dispatch 完成后才开始 combine。这种并发执行使 dispatch 流量 (GPU -> expert) 和 combine 流量 (expert -> GPU) 在网络中同时存在,从而平衡了双向带宽利用率。

处理动态通信与不规则模式

MoE 的通信模式在每个 mini-batch 甚至每个 step 都不同 (因为 gating 决策是动态的)。DySHARP 调度器能够:

  • 动态解析每个 token 的目标 GPU 集合 (varying targets)
  • 根据当前 token 分布调整 dispatch/combine 的并发度
  • 利用交换机的多播/归约能力,在运行时决定最优的数据传输策略
  • 处理极端不均匀的 token 分布 (某些 expert 被大量 token 选中,某些几乎没有)
调度策略与硬件能力协同

调度器不是独立于硬件的软件层——它与 DySHARP 的硬件能力深度协同。调度器了解交换机的多播能力、归约能力、以及动态寻址表的容量限制,并据此调整调度决策。这种协同设计使得调度器能够最大化利用硬件加速,而不是被硬件限制所束缚。

加速比解读

1.79x vs 领先基线: 与当前最优的 MoE 通信优化方案相比,DySHARP 实现了 1.79 倍的端到端加速。这包括了最先进的软件层 token 打包和通信调度优化。
1.93x 几何平均 vs 优化通信库: 与 NCCL 等优化通信库相比,DySHARP 的几何平均加速为 1.93x。这个加速来自交换机内多播/归约消除的冗余传输,以及紧凑寻址降低的元数据开销。
3.38x 几何平均 vs 静态交换机变通方案: 与使用静态交换机 (如 NVSwitch) 的变通方案相比,DySHARP 的几何平均加速达到 3.38x。这反映了动态寻址和多播/归约对静态方案的根本性优势。
4.25x 几何平均 MoE 层加速: 仅看 MoE 层 (dispatch + expert computation + combine),DySHARP 的几何平均加速高达 4.25x。这个数据最能体现 DySHARP 架构对 MoE 通信瓶颈的直接优化效果。

硬件开销分析

极低的硬件面积开销

DySHARP 的硬件扩展极其精简——它不需要在交换机芯片上添加大量的逻辑单元,而是通过巧妙的 ISA 扩展和紧凑的数据结构设计实现功能。

< 0.1%
NVSwitch

NVSwitch 芯片面积

DySHARP 的硬件扩展 (地址翻译表、缓存、多播/归约逻辑) 仅占 NVSwitch 芯片总面积的 0.1% 以下。考虑到 NVSwitch 本身的面积约为 2000 mm² (估算),DySHARP 的扩展面积不到 2 mm²。

H100 GPU

H100 GPU 芯片面积

相对于 H100 GPU 的芯片面积 (约 814 mm²),DySHARP 的开销仅约为 0.024%,约 0.2 mm²。这是一个极小的面积代价,几乎可以忽略不计。

开销来源分析

DySHARP 的硬件开销主要来自以下组件:

  • 片上地址转换表 (Translation Table): SRAM 存储,大小与 expert 数量和 GPU 数量成正比
  • 转换缓存 (Translation Cache): 类似 TLB 的小容量缓存,存储热映射条目
  • 多播复制逻辑 (Multicast Replication Logic): 交换机内部的数据复制单元,根据紧凑目标列表生成多个输出副本
  • 归约单元 (Reduction Unit): 交换机内的元素级求和电路,支持浮点加法归约

极小开销换取巨大收益

DySHARP 最令人印象深刻的设计权衡之一是:以极小的硬件面积代价 (< 0.1% NVSwitch die),换取了 MoE 通信 2-4x 的加速。这种"小硬件改动、大性能收益"的设计哲学是系统架构研究的典范。相比之下,许多其他硬件加速方案需要显著的面积和功耗开销才能获得类似量级的性能提升。

关键发现

发现 1: 仅减少流量不够 —— 并发管线是必需的

减少数据流量本身不足以带来加速,因为单向带宽会变得不平衡 (directional bandwidth becomes unbalanced)。dispatch 阶段减少流量后,如果 combine 阶段不进行并发优化,网络的一个方向可能空闲而另一个方向过载。并发执行 dispatch 和 combine 管线是平衡双向流量、充分利用网络带宽的必要条件。

发现 2: 紧凑寻址维持更高的有效载荷效率

DySHARP 的紧凑寻址方案 (代数索引 + 片上翻译表) 相比传统的完整物理地址方案,维持了更高的 payload efficiency。每个数据包中,有效数据占比显著提升,交换机带宽被更有效地利用。这一设计在极端高吞吐的 MoE 通信中尤为重要。

发现 3: 跨 GPU 数量/序列长度/token 分布的可扩展性

DySHARP 的性能优势在不同实验条件下保持一致:无论是增加 GPU 数量、变化序列长度,还是使用极度不均匀的 token 分布 (某些 expert 被大量选中,某些几乎空闲),DySHARP 都能保持显著的加速。这表明其设计具有良好的可扩展性和鲁棒性。

发现 4: 可扩展到多节点集群

DySHARP 的设计不仅适用于单节点内的多 GPU 系统,还可以扩展到多节点集群 (multi-node clusters)。动态寻址和多播/归约的原语可以跨节点运行,使 DySHARP 能够处理更大规模的 MoE 训练和推理工作负载。

发现 5: 交换机内计算是 MoE 通信的未来方向

随着 MoE 模型规模的持续增长,通信开销占比将进一步上升。DySHARP 证明了在交换机层面执行计算 (多播、归约) 是一个有效且低开销的方向。这为未来的 GPU 互连架构设计提供了重要的启示:交换机不应仅仅是数据转发设备,而应该成为计算-通信协同的加速节点。

发现 6: 硬件-软件协同设计的必要性

DySHARP 的成功证明了单纯优化软件 (通信库) 或单纯优化硬件 (交换机) 都无法充分解决 MoE 的动态通信问题。只有当硬件扩展 (动态寻址、多播/归约) 与软件调度 (并发管线、通信感知) 协同设计时,才能实现最佳的性能效果。

影响分析

对 GPU 互连架构的影响

DySHARP 证明了在交换机层面添加少量逻辑 (地址翻译、多播、归约) 即可带来 MoE 通信 2-4x 的加速,硬件开销不到芯片面积的 0.1%。这一发现可能推动未来 GPU 互连芯片 (如下一代 NVSwitch) 在硬件层面原生支持动态通信原语,而非仅支持静态组播。

对 MoE 大模型训练的影响

对于 DeepSeek-V3、Mixtral、Qwen-MoE 等万亿参数 MoE 模型,DySHARP 提供的 4.25x MoE 层加速可以直接转化为训练时间的显著减少。在大规模训练中,通信通常占总时间的 40% 以上,MoE 层的加速效果将直接影响整体训练效率。

对通信库设计的影响

DySHARP 展示了纯软件层优化 (如 NCCL) 在处理 MoE 动态通信时的根本局限性。这为通信库的设计指明了新方向:未来的通信库可能需要与底层硬件 (交换机) 紧密协同,暴露动态寻址和多播/归约能力给软件层调用。

对数据中心网络的影响

DySHARP 的"智能交换机"思路可以扩展到更广泛的数据中心场景。当交换机能够执行简单的计算 (如归约、聚合) 时,可以大幅减少网络流量和端点处理开销。这一思路与 in-network computing 的研究方向高度一致。

对开源社区的影响

DySHARP 为开源 MoE 框架 (如 DeepSpeed, Megatron-LM) 提供了明确的优化方向。虽然 DySHARP 需要硬件支持,但其调度策略和 token-centric 管线设计可以在软件层面部分实现,为没有定制硬件的用户提供渐进式优化路径。

对未来研究方向的启示

DySHARP 开启了一系列值得探索的研究方向: (1) 动态寻址算法的进一步优化; (2) 交换机内更复杂的计算 (如非线性归约); (3) 与其他通信模式 (如张量并行) 的结合; (4) 在真实硬件上的实现和验证。这些都是系统研究领域的高价值课题。

总结

《Accelerating MoE with Dynamic In-Switch Computing on Multi-GPUs》是上海交通大学与华为联合研究的一项突破性工作,针对 MoE 模型中动态、不规则的 GPU 间通信瓶颈,提出了 DySHARP (Dynamic Switch-based Hardware Accelerated Routing and Processing) 框架。该框架通过扩展静态交换机能力,在交换机层面实现动态寻址、多播分发、归约聚合,并结合 token-centric 内核融合和通信感知调度,实现了 MoE 通信的全面优化。

论文的核心贡献包括:(1) 揭示了 MoE dispatch/combine 通信中的根本挑战——varying targets 和 asymmetric addressing;(2) 设计了动态多内存寻址方案,通过代数索引到布局索引的翻译实现紧凑寻址;(3) 实现了交换机内多播和归约硬件原语;(4) 提出了以 token 为中心的内核融合管线;(5) 设计了通信感知的并发调度策略。在 32-GPU GH200 系统仿真中,DySHARP 实现了近半流量削减和最高 4.25x 的 MoE 层加速,硬件开销仅占 NVSwitch 芯片面积的 0.1% 以下。

这项工作的意义在于,它证明了"让交换机变智能"——以极小的硬件代价在交换机层面添加动态计算能力——是解决 MoE 通信瓶颈的有效途径。随着 MoE 模型在万亿参数 LLM 中的广泛应用,DySHARP 的架构理念和具体设计将对未来 GPU 互连架构、通信库设计、以及数据中心网络产生深远影响。
