CFT Handle 地址构造公式推导、寄存器数据流、以及 UBLKCP 如何消费远端 Handle 操作数

0. 全局视角：从 PTX 到 SASS 的地址构造链路

核心问题

PTX 的 fabric.try_get.async 接受一个 CFT Handle 操作数 [leId, offset]，其中 leId 是裸的 32-bit 逻辑端点 ID，offset 是裸的 64-bit 资源偏移。但在 SASS 层，UBLKCP 指令不接受裸值 — 它要求操作数编码为硬件特定的 64-bit Fabric Handle 地址格式。

阶段3的任务就是完成这个编码；阶段4的任务是用编码后的地址发起实际的 Fabric 数据传输。

PTX 视角（逻辑地址）: ┌──────────────────────────────────────────────────────────┐ │ fabric.try_get.async [%rd5], [%r1, %rd1], %r2, [%rd3] │ │ ↑ leId ↑ offset │ │ (u32) (u64) │ └──────────────────────────────────────────────────────────┘ SASS 视角（编码后物理地址）: ┌──────────────────────────────────────────────────────────┐ │ UBLKCP.S.G.STRONG.SYS [UR12], [UR6], UR10 │ │ ↑ 64-bit encoded CFT handle │ │ = encode(leId) ⊕ offset │ └──────────────────────────────────────────────────────────┘ 阶段3 负责: leId + offset → UR6/UR7 (64-bit encoded handle) 阶段4 负责: 用 UR6/UR7 发起 UBLKCP + 重试循环

1. 阶段3：远端 CFT Handle 地址构造

Phase 3: 0x0140 - 0x0160 | leId 编码 → offset 合并 → 64-bit Handle 就绪

1.1 输入寄存器状态（阶段3开始时）

寄存器来源内容位宽
UR140x0030: LDCU UR14, c[0x0][0x380]src_le_id（逻辑端点 ID）u32
UR8/UR90x0080: LDCU.64 UR8, c[0x0][0x388]src_offset（资源偏移）u64

这两个值直接来自 PTX kernel 参数 src_le_id 和 src_offset，经过阶段1的参数加载后保存在 Uniform 寄存器中。

1.2 指令 0x0140：leId → Fabric 地址空间编码

0x0140
ULEA UR7, UR14, 0x200000, 0xa
LE ID 编码 将裸 leId 编码为 Fabric Handle 地址空间中的路由标识。

ULEA（Uniform Load Effective Address）执行线性地址计算。这里使用两操作数+双立即数形式：
ULEA 编码公式推导：
ULEA 指令语义：
UR7 = f(UR14, imm1=0x200000, imm2=0xa)

其中：
  • UR14 = src_le_id（裸的 32-bit 逻辑端点 ID）
  • 0x200000 = Fabric 地址空间基址常量（bit 21 置位）
  • 0xa = 10 = 位域移位常量

推导：
  UR7 = (0x200000 << 10) | (src_le_id << 10)
      = 0x80000000 | (src_le_id << 10)

含义：
  • 0x200000 << 10 = 0x80000000
    → 设置 bit 31，标记这是 Fabric 地址（非本地内存）
  • src_le_id << 10
    → 将 leId 放入地址的 [30:10] 位域，用于 NVSwitch 路由
  • 低 10 位 = 0
    → 留给 offset 合并时使用

UR7 = 0x80000000 | (src_le_id << 10)

为什么 leId 需要编码？

PTX 的 [%r1, %rd1] 中 %r1 是裸的 leId 值，但 Blackwell 的 Fabric 硬件（TMA/Fabric 命令引擎）期望操作数编码为统一的地址格式，其中包含：

1) 地址类型标记（bit 31 = 1 表示 Fabric 远端地址，区别于本地/系统内存）

2) LE ID 路由字段（放置在特定位域中，NVSwitch 硬件直接从中提取路由信息）

3) Offset 占位区域（低位保留给 offset 合并）

这种编码使得 UBLKCP 指令可以用单一 64-bit 操作数同时携带路由目标和资源偏移，无需额外的操作数字段。

1.3 指令 0x0150：清零高 32 位准备

0x0150
UMOV UR6, URZ
地址构造 清零 UR6。UR6 将作为 64-bit CFT Handle 的高 32 位（初始为 0），UR7 作为低 32 位（已编码 leId）。

此时 64-bit 寄存器对 UR6:UR7 的状态：
64-bit Handle 中间状态（0x0150 之后）：

  UR6 (高32位) = 0x00000000
  UR7 (低32位) = 0x80000000 | (src_le_id << 10)

  64-bit handle = {UR6, UR7}
              = 0x00000000_80000000 | (src_le_id << 10)

此时 handle 只包含 leId 路由信息，offset 尚未合并。

1.4 指令 0x0160：offset 合并到 Handle

0x0160
UIADD3.64 UR6, UPT, UPT, UR6, UR8, URZ
地址合并 64-bit 加法：将 src_offset 合并到已编码 leId 的 Handle 中。

UIADD3.64 是 64-bit Uniform 整数加法，操作数：
  • 目标：UR6（64-bit，覆盖 UR6/UR7）
  • 操作数A：UR6（= 0x00000000，高32位）
  • 操作数B：UR8（= src_offset，64-bit，占 UR8/UR9）
执行 64-bit 加法后，UR6/UR7 包含完整的 CFT Handle。
对应 PTX: [%r1, %rd1] — leId 和 offset 的 SASS 编码合并
64-bit Handle 最终状态（0x0160 之后）：

  UIADD3.64:
    {UR6, UR7} = {UR6, UR7} + {UR8, UR9}
               = (0x00000000_80000000 | (leId << 10)) + src_offset

  由于 leId 编码后低 10 位 = 0，且 src_offset 通常 16B 对齐（低 4 位 = 0），
  加法等价于 OR（无进位冲突）：

  {UR6, UR7} = 0x00000000_80000000 | (leId << 10) | src_offset

  展开为位域：
  ┌─────────────────────────────────────────────────────────┐
  │  Bit 63-32 (UR6): 0x00000000                         │
  │  Bit 31:          1        (Fabric 地址标记)             │
  │  Bit 30-10:      leId     (路由字段)                   │
  │  Bit 9-0:        offset[9:0]  (页内偏移)                   │
  │  Bit 63-32 高位:  offset[41:10] (页号)                       │
  └─────────────────────────────────────────────────────────┘

最终 64-bit CFT Handle = encode(leId) ⊕ offset

为什么加法等价于 OR？

leId 编码后（leId << 10），低 10 位全为 0。而 src_offset 是 NVSHMEM heap 上的偏移，根据 CFT 约束必须 16B 对齐（低 4 位 = 0）。由于 leId << 10 占据 bit [30:10]，而 offset 的低 10 位占据 bit [9:0]，两者位域不重叠，因此加法不产生进位，等价于按位 OR。

但如果 offset 足够大（>= 1024），其高位会与 leId 字段在 bit 空间上"拼接"，形成完整的 64-bit 地址。这正是 CFT Handle 的设计：leId 决定路由目标，offset 决定目标端内的资源位置。

2.2 指令 0x0170：Warp 线程选举

0x0170
@P0 ELECT P1, URZ, PT
线程选举 仅当 P0=True 时执行（首次进入或重试时）。ELECT 从当前 warp 的 32 个 lane 中选出一个 lane 来发起 Fabric 事务：

选举机制：
  • 被选中的 lane：P1 = True（获得发起权限）
  • 其他 31 个 lane：P1 = False（等待）
  • URZ：目标寄存器未使用（选举结果只写入谓词 P1）
  • PT：谓词 true mask，表示所有 lane 都参与选举
为什么需要选举？ Fabric 操作通过 NVLink/TMA 引擎发送请求，这些引擎每个 warp 只需要一个 lane 代表整个 warp 发起事务。这是 Blackwell SIMT 模型在 Fabric 侧的延伸。
对应 PTX: 无 — PTX 不暴露硬件级线程选举

2.3 指令 0x0180：UBLKCP — Fabric 读取核心指令

0x0180
UBLKCP.S.G.STRONG.SYS [UR12], [UR6], UR10
Fabric 读取 这是 fabric.try_get.async 的 SASS 核心实现。

UBLKCP = Unaligned Block Copy，Blackwell 的硬件块拷贝指令，支持 Fabric 远端访问。

UBLKCP 操作数详解 — 阶段3 输出的消费方式

操作数值来源内容UBLKCP 中的角色
[UR12] 阶段2 (0x0130) smem_data 的编码 shared memory 地址 目标地址 (dst)：数据写入本地 shared memory 的位置
[UR6] 阶段3 (0x0160) 64-bit 编码 CFT Handle = 0x80000000 | (leId << 10) | offset 源地址 (src)：远端 Fabric Handle，TMA 引擎从中提取 leId 用于路由，提取 offset 用于寻址

UBLKCP 修饰符解析

修饰符含义对应 PTX
.SSource 端通过 Fabric proxy（远端读取）fabric (源操作数)
.GGeneric 寻址模式（操作数是编码后的通用地址）隐式
.STRONG.SYS系统级强内存序.relaxed.sys

TMA 引擎如何解析 [UR6] 中的 CFT Handle

当 UBLKCP 执行时，TMA/Fabric 命令引擎从 [UR6] 的 64-bit 值中提取：

CFT Handle 位域解析（TMA 引擎视角）：

  64-bit value in UR6 = 0x00000000_8XXXXXXX (示例)

  Bit 31 = 1:
    → 地址类型 = FABRIC（非本地/系统内存）
    → TMA 路由到 Fabric 命令管道

  Bit 30-10 = leId 字段:
    → 从 Fabric Handle 中提取 Logical Endpoint ID
    → NVSwitch 使用 leId 查找路由表，定位目标 GPU
    → 目标 GPU 的 LE 引擎接收请求

  Bit 9-0 (+ UR6 高位拼接) = offset 字段:
    → 目标 GPU 内部使用 offset 定位具体资源
    → offset 相对于 LE 绑定的 heap 基址

TMA 引擎: UR6 → {fabric_flag=1, leId, offset} → NVLink 路由 → 远端内存读取

2.4 指令 0x0190：UBLKCP 状态检查

0x0190
@P1 PLOP3.LUT P0, PT, P1, PT, PT, 0x8, 0x80
状态检查 仅由被 ELECT 选中的 lane (P1=True) 执行。根据 UBLKCP 的返回状态更新 P0（重试控制谓词）：

PLOP3.LUT 通过查找表实现逻辑运算，操作数 PT, P1, PT, PT 和 LUT 值 0x8 配合操作码 0x80 实现：
状态检查逻辑：

  如果 UBLKCP 成功发出 (P1_status = success):
    P0 = False
    → 退出重试循环
    → 继续执行 0x01c0 (UTMACMDFLUSH)

  如果 UBLKCP 发出失败 (P1_status = busy):
    P0 = True
    → 保持重试循环
    → 跳回 0x0170 (ELECT)

  LUT 真值表 (0x8 = 0b00001000):
    输入组合 → 输出
    仅当特定输入组合为 True 时输出 True
    → 实现 "UBLKCP busy 且当前 lane 是 elected lane" 的条件检测

2.5 指令 0x01a0：谓词 P1 重置

0x01a0
PLOP3.LUT P1, PT, PT, PT, PT, 0x8, 0x80
谓词重置 无条件执行（无谓词前缀）。将 P1 重置为初始状态，为下一次循环迭代中的 ELECT 指令做准备。确保每次循环迭代都从干净的 P1=False 状态开始。

2.6 指令 0x01b0：条件回跳（重试循环闭合）

0x01b0
@P0 BRA.U.ANY 0x170
条件回跳 如果 P0=True（UBLKCP 发出失败），跳转回 0x0170 的 ELECT 指令。BRA.U.ANY 的 ANY 表示只要 warp 中有任意 lane 的 P0 为 True 就跳转。

3.1 完整公式汇总

公式 1：leId 编码（0x0140）

  UR7 = (0x200000 << 10) | (src_le_id << 10)
      = 0x80000000 | (src_le_id << 10)

  其中:
    0x200000 = Fabric 地址空间基址常量
    0xa = 10 = 位域移位量
    src_le_id = PTX kernel 参数 (u32)

公式 2：64-bit Handle 构造（0x0150 + 0x0160）

  {UR6, UR7} = {0, 0x80000000 | (leId << 10)} + src_offset
             = 0x80000000 | (leId << 10) | offset

  其中:
    src_offset = PTX kernel 参数 (u64)
    加法等价于 OR（位域不重叠）

公式 3：CFT Handle 位域布局

  Bit 63:32  = offset[41:10]  (高位偏移，通常 = 0)
  Bit 31     = 1              (Fabric 地址标记)
  Bit 30:10  = leId[20:0]     (逻辑端点 ID)
  Bit 9:0    = offset[9:0]    (页内偏移)

公式 4：UBLKCP 语义

  UBLKCP.S.G.STRONG.SYS [dst_smem], [src_cft_handle], [tx_size]

  等价于 PTX:
    fabric.try_get.async.shared::cta
      .mbarrier::complete_tx::bytes.mbarrier::report::fabric
      .relaxed.sys.b128
      [dst_smem], [src_cft_handle], tx_size, [mbarrier]

  其中:
    dst_smem          = UR12 (阶段2 编码的 SMEM 地址)
    src_cft_handle    = UR6 (阶段3 编码的 64-bit CFT Handle)
    tx_size           = UR10 (阶段2 编码的 byte_count)

公式 5：重试循环

  do {
    elected_lane = ELECT(warp)           // 0x0170
    status = UBLKCP(dst, src, size)      // 0x0180
    if (status == SUCCESS) break         // 0x0190: P0 = False
    if (status == BUSY) continue         // 0x0190: P0 = True → 0x01b0 → 0x0170
  } while (true)

核心洞察

阶段3 的本质是将 PTX 的"逻辑"地址（裸 leId + 裸 offset）转换为 Blackwell TMA/Fabric 引擎能直接消费的"物理"编码地址。这种编码不是任意的 — 它精确对应了 NVSwitch 硬件的路由表查找格式：bit 31 标记地址类型，bit [30:10] 承载 leId 用于路由，低位承载 offset 用于目标端内寻址。

阶段4 的本质是将编码后的 CFT Handle 交给 UBLKCP 指令，由 TMA 硬件引擎解析并发起 NVLink 传输。"try" 语义通过 ELECT + 条件分支实现软件重试 — 这揭示了 PTX 的 "try" 前缀不是装饰，而是对应了真实的硬件 busy 可能性。
