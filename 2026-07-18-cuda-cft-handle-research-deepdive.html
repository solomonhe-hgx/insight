1. CFT 概述

CUDA Compute Fabric Transport (CFT) 是 NVIDIA 在 PTX ISA 9.3 / CUDA 13.3 中引入的一种编程模型，专为 NVLink 互联域内的跨 GPU 内存访问而设计。它是对传统指针式 P2P（Peer-to-Peer）内存访问的演进，核心思想是将远程内存访问从"基于指针"转变为"基于句柄"，从而实现对 NVLink 故障的容错能力。

1.1 核心动机

传统的基于指针的 NVLink 内存操作（如直接通过 peer pointer 进行 load/store）在遇到 NVLink 故障时是致命的（fatal）——程序会直接崩溃。CFT 编程模型引入了 fabric operations，能够将故障直接报告给应用程序，使应用具备弹性恢复能力：

"NVLink failures encountered by pointer-address-based memory operations are fatal to the application. The CUDA Compute Fabric Transport (CUDA CFT) programming model introduces fabric operations that report failures directly to the application, enabling them to be resilient over NVLink failures."
— PTX ISA 9.3, Section 9.7.10

应用可以选择如何处理这些故障：中止、优雅地拆除进程、或者重试操作。

2. CFT Handle 的结构

一个 CUDA Fabric Transport Handle 由两个核心组件构成：

组件 类型 含义
leId uint32_t Logical Endpoint Identifier — 逻辑端点标识符，标识一个远端逻辑端点
off (offset) uint64_t Resource Offset — 该逻辑端点内某个资源的偏移量 
CFT Handle = { leId (32-bit), offset (64-bit) }

2.1 Logical Endpoint（逻辑端点）

逻辑端点是 CFT 模型中的核心抽象，代表 fabric 上一个可寻址的实体。它有两种类型：

  • Unicast（单播）：指向单个 GPU 上的特定资源
  • Multicast（多播）：指向一组 GPU 上的资源（用于集合通信等场景）

2.2 生命周期

CFT Handle 的生命周期始于并终于主机端（host）：

  1. 主机端检测系统是否支持 fabric operations
  2. 创建逻辑端点（cuLogicalEndpointCreate）
  3. 绑定资源到逻辑端点（cuLogicalEndpointBindAddr / cuLogicalEndpointBindMem）
  4. 通过 IPC 导出/导入句柄，实现跨进程共享（cuLogicalEndpointExport / cuLogicalEndpointImport）
  5. 查询端点就绪状态（cuLogicalEndpointQuery）
  6. 设备端（device）发起 fabric operations
  7. 解绑并销毁端点
  8. 释放 ID 范围

3. le_id 和 offset 在 Driver 层的处理

3.1 Driver 层的职责

CUDA Driver 在整个流程中承担以下关键职责：

(a) ID 管理

CUresult cuLogicalEndpointIdReserve(CUlogicalEndpointId* baseLeId, cuuint32_t count);
CUresult cuLogicalEndpointIdRelease(CUlogicalEndpointId baseLeId, cuuint32_t count);

Driver 负责分配和回收 leId 空间。leId 是一个 32 位无符号整数，driver 维护着从 leId 到实际物理端点（GPU 设备 + 内存区域）的映射关系。

(b) 端点创建与配置

CUresult cuLogicalEndpointCreate(CUlogicalEndpointId leId, const CUlogicalEndpointProp* prop);

Driver 根据 CUlogicalEndpointProp 中的属性配置端点：
- Unicast 端点：绑定到一个 owner device
- Multicast 端点：需要指定 team size，后续通过 cuLogicalEndpointAddDevice 添加成员

(c) 资源绑定（核心：建立 offset → 物理地址的映射）

CUresult cuLogicalEndpointBindAddr(
    CUlogicalEndpointId leId,    // 哪个逻辑端点
    CUdevice dev,                // 哪个设备
    cuuint64_t offset,           // 端点空间内的偏移
    void* ptr,                   // 虚拟地址
    cuuint64_t size,             // 大小
    unsigned long long flags
);

CUresult cuLogicalEndpointBindMem(
    CUlogicalEndpointId leId,
    CUdevice dev,
    cuuint64_t offset,           // 端点空间内的偏移
    CUmemGenericAllocationHandle memHandle,
    cuuint64_t memOffset,        // 物理内存块内的偏移
    cuuint64_t size,
    unsigned long long flags
);

这是 offset 含义建立的关键步骤。 Driver 在此处建立了如下映射：

(leId, offset) → (device, physical_memory_location)

具体来说：
- offset 是逻辑端点地址空间内的偏移量，代表资源在该端点内的起始位置
- Driver 将 (leId, offset, size) 与具体的物理内存（由 ptr 或 memHandle+memOffset 指定）关联起来
- 绑定操作必须满足对齐和大小限制（通过 cuLogicalEndpointGetLimits 获取）

(d) IPC 导出/导入

CUresult cuLogicalEndpointExport(void* handle, CUlogicalEndpointId leId,
                                  CUlogicalEndpointIpcHandleType handleType);
CUresult cuLogicalEndpointImport(CUlogicalEndpointId leId, const void* handle,
                                  CUlogicalEndpointIpcHandleType handleType);

Driver 将 leId 及其绑定信息编码到 CUlogicalEndpointFabricHandle 中，使得其他进程可以通过 leId 访问同一份远程资源。底层依赖 IMEX daemon（NVLink 域内的进程间通信守护进程）和 nvidia-caps-imex-channels 字符设备。

(e) 就绪查询

CUresult cuLogicalEndpointQuery(CUlogicalEndpointId leId, cuuint32_t count, int* queryStatus);

非阻塞地检查端点是否已完成构建（所有远端 GPU 都已同步元数据）。

4.2 leId 的硬件含义

leId 在硬件层面充当路由标识符：

  1. NVSwitch 路由表查找：NVSwitch 芯片维护着入口路由表（ingress routing table），其中包含 map slot entries。每个 map slot 定义了地址边界和一个软件可编程的 Target ID (TgtID)。
  2. 来源：US Patent 11182309B2 — "The central routing switch manages traffic via ingress tables containing 'map slots.' These entries define address boundaries and utilize software-programmable 'Target IDs' to direct traffic to specific hardware targets. The switch evaluates incoming physical addresses against these programmed parameters to determine the correct egress path."

  3. [推断] 报告中声称"leId 被映射到特定的 TgtID"——专利原文描述的是物理地址通过 map slot 查 TgtID 的路由机制，并未直接使用 leId 这一术语（leId 是 PTX ISA 9.3 / CUDA Driver API 中的概念）。将 leId 与 TgtID 关联是基于两套文档的合理推断：leId 在 driver 层绑定了具体设备和内存资源，硬件层面需要通过某种机制（推测为 TgtID 映射）将请求路由到正确的目标。

  4. Unicast vs Multicast 路由：

  5. Unicast leId：路由到单个目标 GPU
  6. Multicast leId：路由到多个 GPU（NVSwitch 执行多播复制）

  7. 来源：CUDA Driver API 定义了 CU_LOGICAL_ENDPOINT_TYPE_UNICAST 和 CU_LOGICAL_ENDPOINT_TYPE_MULTICAST；NVSwitch 的多播能力来自其 crossbar 架构的公开描述。

  8. Peer Aperture 机制：Source GPU 使用 peer aperture（物理地址空间中的映射窗口）来发起对远端的访问。Page table entry 引导流量到特定的 hub，hub 决定分发宽度。

  9. 来源：US Patent 11182309B2 — "source processor utilizes a 'peer aperture,' functioning as a specific window into a remote memory slice"

4.3 offset 的硬件含义

offset 在硬件层面是端点内地址空间中的偏移：

  1. 资源定位：offset 标识了逻辑端点内特定资源的起始位置。当 driver 执行 BindAddr(leId, dev, offset, ptr, size) 时，它建立了 offset → ptr 对应的物理地址 的映射。

  2. 来源：CUDA Driver API — cuLogicalEndpointBindAddr 参数描述为 "Offset into the logical endpoint space"

  3. 经过 NVSwitch 的变换：

  4. Source GPU 先对 offset 做 swizzle（位操作生成 entropy，用于链路选择）
    • 来源：US Patent 11182309B2 — "source processor employs a 'swizzle' technique. By manipulating address bits via mathematical strings, the system generates entropy. This pseudo-randomly distributes requests across multiple physical links"
  5. NVSwitch 做 de-swizzle（逆转 swizzle）
    • 来源：US Patent 11182309B2 — "the switch must first reverse the initial entropy manipulation"
  6. NVSwitch 做 compaction（除以活跃链路数，消除空洞）
    • 来源：US Patent 11182309B2 — "divides the address by the number of active links so the memory module receives consecutive, usable addresses without collisions"
  7. NVSwitch 可能再施加一个固定偏移（用于虚拟化隔离）
    • 来源：US Patent 11182309B2 — "the switch can apply 'adding a fixed offset to the compacted' address to create distinct, isolated partitions"

  8. 最终得到目标内存模块上的实际物理地址

  9. 对齐要求：offset 和 size 必须满足 driver 报告的对齐限制（通过 cuLogicalEndpointGetLimits 获取的 bindAlignment 和 maxSize）。

  10. 来源：CUDA Driver API — cuLogicalEndpointGetLimits

4.4 硬件错误处理

与传统指针式 P2P 访问不同，CFT 的硬件路径在遇到 NVLink 故障时：
- 不会致命崩溃
- 将错误状态编码到 mbarrier 的 report 中
- 应用可以通过 cudaFabricOpErrorStatusGet/Count API 解码错误信息
- 应用可以选择重试或优雅降级

来源：PTX ISA 9.3 Section 9.7.10.4.1 — "If the operation succeeds, the payload report is not modified... Otherwise, the operation fails, the mbarrier report predicate is set, and the opaque report value may contain more information, which may be decoded via the cudaFabricOpErrorStatusGet/Count APIs."

5. PTX 指令层面的使用

5.1 四种 Fabric 操作

指令 功能 方向
fabric.try_get 从 fabric handle 异步读取 Remote → Local (shared memory)
fabric.try_put 向 fabric handle 异步写入 Local (shared memory) → Remote
fabric.try_red 向 fabric handle 做归约写入 Local (shared memory) → Remote (with reduction)
fabric.try_pullred 从 fabric handle 做拉取归约 Remote → Local (with reduction)

5.2 指令语法示例

// 从远端读取: srcLeId 标识远端端点, srcLeOff 标识端点内偏移
fabric.try_get.async.shared::cta
    .mbarrier::complete_tx::bytes.mbarrier::report::fabric
    .relaxed.sys.b128
    [dstSmem],              // 目标: 本地 shared memory
    [srcLeId, srcLeOff],    // 源: CFT Handle = {leId, offset}
    sizeBytes,              // 传输大小
    [mbar];                 // 用于完成追踪的 mbarrier

// 向远端写入: dstLeId 标识远端端点, dstDataOff 标识端点内偏移
fabric.try_put.async.shared::cta
    .mbarrier::complete_tx::16B.mbarrier::report::fabric
    .relaxed.sys.b128
    [dstLeId, dstDataOff],  // 目标: CFT Handle = {leId, offset}
    [src],                  // 源: 本地 shared memory
    size,                   // 传输大小
    [bar];                  // 用于完成追踪的 mbarrier

5.3 操作流程

1. Issue:    线程执行 fabric.try_get/put/red/pullred 指令
2. Submit:   线程执行 fabric.submit 提交操作
3. Arrive:   线程在 mbarrier 上 arrive + expect_tx
4. Wait:     线程等待 mbarrier phase 完成
5. Inspect:  检查 report predicate 判断是否有错误
6. Handle:   根据错误情况决定重试/中止/降级

层级 le_id 的作用 offset 的作用 来源
