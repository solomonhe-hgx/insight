🟢 NVIDIA NCCL

NCCL v2.30.3-1 发布 — 引入 Device API 新版本

NCCL 发布 v2.30.3-1 版本，最大亮点是引入了 Device-side API，允许在GPU端直接发起集合通信操作，而不再完全依赖Host端调度。这为低延迟推理场景和GPU-initiated通信打开了新可能性。

GitHub Releases Issue #2105

NCCL EP 论文公开 — 统一Expert Parallel通信API 研究

NVIDIA 在 arXiv 发布 "NCCL EP: Towards a Unified Expert Parallel Communication API" 论文（2603.13606），提出基于Device API构建的MoE通信扩展。核心特点：

  • LL模式（Low-Latency）：针对1-128 token小批量，采用直接all-to-all网格连接 + 双缓冲重叠通信
  • HT模式（High-Throughput）：针对4096+ token大批量，NVLink域内聚合后再节点间RDMA
  • 完全基于NCCL Device API重构，提供C/Python标准接口 ncclEpDispatch / ncclEpCombine
  • 已在vLLM中集成验证端到端可用性

与DeepEP等独立专用库不同，NCCL EP选择将MoE通信原生集成在NCCL内部，依托官方生态降低碎片化。

DeepEP v1.2.1 Stable Release 稳定版

DeepEP 发布 v1.2.1 稳定版本。DeepEP 是 DeepSeek 开源的 Expert Parallel 通信库，专为MoE模型优化，支持节点内NVLink + 节点间RDMA的混合通信。

社区有用户提出将 v1.2.1 发布到 PyPI 以方便安装使用。

🔶 其他通信库动态

▼

Meta 开源 RCCLX — AMD GPU通信增强 新库

Meta 开源了 RCCLX，基于AMD RCCL的增强版通信库，完全集成到 Torchcomms 框架。

  • DDA算法：Direct Data Access flat/tree算法优化节点内消息，MI300X上decode提升10-50%，prefill提升10-30%
  • 低精度集合：FP8量化压缩，计算保持高精度，延迟降低~9-10%，吞吐提升~7%，精度损失仅~0.3%
  • 统一API：与NVIDIA NCCLX共享Torchcomms接口，跨GPU厂商无需改API

🚀 趋势展望

▼

🔥 GPU-Initiated通信成为主流

NCCL v2.30.3引入Device API、NCCL EP基于Device API构建，标志着GPU自主发起集合通信正在成为行业标准。这将显著降低推理场景下的通信延迟，特别是对MoE模型的All-to-All操作。

⚔️ NCCL EP vs DeepEP：MoE通信路线之争

NCCL EP选择将MoE通信原生集成在NCCL内部，而DeepEP是独立专用库。NVIDIA官方路线的加入可能改变MoE通信的竞争格局，未来MoE训练/推理框架（如vLLM）可能优先采用NCCL EP作为标准后端。

🌐 跨厂商统一API趋势

Meta RCCLX与NVIDIA NCCLX共享Torchcomms接口，表明业界正在向"一次编写、跨厂商运行"的方向迈进。开发者无需因GPU品牌切换而修改通信代码，这将降低AI基础设施的厂商锁定风险。

📉 低精度通信加速落地

DeepSeek V4确认FP4/FP8支持、RCCLX实现FP8低精度集合（精度损失仅0.3%）、NCCL EP双模式架构——2026年低精度通信正在从研究走向生产环境，成为降低通信带宽需求的关键手段。

📌 本周观察

通信库领域正在经历"GPU-initiated革命"：NCCL的Device API和NCCL EP标志着通信控制权从Host向GPU转移的重大转变。与此同时，MoE通信路线开始分化（NCCL EP vs DeepEP），而Meta RCCLX的开源进一步推动了跨厂商API统一的趋势。本周是通信库生态格局变化的关键一周。
