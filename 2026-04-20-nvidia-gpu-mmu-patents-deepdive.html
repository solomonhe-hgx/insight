Memory management unit and method of walking page table

摘要

本发明公开了一种内存管理单元 (MMU) 架构，集成了快速地址翻译缓存 (translation cache)、待命命令缓冲区 (pending command buffer) 和并行遍历引擎 (parallel traversal engines)。当缓存未命中时，系统将共享相同虚拟地址前缀的待翻译命令分组，然后发出顺序内存访问请求，检索所需的映射条目并更新缓存。该设计通过批量处理针对同一物理内存行的翻译命令，顺序访存提升了 DRAM Row Buffer 命中率，从而加速地址解析而无需增加额外的并行遍历电路。

来源: Google Patents - US20240202136A1

技术背景

现代处理器依赖虚拟内存系统，需要通过分层目录结构 (页表) 进行地址翻译。页表遍历速度可能远低于处理器的处理速度。在 GPU 大规模并行工作负载中，频繁的 TLB Miss 会产生系统瓶颈。简单地增加更多遍历硬件会增加功耗和硅片面积。

关键问题: GPU 的 SIMT 架构产生数千个并行线程，每个线程可能访问不同的虚拟地址页面。当多个线程同时触发页表遍历时，传统的单线程遍历方式成为严重瓶颈。不规则内存访问模式 (如图遍历、稀疏矩阵、粒子模拟) 加剧了这一问题。

发明内容

核心架构组件

1. 翻译缓存 (Translation Cache)

相当于 TLB，存储近期使用过的虚拟-物理地址映射

2. 待命队列 (Pending Queue)

缓存未命中的翻译命令排队缓冲区

3. 遍历引擎 (Traversal Engine)

包含搜索电路、分组缓冲区、缓存行选择器、获取控制器和处理逻辑

4. 概率成员过滤器 (Probabilistic Membership Filter)

计数过滤器 (Counting Filter)，快速检测具有匹配地址前缀的命令

工作流程

1

选择待处理翻译命令: 遍历引擎从待命队列中选择一个翻译请求作为种子命令。

2

前缀匹配扫描: 使用计数过滤器快速扫描队列，找出所有在当前遍历深度具有相同虚拟地址前缀的命令。

3

目标缓存行识别: 通过位向量 (bit vector) 识别目标内存行中需要获取的缓存行。

4

顺序内存访问: "连续发出对缓存行的请求" 给内存控制器。第一次访问加载 DRAM 行，后续访问直接从感应放大器 (sense amplifiers) 读取，绕过主阵列。

5

提取映射数据: 从获取的数据中提取翻译条目，转发到缓存，或推进到下一级目录。

关键技术细节

多级页表支持

分组逻辑可根据当前遍历深度，对目录条目 (directory entries) 或最终翻译条目 (final translation entries) 进行分组。每个层级的基地址使用特定的高位比特范围动态计算。支持 2 级、3 级、4 级页表结构。

Row Buffer 大小与页面大小适配

分组逻辑适应不同的内存行 (Row Buffer) 和操作系统页面大小比例。处理 Row Buffer 大于或小于分配页面的情况。当 Row Buffer = 2KB、页面 = 4KB 时，一次激活可服务 2 个页面的翻译请求。

并行处理能力

每个遍历引擎每周期可同时处理 4 个或更多翻译命令。无需增加额外的并行遍历电路 (walker circuits)，仅通过智能分组和顺序访存即可实现性能提升。

Memory sharing via a unified memory architecture

技术影响

摘要

本发明公开了一种通过统一内存架构 (Unified Memory Architecture, UMA) 实现 CPU 和 GPU 内存共享的方法。该方法包括在内存中分配一个表面 (surface)，创建将该表面映射到虚拟地址空间的页表条目，MMU 包含 CPU 页表，支持统一内存架构下 CPU/GPU 共享虚拟地址空间。GPU 可以直接读取 CPU 页表条目，消除显式数据拷贝，简化编程模型。

来源: Google Patents - US9514559B2

技术背景

在传统的 GPU 编程模型中，CPU 和 GPU 拥有各自独立的虚拟地址空间和页表。数据需要在 CPU 内存和 GPU 显存之间显式拷贝 (cudaMemcpy)，这带来了以下问题：

  • • 程序员需要手动管理数据迁移，增加编程复杂度
  • • 拷贝操作消耗带宽和时间
  • • CPU 和 GPU 页表冗余，同步困难
  • • 无法实现细粒度的页面级迁移

发明内容

统一内存架构核心设计

1. 共享虚拟地址空间

CPU 和 GPU 使用相同的虚拟地址引用数据，消除地址空间分离。一个指针在 CPU 和 GPU 代码中指向相同的物理内存位置。

2. GPU MMU 读取 CPU 页表

GPU 的 MMU 包含 CPU 页表的访问能力。当 GPU 访问尚未迁移到本地显存的页面时，MMU 可以通过读取 CPU 页表找到物理位置，触发页面迁移。

3. 页面级迁移机制

当 GPU 访问未在本地显存中的页面时，产生页错误 (page fault)，驱动页面从 CPU 内存迁移到 GPU 显存。迁移以页面为单位 (通常 4KB)，支持细粒度数据管理。

4. 表面分配与映射

方法包括 "在内存中分配一个表面 (surface)"，创建将该表面映射到虚拟地址空间的页表条目。表面可以是纹理、缓冲区或任意数据结构。

工作流程

1

统一分配: 程序员使用 cudaMallocManaged() 分配统一内存，系统在虚拟地址空间中创建映射。

2

页表创建: 系统创建页表条目，将分配的表面对象映射到虚拟地址空间。CPU 和 GPU 共享相同的页表结构。

3

GPU 访问: GPU 线程访问虚拟地址时，GPU MMU 首先查询本地 TLB。如果 Miss，则遍历页表。页表可能在 CPU 内存中。

4

页面迁移: 如果页面不在 GPU 本地显存中，GPU MMU 产生页错误。驱动将页面从 CPU 内存迁移到 GPU 显存，更新页表。

5

后续访问: 迁移完成后，GPU 直接访问本地副本。页面可以在 CPU 和 GPU 之间动态迁移，由硬件和驱动自动管理。

关键技术细节

MMU 设计

GPU MMU 包含 CPU 页表的读取能力。MMU 126 包含 CPU 页表 (CPU page table)，可以解析 CPU 管理的虚拟地址。这使得 GPU 可以在不需要显式数据拷贝的情况下访问 CPU 分配的内存。

页错误处理

GPU 产生的页错误由 CPU 驱动处理。驱动决定将页面迁移到 GPU 显存还是保持原位。这支持了按需迁移 (on-demand migration) 策略，仅在真正需要时才迁移数据。

一致性模型

统一内存提供弱一致性模型。程序员需要使用 cudaDeviceSynchronize() 或 cudaStreamSynchronize() 确保 CPU 和 GPU 对共享内存的访问顺序正确。硬件不保证自动缓存一致性。

技术影响

该专利是 NVIDIA CUDA 统一内存 (Unified Memory) 特性的核心技术基础。从 Kepler 架构开始引入，在 Pascal、Volta、Ampere、Hopper 架构中持续增强。统一内存已成为 GPU 编程的标准范式，极大简化了异构编程模型。

编程简化

消除了显式 cudaMemcpy 调用，单个指针在 CPU 和 GPU 代码中通用

细粒度迁移

页面级 (4KB) 迁移粒度，相比粗粒度整个缓冲区拷贝更加高效

摘要

本发明公开了一种在并行图形处理器中控制缓存操作以提高缓存效率的系统和方法。架构动态调整主缓存和次缓存的优先级，结合指令级属性、默认设置和页表覆盖 (PTE overrides)。采用可配置的共享缓存分区和可选的硬件一致性协议。这些特性使缓存行为适应特定工作负载模式，最大化并行计算集群间的本地数据重用。

来源: Google Patents - US11620256B2

技术背景

GPU 包含多级缓存层次结构 (L1/L2/L3)，不同工作负载对缓存的使用模式差异很大。图形渲染需要高空间局域性缓存，而计算工作负载可能需要更大的容量或不同的替换策略。传统的固定缓存配置无法适应多样化工作负载需求，导致缓存命中率低下和资源浪费。

发明内容

核心技术创新

1. 动态缓存优先级调整

结合三种来源确定缓存行为: (a) 指令级属性 - 每条指令可以指定缓存优先级; (b) 默认设置 - 系统级默认缓存策略; (c) 页表条目 (PTE) 覆盖 - 页面级别的缓存策略覆盖默认规则。PTE 配置可以覆盖标准 L2/L3 优先级规则，实现细粒度、工作负载特定的内存路由。

2. PTE 内嵌缓存行索引

页表条目 (PTE) 包含用于映射虚拟地址到物理地址的条目，同时内嵌索引值。该索引值支持快速的缓存状态验证，无需额外的翻译步骤。通过在内嵌转换条目中包含缓存行索引，硬件可以即时判断获取是否成功。

3. 嵌入式 TLB 部署

为减少翻译延迟，设计将 "地址翻译后备缓冲区 (TLB) 或缓存" 直接部署在图形多处理器 (graphics multiprocessors) 或主缓存结构内部。靠近执行单元的部署允许线程在本地解析虚拟地址，避免到集中式内存控制器的昂贵往返。

4. 可配置共享缓存分区

支持 L2/L3 缓存的动态分区配置。不同计算集群 (SM/TPC) 可以分配不同比例的共享缓存容量。支持硬件可选的一致性协议，在需要时保持多集群间缓存数据一致性。

MMU 与 PTE 结构设计

PTE 用于 "将虚拟地址映射到 tile 的物理地址"，同时包含内嵌索引值。这使得硬件可以即时判断缓存命中状态，实现 "分区单元间高效请求交错"，减少空闲周期和内存争用。

TLB Miss 减少机制

嵌入式 TLB 架构

TLB 不集中在内存控制器，而是分布在每个 graphics multiprocessor 内部。这允许每个 SM 独立处理地址翻译，减少跨芯片通信延迟。SM 内的线程可以直接访问本地 TLB，命中时零延迟获取物理地址。

早期缓存状态判断

通过 PTE 内嵌的缓存行索引，硬件在地址翻译完成的同时即可判断目标数据是否在缓存中。这实现了 "高效请求交错"，允许内存控制器提前调度请求，减少等待时间。

缓存命中率提升

通过动态优先级调整和 PTE 覆盖，系统可以根据工作负载特点优化缓存行为。图形工作负载获得高空间局域性缓存，计算工作负载获得更大的有效容量，ML 工作负载获得优化的权重缓存策略。综合优化目标是 "命中率提升" 和 "延迟降低"。

摘要

本文提出 LATPC，一种利用空间和时间局域性预取 TLB 条目的新机制，并结合 MSHR (Miss Status Holding Register) 压缩技术来加速 GPU 地址翻译。LATPC 通过检测 GPU 工作负载中页面访问的局域性模式，预测性地预取可能需要的页表条目到 TLB 中。同时，MSHR 压缩技术通过合并具有相同页面翻译需求的待处理请求，减少待处理翻译条目数量，提高 TLB 预取的覆盖率和准确率。

来源: ACM Digital Library | ResearchGate

技术背景

GPU 的大规模并行架构使得地址翻译成为性能瓶颈。随着 GPU 内存容量增长到数十 GB，页表本身也变得更加庞大和多级化。传统的 TLB 预取器主要面向 CPU 设计，没有充分考虑 GPU 工作负载的独特特征：

  • • GPU 线程束 (warp) 内的线程往往访问相邻或相关的内存页面 (空间局域性)
  • • 同一 kernel 的不同执行阶段可能重复访问相同的页表区域 (时间局域性)
  • • 不规则应用 (图遍历、稀疏矩阵) 的页面访问模式难以用简单的 stride 预测
  • • MSHR 中大量待处理翻译请求造成资源浪费

LATPC 核心技术

一、局域感知 TLB 预取机制 (Locality-Aware TLB Prefetching)

1. 空间局域性检测

LATPC 检测 GPU 线程束内线程访问的页面之间的空间关系。当同一 warp 的多个线程访问相邻页面时，预取器预测这些页面所属的页表目录节点 (page table directory nodes) 可能被后续访问，提前将其预取到 TLB 中。预取器利用页面表本身的多级结构，在访问叶子节点 (PTE) 之前预取中间级目录。

2. 时间局域性跟踪

LATPC 跟踪页面访问的历史模式，识别重复访问的页面区域。通过记录近期访问的页表区域，当检测到类似的访问模式重现时，提前触发预取。这对于迭代算法 (如图算法的 BFS/SSSP、稀疏矩阵的 SpMV) 特别有效，因为这些算法在多轮迭代中访问相同的页面结构。

3. 页面表局域性利用

页表本身在内存中是连续分配的。当访问某个页面时，其相邻页面的页表条目很可能在相同的页表页面中。LATPC 利用这一特性，在获取一个 PTE 的同时预取整个页表页面中可能需要的相邻条目。这显著提高了预取覆盖率，减少了后续 TLB Miss。

二、MSHR 压缩技术 (MSHR Compression)

MSHR 的问题

MSHR (Miss Status Holding Register) 用于跟踪待处理的 TLB Miss 请求。在 GPU 中，数千个并发线程可能产生大量 TLB Miss，每个 Miss 需要一个 MSHR 条目。MSHR 容量有限，当条目耗尽时，新的翻译请求必须等待，造成性能瓶颈。

压缩机制

LATPC 的 MSHR 压缩技术通过合并具有相同页面翻译需求的请求来减少 MSHR 条目占用。具体来说：当多个线程需要翻译同一页面 (或同一页表页面内的不同偏移) 时，LATPC 将这些请求合并为一个 MSHR 条目。翻译完成后，结果分发给所有等待的线程。

实现方式

MSHR 压缩使用哈希表或 Bloom Filter 快速检测待处理请求中是否存在页面重叠。当新的翻译请求到达时，首先检查是否有已有的 MSHR 条目覆盖该页面。如果有，则将请求附加到现有条目 (而不是创建新条目)。如果没有，则创建新的 MSHR 条目。这显著减少了 MSHR 压力，提高了并发翻译能力。

LATPC 工作流程

1

监控 GPU 内存访问: 跟踪每个 warp 的页面访问模式，记录空间和时间局域性。

2

检测局域性模式: 分析相邻页面访问关系，识别重复访问区域。

3

触发 TLB 预取: 基于检测到的局域性模式，提前将可能的页表条目预取到 TLB。

4

MSHR 压缩: 合并待处理的翻译请求，减少 MSHR 条目占用。

5

分发结果: 翻译完成后，将结果分发给所有等待的线程。

评估结果

LATPC 在多种 GPU 工作负载上进行了评估，包括规则和不规则内存访问模式。

TLB Miss 减少

通过局域感知预取，显著减少 L1/L2 TLB Miss 率。对于具有强空间局域性的工作负载 (如矩阵操作、卷积)，预取准确率更高。

MSHR 压力降低

MSHR 压缩减少了待处理翻译条目数量，提高了并发翻译能力。对于多线程密集型工作负载效果显著。

预取覆盖率和准确率

LATPC 利用页面表局域性提高了预取覆盖率，同时通过局域性感知过滤减少了无用预取，提高了准确率。

性能提升

综合 TLB 预取和 MSHR 压缩，LATPC 在多种工作负载上实现了显著的性能提升，特别是不规则内存访问模式。
