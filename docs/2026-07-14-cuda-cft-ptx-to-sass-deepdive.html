PTX → SASS 逐指令映射解析

fabric_try_get_probe: 从 PTX 虚拟指令到 Blackwell SASS 硬件指令的完整翻译链路

PTX 9.3 → SASS sm_100a • cuobjdump 反汇编分析

0. PTX 源码回顾

.version 9.3
.target sm_100a
.address_size 64

.visible .entry fabric_try_get_probe(
    .param .u32 src_le_id,
    .param .u64 src_offset,
    .param .u32 byte_count
) {
    .reg .b32 %r<8>;
    .reg .b64 %rd<8>;
    .shared .align 16 .b8 smem_data[16];
    .shared .align 8 .b64 smem_bar;

    ld.param.u32 %r1, [src_le_id];       // 读取 src_le_id
    ld.param.u64 %rd1, [src_offset];     // 读取 src_offset
    ld.param.u32 %r2, [byte_count];      // 读取 byte_count

    mov.u64 %rd2, smem_bar;
    cvta.to.shared.u64 %rd3, %rd2;
    cvt.u32.u64 %r3, %rd3;
    mbarrier.init.layout::v1.shared::cta.b64 [%r3], 1;   // 初始化 mbarrier

    mov.u64 %rd4, smem_data;
    cvta.to.shared.u64 %rd5, %rd4;
    fabric.try_get.async.shared::cta
        .mbarrier::complete_tx::bytes.mbarrier::report::fabric
        .relaxed.sys.b128
        [%rd5], [%r1, %rd1], %r2, [%rd3];  // 发起 fabric get

    fabric.submit;                          // 提交 fabric 操作
    ret;
}

PTX → SASS 翻译的关键差异

PTX 是虚拟 ISA，使用符号化的 shared memory 地址和逻辑寄存器。SASS 是 Blackwell 硬件实际执行的指令集，使用统一寄存器（Uniform Register, UR）、shared memory 符号地址编码、以及硬件特有的同步原语（SYNCS.EXCH）。编译器需要完成地址计算、寄存器分配、调度控制字插入等工作。

1. 阶段一：运行时初始化与参数加载

Phase 1: 常量加载与 CTA 上下文建立 (0x0000 - 0x0040)
0x0000
LDC R1, c[0x0][0x37c]
初始化 从常量内存 c[0x0][0x37c] 加载运行时状态到通用寄存器 R1。这通常是 kernel 启动参数区的栈指针或 CGA/CTA 上下文指针。LDC = Load Constant，访问只读常量缓存。
对应 PTX: 隐式 — PTX 不显式加载运行时上下文，由编译器插入
0x0010
S2UR UR11, SR_CgaCtaId
CTA ID 从特殊寄存器 SR_CgaCtaId 读取当前 CTA 的标识符，存入统一寄存器 UR11。S2UR = Special to Uniform Register。这个 CTA ID 后续用于计算 shared memory 的物理地址基准 — 每个 CTA 的 shared memory 在不同物理位置，需要通过 CTA ID 计算偏移。
对应 PTX: 隐式 — PTX 的 .shared 变量地址由硬件自动绑定到当前 CTA，无需显式计算
0x0020
UMOV UR10, 0x400
地址计算 将立即数 0x400 加载到 UR10。这是 shared memory 符号地址的基准偏移常量。在 Blackwell 的 shared memory 编码方案中，0x400 是 CTA shared memory 区域内的一个基础偏移，用于后续计算 smem_data 和 smem_bar 的实际地址。
对应 PTX: 隐式 — PTX 中 mov.u64 %rd2, smem_bar 直接取符号地址，SASS 需要手动计算偏移
0x0030
LDCU UR14, c[0x0][0x380]
参数加载 从参数区加载 src_le_id（u32）到统一寄存器 UR14。LDCU = Load Constant to Uniform。注意这里使用了 Uniform 寄存器而非通用寄存器 — Blackwell 的 Fabric 操作数必须在 Uniform 寄存器中。
对应 PTX: ld.param.u32 %r1, [src_le_id]
0x0040
PLOP3.LUT P0, PT, PT, PT, PT, 0x80, 0x8
Retry 控制 通过 LUT（查找表）操作初始化谓词寄存器 P0。LUT 值 0x80 配合操作码 0x8 将 P0 设置为特定初始值（True）。P0 是 fabric.try_get 的重试循环控制谓词 — 如果 fabric 操作发出失败（busy），需要重新尝试。
对应 PTX: 无 — PTX 不暴露 retry 语义，这是 SASS 级别的硬件重试逻辑

2. 阶段二：Shared Memory 地址编码

Phase 2: 计算 smem_bar 和 smem_data 的硬件编码地址 (0x0050 - 0x0130)

Blackwell Shared Memory 地址编码

Blackwell 的 shared memory 指令不使用直接的虚拟地址，而是使用一种编码后的地址格式，包含 CTA ID、bank 选择、属性位等字段。编译器需要将 PTX 的符号地址转换为这种编码格式。以下指令序列就是在构建这种编码地址。

0x0050
UIADD3 UR4, UPT, UPT, UR10, 0x10, URZ
地址计算 UR4 = UR10 + 0x10 = 0x400 + 0x10 = 0x410。这是 smem_bar 相对于 CTA shared memory 基址的偏移。0x10 = 16 字节偏移，对应 smem_data[16] 之后的 smem_bar（smem_data 占 16 字节）。UIADD3 = Uniform Integer Add 3-operand。
对应 PTX: mov.u64 %rd2, smem_bar 的 SASS 地址计算部分
0x0060
UMOV UR6, 0x1
mbarrier 初始化 将 mbarrier 的 participant count = 1 加载到 UR6。这个 kernel 只有 1 个线程参与 mbarrier（单线程 probe kernel）。
对应 PTX: mbarrier.init.layout::v1.shared::cta.b64 [%r3], 1 中的立即数 1
0x0070
LDCU UR12, c[0x0][0x390]
参数加载 加载 byte_count（u32）到 UR12。后续会被编码为 UBLKCP（Fabric 读取指令）的 transaction size 操作数。
对应 PTX: ld.param.u32 %r2, [byte_count]
0x0080
LDCU.64 UR8, c[0x0][0x388]
参数加载 加载 64-bit src_offset 到 UR8/UR9（64 位占用两个 32-bit uniform 寄存器槽位）。
对应 PTX: ld.param.u64 %rd1, [src_offset]
0x0090
UIADD3 UR6, UPT, UPT, -UR6, 0x100000, URZ
地址编码 UR6 = -1 + 0x100000 = 0x0FFFFF。这是 shared memory 地址编码中的高位字段/掩码中间值。Blackwell 的 shared memory 地址需要将 CTA 内偏移编码为特定位域格式。
对应 PTX: 无直接对应 — 纯 SASS 级地址编码
0x00a0
USHF.L.U32 UR7, UR6, 0xb, URZ
地址编码 UR7 = UR6 << 11 = 0x0FFFFF << 11。左移 11 位将地址值对齐到 shared memory 编码中的高位字段位置。USHF = Uniform Shift。
纯 SASS 地址编码
0x00b0
USHF.L.U32 UR6, UR6, 0x1, URZ
地址编码 UR6 = UR6 << 1。左移 1 位，为 mbarrier 地址字段做准备 — mbarrier 需要 8 字节对齐，地址最低 3 位为 0，左移 1 位是编码过程的一部分。
纯 SASS 地址编码
0x00c0
ULO P3.LUT UR6, UR6, 0x1, URZ, 0xfe, !UPT
地址编码 通过 LUT 操作设置 shared memory 地址的低位属性位。最终生成的 UR6 包含 smem_bar 的完整编码地址，包括对齐位、属性位等，可以直接传给 SYNCS.EXCH 指令使用。
对应 PTX: cvta.to.shared.u64 %rd3, %rd2 + cvt.u32.u64 %r3, %rd3 的 SASS 等价实现
0x00d0
UMOV UR5, URZ
地址编码 清零 UR5。作为 64 位地址构造的高 32 位（= 0），与低 32 位合并形成完整的 shared memory 地址。
纯 SASS 辅助操作
0x00e0
ULEA UR4, UR11, UR4, 0x18
地址计算 UR4 = UR11 + UR4 + 0x18 的线性组合（LEA = Load Effective Address）。将 CTA ID (UR11) 与 shared 偏移 (UR4=0x410) 合并，生成 smem_bar 的物理地址。0x18 是额外的偏移修正。
对应 PTX: mov.u64 %rd2, smem_bar 的最终地址解析
0x00f0
UMOV UR13, UR4
地址保存 将 smem_bar 的最终地址保存到 UR13。UR13 将在后续 SYNCS.EXCH（mbarrier 初始化）中作为目标地址使用。
对应 PTX: %r3（mbarrier 地址寄存器的保存）
0x0100
ULEA UR4, UR11, UR10, 0x18
地址计算 UR4 = UR11 + UR10 + 0x18。这次使用 UR10（= 0x400，即 smem_data 的偏移），生成 smem_data 的物理地址。
对应 PTX: mov.u64 %rd4, smem_data + cvta.to.shared.u64 %rd5, %rd4
0x0110
USHF.R.U32.HI UR10, URZ, 0x4, UR12
size 编码 将 byte_count (UR12) 编码为 UBLKCP 指令使用的 transaction size 格式。USHF.R = 右移，HI 表示取高半部分。这是将字节数转换为硬件期望的 block copy size 编码格式。
对应 PTX: %r2（byte_count 直接作为 size 操作数）— PTX 不要求编码，SASS 需要
0x0120
SYNCS.EXCH.64 URZ, [UR13], UR6
mbarrier 初始化 这是 mbarrier.init.layout::v1 的 SASS 实现。SYNCS.EXCH = Synchronize Exchange，在 shared memory 地址 [UR13]（即 smem_bar）上执行原子交换初始化操作。UR6 包含初始化值（编码后的 participant count + layout::v1 标志）。这是 Blackwell 特有的 mbarrier 初始化原语 — PTX 的 mbarrier.init 在 SASS 层被翻译为 SYNCS.EXCH 指令。
对应 PTX: mbarrier.init.layout::v1.shared::cta.b64 [%r3], 1
0x0130
UMOV UR12, UR4
地址保存 将 smem_data 的地址保存到 UR12。UR12 将作为 UBLKCP（fabric try_get）的本地目标地址。
对应 PTX: %rd5（fabric.try_get 的 dst 操作数）

3. 阶段三：远端 CFT 地址构造

Phase 3: 构造 [src_le_id, src_offset] 远端 Handle 地址 (0x0140 - 0x0160)
0x0140
ULEA UR7, UR14, 0x200000, 0xa
LE ID 编码 将 src_le_id (UR14) 编码到 CFT 源地址的高位字段中。0x200000 和 0xa 是编码常量 — Blackwell 的 Fabric 指令期望 LE ID 以特定位域格式出现在操作数中，而非裸的 32-bit 值。ULEA 的三操作数模式将 LE ID 与编码常量合并。
对应 PTX: [%r1, %rd1] 中的 %r1（src_le_id）— PTX 直接使用裸值，SASS 需要位域编码
0x0150
UMOV UR6, URZ
地址构造 清零 UR6，准备作为 64-bit 源地址的高 32 位。与 UR7（低 32 位，已编码 LE ID）配合，形成完整的 [leId, offset] 编码地址。
对应 PTX: 地址组装的中间步骤
0x0160
UIADD3.64 UR6, UPT, UPT, UR6, UR8, URZ
地址合并 64-bit 加法：UR6/UR7 = UR6 + UR8。将 src_offset (UR8/UR9) 合并到已编码 LE ID 的地址中，形成最终的完整 CFT 源操作数 [src_le_id, src_offset]。UR6/UR7 现在包含了完整的远端 Handle 编码地址。
对应 PTX: [%r1, %rd1] 的完整 SASS 编码实现

4. 阶段四：Fabric try_get 执行与重试

Phase 4: Fabric 操作发出、状态检查与重试循环 (0x0170 - 0x01b0)

核心区域：PTX → SASS 的关键翻译

PTX 的 fabric.try_get.async 是一条指令，但在 SASS 层被翻译为 ELECT + UBLKCP + 状态检查 + 条件分支 的重试序列。这揭示了硬件的一个重要事实：Fabric 操作可能因为资源繁忙而失败，需要软件级重试。

0x0170
@P0 ELECT P1, URZ, PT
线程选举 条件执行（仅当 P0=True 时）。ELECT 是 Blackwell 特有的warp 线程选举指令 — 从当前 warp 中选出一个 lane 来发起 Fabric 事务。被选中的 lane 将 P1 设为 True，其他 lane 的 P1 为 False。

为什么需要选举？Fabric 操作通过 NVLink 发送请求，每个 warp 只需要一个 lane 发起事务即可（避免多个 lane 同时发送造成冲突）。这与 CUDA 的 SIMT 模型一致 — 单条指令代表整个 warp 执行。
对应 PTX: 无 — PTX 不暴露硬件级线程选举，这是 SASS 特有的重试机制
0x0180
UBLKCP.S.G.STRONG.SYS [UR12], [UR6], UR10
Fabric 读取核心指令 这是 fabric.try_get.async 的真正 SASS 实现。

字段含义
UBLKCPUnaligned Block Copy — 硬件块拷贝指令，支持 Fabric 远端访问
.SSource 端通过 Fabric proxy（远端读取）
.GGeneric addressing
.STRONG.SYS系统级强内存序（对应 PTX 的 .relaxed.sys）
[UR12]目标地址 = smem_data 的编码地址
[UR6]源地址 = [src_le_id, src_offset] 的编码 Handle
UR10transaction size（已编码的 byte_count）

UBLKCP 执行从远端 CFT Handle 到本地 SMEM 的数据传输。如果 NVLink 资源可用，操作成功发出；如果资源繁忙，操作返回失败状态，需要重试。
对应 PTX: fabric.try_get.async.shared::cta.mbarrier::complete_tx::bytes.mbarrier::report::fabric.relaxed.sys.b128 [%rd5], [%r1, %rd1], %r2, [%rd3]
0x0190
@P1 PLOP3.LUT P0, PT, P1, PT, PT, 0x8, 0x80
状态检查 仅由被 ELECT 选中的 lane (P1=True) 执行。根据 UBLKCP 返回的状态更新 P0：
• 如果 UBLKCP 成功发出 → P0 = False → 退出重试循环
• 如果 UBLKCP 失败（busy）→ P0 = True → 继续重试

LUT 值 0x8 配合操作码 0x80 实现了基于 P1 的条件更新逻辑。
对应 PTX: 无 — PTX 不暴露重试语义
0x01a0
PLOP3.LUT P1, PT, PT, PT, PT, 0x8, 0x80
谓词重置 重置 P1 为初始状态，为下一次重试循环的 ELECT 指令做准备。确保每次循环迭代都从干净的谓词状态开始。
纯 SASS 重试控制逻辑
0x01b0
@P0 BRA.U.ANY 0x170
条件回跳 如果 P0=True（UBLKCP 发出失败），则跳转回 0x0170 的 ELECT 指令，重新执行线程选举和 UBLKCP 发出。这形成了一个软件重试循环：

0x0170: ELECT → 0x0180: UBLKCP → 0x0190: 检查状态 → 0x01b0: 失败则回跳

BRA.U.ANY 的 ANY 表示只要 warp 中有任意 lane 的 P0 为 True 就跳转。
对应 PTX: 无 — PTX 的 fabric.try_get 是单次发出，重试是 SASS 级别的实现细节
PTX 视角（1 条指令）: ┌─────────────────────────────────────────────────────┐ │ fabric.try_get.async ... [%rd5], [%r1,%rd1], %r2 │ └─────────────────────────────────────────────────────┘ SASS 视角（重试循环）: ┌─────────────────────────────────────────────────────┐ │ 0x0170: ELECT P1 ← 选一个 lane 发起 │ │ 0x0180: UBLKCP [dst],[src],size ← 实际发出 │ │ 0x0190: 检查状态 → P0 ← 成功? │ │ 0x01a0: 重置 P1 │ │ 0x01b0: @P0 BRA 0x0170 ← 失败则重试 │ └─────────────────────────────────────────────────────┘ 关键洞察: PTX 的 "try" 语义在 SASS 层体现为硬件级重试循环

5. 阶段五：fabric.submit 与收尾

Phase 5: Fabric 操作提交与 kernel 退出 (0x01c0 - 0x01f0)
0x01c0
UTMACMDFLUSH
fabric.submit 这是 fabric.submit 的 SASS 实现。UTMACMDFLUSH = TMA Command Flush — 刷新 TMA/Fabric 命令队列，将所有 pending 的 Fabric 操作提交给硬件。

命名中的 "TMA" 表明 Fabric submit 和 TMA (Tensor Memory Accelerator) 共享同一套命令刷新机制。在 Blackwell 硬件中，Fabric 操作和 TMA 操作通过同一个命令管道提交。
对应 PTX: fabric.submit
0x01d0
DEPBAR.LE SB0, 0x0
依赖屏障 DEPBAR = Dependency Barrier。等待 Scoreboard SB0 上的所有操作完成到 LE（Less than or Equal）级别。这确保 fabric.submit 之后的指令不会在 Fabric 操作实际完成之前执行后续依赖操作。

Scoreboard 是 Blackwell 的指令级依赖追踪机制，DEPBAR 提供了软件可控的同步点。
对应 PTX: 无直接对应 — 这是 SASS 级的指令排序控制
0x01e0
CCTL.IVALL
缓存控制 CCTL = Cache Control。IVALL = Invalidate All。执行 Fabric 操作后的缓存失效操作，确保后续读取不会看到陈旧的缓存数据。这是 Fabric 操作完成后的必要副作用 — 远端写入的数据可能需要使本地缓存失效。
对应 PTX: 隐式 — PTX 的内存序语义在 SASS 层需要显式缓存控制
0x01f0
EXIT
kernel 退出 线程退出。对应 PTX 的 ret。
对应 PTX: ret
0x0200+
BRA 0x200 / NOP × 14
填充 自旋分支 + NOP 填充。这些是编译器生成的对齐填充指令，确保后续 kernel 代码的指令对齐。无执行语义。
无对应

6. 完整映射总结

PTX 指令SASS 指令序列指令数说明

ld.param.u64 %rd1, [src_offset] LDCU.64 UR8, c[0x0][0x388] 1 64-bit 参数加载
ld.param.u32 %r2, [byte_count] LDCU UR12, c[0x0][0x390] 1 参数加载 + 后续编码

fabric.submit UTMACMDFLUSH + DEPBAR + CCTL 3 提交 + 排序 + 缓存失效

关键架构洞察

  1. PTX 1 条 → SASS 多条：PTX 的 fabric.try_get 在 SASS 层被展开为包含线程选举、实际传输、状态检查、条件重试的完整序列。这揭示了 "try" 语义的硬件实现本质。
  2. Uniform Register 主导：Blackwell 的 Fabric 操作大量使用 Uniform Register（UR）而非通用寄存器（R）。UR 的特点是 warp 内所有 lane 共享相同值，适合存储地址、Handle 等共享操作数。
  3. SYNCS.EXCH = mbarrier.init：PTX 的高级同步原语在 SASS 层被翻译为 Blackwell 特有的 SYNCS（Synchronize）指令族。
  4. UTMACMDFLUSH = fabric.submit：Fabric 提交与 TMA 共享命令管道，统一通过 command flush 机制提交。
  5. 软件重试循环：Fabric 操作的 "try" 语义意味着硬件可能拒绝请求（busy），SASS 通过 ELECT + 条件分支实现 warp 级重试。

调度控制字说明

每条 SASS 指令后面的 /* 0x... */ 注释是调度控制字（Scheduler Control Word），描述该指令的 stall cycle、yield 行为、寄存器 reuse 提示等调度信息。这些不是独立的执行指令，而是 cuobjdump 输出的指令编码的一部分。例如 0x000fe20000000000 表示无 stall（完全流水线化），而包含非零 stall 值的控制字表示需要等待特定周期数。
