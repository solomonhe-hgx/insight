Issue #1187 — Turing (256 MiB BAR1) VA-space 耗尽导致全系统硬锁

2026-06-09 新 Issue

新 Bug: Turing GPU (256 MiB BAR1) 在 WebGL 工作负载耗尽虚拟地址空间后触发全系统硬锁。错误信息包括 dmaAllocMapping_GM107: can't alloc VA space 和 krcWatchdog: GPU is probably locked!。该问题涉及 NVLink/P2P 虚拟地址空间管理。

New Bug NVLink / VA-space / MMU
来源: Issue #1187

技术详情

该 Issue 报告了 Turing 架构 GPU 在 BAR1 仅 256 MiB 的配置下，WebGL 工作负载逐步耗尽虚拟地址空间后触发系统级硬锁。关键错误序列：(1) dmaAllocMapping_GM107: can't alloc VA space — 虚拟地址分配失败；(2) krcWatchdog: GPU is probably locked! — 看门狗检测到 GPU 锁定。256 MiB BAR1 是 Turing GPU 的最小配置，虚拟地址空间极为有限。当 WebGL 应用持续分配显存映射时，VA 空间碎片化最终导致分配失败，而驱动未能优雅处理此错误，触发了系统级硬锁。该问题与上周 #1148 (RTX 3090 Ti S3 挂起 virtualBar2 NULL) 类似，均指向 NVIDIA 开源驱动在虚拟地址空间管理上的深层缺陷。

Issue #1187 — Turing VA-space 耗尽: dmaAllocMapping 分配失败 → 全系统硬锁 (2026-06-09)

新 Bug / MMU

新 Bug: Turing GPU (256 MiB BAR1) 在 WebGL 工作负载下虚拟地址空间耗尽，dmaAllocMapping_GM107 无法分配 VA space，触发 krcWatchdog 全系统硬锁。这是本周唯一直接涉及 MMU/VA 管理的新 Issue。

New Bug MMU / VA-space / BAR1
来源: Issue #1187

技术详情

该 Issue 涉及 GPU MMU 虚拟地址空间管理缺陷。Turing GPU 配置 256 MiB BAR1 时，虚拟地址空间极为有限。WebGL 工作负载持续分配显存映射，VA 空间碎片化最终导致 dmaAllocMapping_GM107 分配失败。驱动未能优雅处理此错误路径，触发看门狗超时和系统级硬锁。涉及模块路径：kernel-open/nvidia/ 中的 DMA 映射和 VA 分配代码。与 #1148 (virtualBar2.pCpuMapping NULL) 属同一类 VA 空间管理问题。

Issue #1177 — DIFR Prefetch 导致 nvidia-modeset soft lockup (2026-06-03)

新 Bug / Modeset

新 Bug: Pop!_OS 上使用 nvidia-open 580.159.03 的 RTX 5070 用户报告 DIFR Prefetch 机制触发 nvidia-modeset soft lockup，导致系统冻结。DIFR (Display Internal Frame Rate) 预取机制与 modeset 驱动的交互存在缺陷。

New Bug DIFR / nvidia-modeset
来源: Issue #1177

技术详情

DIFR (Display Internal Frame Rate) Prefetch 是 NVIDIA 显示驱动的帧预取机制。该 Issue 报告此机制在 Blackwell RTX 5070 上触发 nvidia-modeset 模块的 soft lockup，导致系统完全冻结。环境：Pop!_OS, nvidia-open 580.159.03, RTX 5070。Soft lockup 表明内核线程在 CPU 上长时间无法调度，通常由自旋锁死循环或中断处理阻塞引起。涉及模块路径：kernel-open/nvidia-modeset/。

Issue #1179 — RTX 5070 (GB205) Xid 32 → Xid 56 致命 GPU 冻结 (2026-06-04)

新 Bug / Blackwell

新 Bug: RTX 5070 (GB205) 在混合显示/计算负载下触发 Xid 32 → Xid 56 错误序列，随后系统完全挂起需断电重启。驱动版本 595.71.05。Xid 32 为 GPU 语义错误（通常涉及 MMU 页表），Xid 56 为显示引擎错误。

New Bug Xid 32/56 / GB205
来源: Issue #1179

技术详情

RTX 5070 (GB205) 在混合显示和计算工作负载下触发致命 GPU 错误序列。Xid 32 (GPU semantic error) 通常涉及 MMU 页表错误或命令流异常，随后 Xid 56 (display engine error) 表明显示管线也受到影响。错误序列的级联特性表明根因可能在 GPU 内存管理或命令缓冲区处理路径。驱动 595.71.05。需要断电重启，表明 GPU 进入了不可恢复的错误状态。

Arch Linux 默认切换至 nvidia-open 包 — 590 分支起生效

生态里程碑

生态动态: Arch Linux 主要 NVIDIA 驱动包已从 590 版本起默认使用开源内核模块 (nvidia-open)，不再使用闭源包。同时 590 驱动停止支持 Pascal 及更早架构 (GTX 10xx 及之前)。这一变化显著扩大了 NVIDIA 开源驱动的用户基数。

Ecosystem Arch Linux
来源: Phoronix | Arch Linux Forums

详情

Arch Linux 项目已将其主要 NVIDIA 驱动包从闭源切换到开源内核模块。从驱动版本 590 起，用户默认安装的是 nvidia-open 包。这意味着 Arch Linux 的所有 NVIDIA 用户（占 Linux 桌面发行版的显著份额）现在都在使用开源内核模块。同时，590 版本停止支持 Pascal (GTX 10xx) 及更早架构，Turing (GTX 16xx/20xx) 和更早架构的用户需要特别关注兼容性。这一变化使开源驱动的问题（如 #1108 UVM 内核恐慌）影响面急剧扩大。

本周核心洞察

1. Blackwell 全系列驱动危机 — 本周最重要的系统性趋势

本周 8 个新 Issue 中有 4 个直接涉及 Blackwell 架构 GPU：#1185 (RTX 5050 Laptop GB107 Xid 109)、#1180 (RTX 5070 Laptop 610 黑屏)、#1179 (RTX 5070 GB205 Xid 32→56)、#1177 (RTX 5070 DIFR Prefetch 冻结)。加上持续的 #1151 (RTX 5080 GB203 Xid 79) 和论坛 RTX 5090 Xid 79 报告，Blackwell 全系列在 Linux 开源驱动上的支持远未成熟。610 分支在 Blackwell 移动 GPU 上的显示输出回归尤其令人担忧，因为这意味着新分支可能引入了 Blackwell 显示管线的重大变更。

2. 610 新分支发布即遇回归 — 用户迁移风险升高

NVIDIA 全新 610 主要分支 (610.43.02) 发布不到两周即暴露回归问题。#1180 报告 RTX 5070 Laptop 在 610 上内屏黑屏，595.71.05 正常。论坛 "610 release feedback & discussion" 帖已累积 101 回复，暗示更多兼容性问题。在 Arch Linux 默认采用开源模块的背景下，610 分支的稳定性将直接影响大量用户。建议用户在 NVIDIA 发布更多修复前暂留 595 分支。

3. VA-space 管理成为系统性问题 — #1187 + #1148 需联合修复

本周 #1187 (Turing 256 MiB BAR1 VA-space 耗尽 → 硬锁) 与上周 #1148 (RTX 3090 Ti S3 virtualBar2.pCpuMapping NULL) 共同揭示了 NVIDIA 开源驱动在虚拟地址空间管理上的系统性缺陷。两个 Issue 分别涉及 VA 分配失败处理和 VA 映射生命周期管理，根因可能共享。建议 NVIDIA 对 VA-space 管理代码进行系统性审查和修复。

4. UVM 稳定性危机持续深化 — #1108 跨越 7 周 3 个版本未修复

关键 Issue #1108 (nvidia_uvm 内核恐慌) 已开放超过 7 周，跨越 595.71.05、595.80 和 610.43.02 三个版本均未修复。这是目前持续时间最长的开源模块特定回归 bug。在 Arch Linux 默认采用开源模块、用户基数急剧扩大的背景下，NVIDIA 的修复节奏严重滞后。

5. 社区贡献活跃度提升 — 本周 4 个 PR 覆盖显示/构建/电源/流程

本周仓库收到 4 个社区 PR，覆盖多个领域：#1186 (HDMI 2.1 FRL rate 强制设置)、#1183 (TARGET_ARCH 交叉编译修复)、#1181 (RTD3 dGPU ACPI 唤醒抑制)、#1182 (Bug 模板闭源对比可选化)。其中 #1181 由 ElXreno 提交的电源管理优化和 #1183 的跨平台构建修复尤其有价值。社区贡献的活跃度表明开源驱动的生态正在逐步成熟，但核心稳定性问题仍需 NVIDIA 官方解决。
