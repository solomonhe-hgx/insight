Multi-Channel Workload Balance Algorithm

硬件多channel workload均匀分配算法设计

1. Problem Statement

输入

ParameterValueDescription
PA width40 bitPhysical Address位宽
flow_id width12 bit每段切分为4K，即一个flow_id
N9 / 18 / 36 / 72 / 144 / 288 (可配)物理channel数量
R = 4096 % N1(N=9) / 10(N=18) / 28(N=36) / 64(N=72,144,288)每block余数

核心矛盾

每个4K block有4096个flow_id需要映射到N个channel。简单取模 flow_id % N 时：

  • 4096 = 28 × 144 + 64（当N=144）
  • 前64个channel（0~63）各得29个flow_id，后80个channel（64~143）各得28个
  • 这个偏差在每个4K block中完全相同，累积后channel 0比channel 64多144个workload（144 blocks后）
Naive Modulo的问题：3.52%的固定不均衡，且永远不会收敛。144个block后，channel 0累计4176次，channel 64只有4032次，差值144。

2. Algorithm Design

Approach A: Rotated Modulo 最简实现

核心思路：给不同block的flow_id加上不同的偏移量，让"多出来的64个"轮流落在不同channel上。

channel = (local_id + block_id × STRIDE) % N

其中：

  • local_id = flow_id[11:0]（block内偏移，0~4095，来自flow_id低12位）
  • block_id（block编号，驱动rotation），两种取法：
    • 简单：block_id = dst_route_na[27:12]（16 bit，直接截取NA高位）
    • 复杂：block_id = hash(dst_route_na[27:12])（16 bit，加hash打散地址相关性）
  • STRIDE 需满足 R % gcd(STRIDE, N) == 0。六种N均推荐STRIDE=32（<<5）
STRIDE选择条件：R % gcd(STRIDE, N) == 0
rotation offset 以 gcd(STRIDE, N) 为步长移动一个宽度为R的"extra窗口"。 只有当R能被步长整除时，窗口才能像砖块一样无缝铺满channel空间。

STRIDE=32为什么对所有N都最优？
以N=144为例：gcd(32, 144) = 16，R=64，64%16=0。9个不同offset {0, 32, 64, 96, 128, 16, 48, 80, 112}， 每个offset把64宽的extra窗口移动16格，恰好整除窗口宽度，9个block即完美均衡。 STRIDE=32同时对六种N均满足 R%gcd==0 条件，且实现仅需 block_id << 5，适合可配设计共用一个常量。 注意 N/gcd 对六种N都恰好等于9，收敛速度统一。

硬件开销：1个加法器 + 1个取模器（divider）。极其简单。

Approach B: Bresenham Uniform Distribution 推荐方案

核心思路分两步：前 (4096−R) 个flow_id用简单取模（完美均衡），剩余R个用Bresenham直线算法均匀散布，再叠加block rotation实现跨block均衡。

算法公式

if local_id < base_limit:
    channel = local_id % N     // base路径：Q个/channel，完美均衡

else:
    extra_idx = local_id − base_limit   // 0 ~ R−1
    raw_ch = (extra_idx × N + N/2) / R   // Bresenham均匀散布
    block_rot = (block_id × STRIDE) % N   // stride rotation
    channel = (raw_ch + block_rot) % N

关键公式解读

raw_ch = (extra_idx × N + N/2) / R — 这是Bresenham直线算法的变体。 本质上是在R个点中均匀选取N个位置（R < N时反过来）。 + N/2 是四舍五入偏移，让extra不扎堆在channel 0附近。 以N=144, R=64为例：extra_idx每增1，raw_ch增 ≈144/64=2.25，实际间距在2和3之间交替（48次gap=2，15次gap=3）， 64个extra均匀散布在144个channel上。

block_rot = (block_id × STRIDE) % N — 每个block把extra的落点整体旋转一个偏移量。 STRIDE=32时，gcd(32,144)=16，rotation步长16格，9个block覆盖全部144个channel。

Bresenham间距分析（N=144, R=64）

  • 平均间距 = N/R = 144/64 = 2.25
  • 实际间距分布：gap=2 出现48次，gap=3 出现15次（48×2 + 15×3 = 141 ≈ 144）
  • Extra channel序列（block_id=0）：1, 3, 5, 7, 10, 12, 14, 16, 19, ... （每3个extra中2个间距2、1个间距3）

优化提示：
• block_id × STRIDE：STRIDE=32 → block_id << 5，零成本移位
• extra_idx × N：N=288→(i<<8)+(i<<5)；N=144→(i<<7)+(i<<4)；N=72→(i<<6)+(i<<3)；N=36→(i<<5)+(i<<2)；N=18→(i<<4)+i+i；N=9→(i<<3)+i
• / R：R=64 (N=72/144/288) → >> 6，零成本移位
• R非2的幂：N=36(R=28)、N=18(R=10)、N=9(R=1) 需LUT或小型除法器。但extra_idx位宽很小（1~5 bit），LUT面积可忽略。N=9时R=1，/R退化为除以1（直通）
为什么推荐Bresenham方案？
与Rotated Modulo跨block收敛速度相同（都是9个block），但单block内的extra分布更优： Rotated Modulo中，64个extra集中在连续的64个channel上（如0-63）； Bresenham将64个extra均匀分散到144个channel上（间距2-3），单block内负载更平滑。 如果系统对瞬时均衡要求不高，Rotated Modulo实现更简单。

               │            │       │       │

               │            │       │

                   channel[7:0]

Pipeline stages：

  1. Stage 1：提取local_id(flow_id[11:0]) / block_id(dst_route_na[27:12]) + 取模(local_id % N) + 比较(local_id < base_limit) — 并行
  2. Stage 2：Extra路径的乘加除（仅extra时激活）+ block rotation取模
  3. Stage 3：最终取模 + MUX选择

通用性：对于任意N值，只需重新计算 R = 4096 % N，并选择满足 R % gcd(STRIDE, N) == 0 的STRIDE。 最优选择是 gcd(STRIDE, N) 尽量大（收敛越快），但不能超过R。 例如N=128时R=0（完美整除），算法退化为简单取模。N=100时R=96，STRIDE可取96（gcd=4, 96%4=0）。

5. Key Takeaways

  1. 六种N配置统一STRIDE=32：N=9/18/36/72/144/288均满足 R%gcd(32,N)==0，9个block完美均衡，实现仅需 block_id << 5
  2. 单block内不均衡不可避免（4096不能整除N），最大差值为1，这是数学下界
  3. 跨block均衡是关键创新：通过rotation让每个block的"extra"落在不同channel上
  4. Rotated Modulo最简实现：硬件仅需1个加法器+1个取模器，9个block收敛
  5. Bresenham单block内更优：extra均匀分散，瞬时负载更平滑
  6. STRIDE条件：R % gcd(STRIDE, N) == 0，gcd越大收敛越快但不能超过R
  7. 除法优化：N=72/144/288的R=64可直接移位(>>6)；N=9/18/36的R非2的幂，用小型LUT实现
