NVLink 模块更新

Release

580.167.08: 修复 B200 NVLink 持久中断与系统变慢问题

关键修复: 修正 B200 平台上 NVLink 持久中断(persistent NVLINK interrupts)的寄存器设置，解决由此引发的系统性能下降(system slowness)问题。
详细技术方案:
  • 修复 NVLink 中断寄存器配置错误，该错误导致 B200 GPU 上中断无法正确清除，形成持久中断状态
  • 持久中断会占用 CPU 中断处理资源，导致系统整体性能下降
  • 此修复确保 NVLink 中断在触发后能被正确响应和清除
  • 影响范围: 所有搭载 B200 GPU 的 Data Center 系统 (GB200 NVL72 等)
其他 580.167.08 修复:
  • 修复 nvidia-imex-ctl JSON 状态关闭顺序和 IMEX 启动失败遗留僵尸进程
  • 修复心跳逻辑中的竞态条件，防止 double free/堆损坏
  • 修复 Xid 145、137、94 错误，这些错误影响内核启动性能约 15%
  • 修复 gRPC 流式服务器拆解时的崩溃问题
已知问题:
  • 与 EUD 版本 580.126.12 及更早版本不兼容，需升级
  • Hopper GPU (subrevision = 3) 需要 VBIOS 96.00.68.00.xx 或更高版本
  • 默认 TCC 模式不支持 L40/L20 等 GPU 的 IOMMU 隔离
  • RHEL 10 上 doca-ofed 缺少 ib_umad.ko，导致 Fabric Manager 无法工作
来源: GitHub Release 580.167.08 | NVIDIA Release Notes

NVLink-C2C 模块更新

New Feature

580.167.08 引入 CDMM — GB200 NVLink-C2C 内存管理新模式

核心特性: Coherent Driver-Based Memory Management (CDMM) — 驱动层一致性内存管理。在 GB200 平台上，由 NVIDIA 驱动而非操作系统来管理 GPU 内存，利用 NVLink-C2C 的一致性域实现 CPU-GPU 统一内存管理。
技术细节:
  • CDMM 通过 NVLink-C2C 互连实现 CPU (Grace) 与 GPU (Blackwell) 之间的一致性内存管理
  • 驱动接管内存管理后，可绕过 OS 页表的部分开销，直接利用 NVLink-C2C 的硬件一致性
  • 启用方式: 在 /etc/modprobe.d/nvidia-openrm.conf 中添加 options nvidia NVreg_CoherentGPUMemoryMode=driver
  • 注意: CDMM 模式下不支持 GPU Direct Storage
  • 新增 HAL 支持: GPU_OFFSET_MAX 可见性
架构意义:
  • GB200 Superchip 通过 NVLink-C2C 将 Grace CPU 和 Blackwell GPU 以 900 GB/s 带宽互连
  • CDMM 是 NVLink-C2C 一致性能力的软件层实现，使驱动可直接管理跨 CPU-GPU 的内存页
  • 对 AI 推理场景(如长上下文 LLM)尤为重要，可减少内存拷贝和页表遍历开销
来源: Release Notes 580.167.08 - CDMM | NVIDIA NVLink-C2C 官方页面

MMU/UVM 模块更新

PR

#1139: UVM 避免 tracker entry 过度预留

关键改进: 优化 UVM (Unified Virtual Memory) 的 tracker entry 分配策略，避免过度预留导致的内存浪费和潜在的性能退化。
技术细节:
  • UVM tracker 用于跟踪 GPU 页表操作和内存迁移的状态
  • 此前的实现会过度预留 tracker entry，在高并发内存操作场景下造成资源浪费
  • 优化后的分配策略按需分配，减少不必要的内存占用
  • 关键代码路径: kernel-open/nvidia-uvm/uvm_tracker.c
  • 对大规模内存操作(如 page migration、prefetch)的性能有正面影响
来源: GitHub PR #1139
PR

#1096: 修复 mmap_context 在 VMA 访问中的使用前初始化

关键修复: 修复 nvidia 驱动中 mmap_context 在 VMA (Virtual Memory Area) 访问路径上的使用前初始化(use-before-initialization) 缺陷，可能导致内存访问异常。
  • mmap_context 是 GPU 内存映射的核心数据结构
  • 在特定 VMA 访问路径上，mmap_context 可能在完全初始化前被引用
  • 该缺陷可能导致 GPU 内存映射失败或数据损坏
  • PR 状态: 已关闭 (可能已合入其他分支或以其他方式修复)
来源: GitHub PR #1096
PR

#1192: 约束 nvWriteGpEntry() 中 GPFIFO-space 等待范围

关键改进: 约束 nvWriteGpEntry() 函数中的 GPFIFO-space 等待边界，防止 GPU 命令队列操作中的潜在死锁或超时。
  • GPFIFO 是 GPU 命令队列的核心机制
  • nvWriteGpEntry() 负责向 GPU 命令队列写入执行条目
  • 此前在高负载场景下，GPFIFO-space 等待可能超出预期范围
  • 修复后添加了明确的等待边界，提升 GPU 调度的可预测性

Bug

#1187: BAR1 VA 空间耗尽导致系统硬锁定 (Turing, 256 MiB BAR1)

严重问题: 浏览器 WebGL 工作负载即可触发 BAR1 VA 空间耗尽，导致 RC watchdog 报告 "GPU is probably locked"，系统完全硬锁定。影响 Turing 架构 (256 MiB BAR1) 的 595.71.05 驱动。
  • BAR1 (Base Address Register 1) 是 CPU 映射 GPU 内存的窗口
  • Turing GPU 仅 256 MiB BAR1 空间，VA 空间有限
  • 浏览器 WebGL 上下文的纹理/缓冲区分配可能耗尽 BAR1 VA 空间
  • 耗尽后触发 Xid 错误链，最终导致系统硬锁定
  • 关联 Issue #1134: 类似问题在 RTX 3090 + kernel 7.0.3 上通过 __nv_drm_gem_nvkms_map 触发
来源: GitHub Issue #1187 | Related Issue #1134
Release

580.167.08 修复 Xid 145/137/94 — 内核启动性能提升 15%

性能修复: 修复 Xid 145、137、94 错误，这些错误影响 GPU 内核启动性能约 15%。Xid 错误是 NVIDIA 驱动报告的 GPU 硬件/软件异常。

  • 修复后内核启动性能提升约 15%，表明这些 Xid 的触发导致了重试或降级路径

本周核心洞察

1. CDMM 是 NVLink-C2C 软件生态的关键拼图

580.167.08 引入的 CDMM 让驱动直接管理 GPU 内存，绕过 OS 页表开销。这是 NVLink-C2C 硬件一致性能力在软件层的首次落地，对 Grace-Blackwell 及后续 Vera-Rubin 平台的 AI 推理性能至关重要。

2. BAR1 空间管理成为开源驱动痛点

本周 #1187 和 #1134 两个 Issue 表明，小 BAR1 GPU (256 MiB) 在日常浏览器工作负载下即可触发 VA 空间耗尽。PR #1109 的 Resizable BAR 修复是正确方向，但需要更根本的 VA 空间管理优化。

3. Blackwell 移动端电源管理仍需打磨

#1205 (CPU soft lockup)、#1136 (黑屏)、#1117 (s2idle 挂起) 三个 Issue 加上 PR #1199 和 #1181 两个修复 PR，说明 Blackwell 移动端的 suspend/resume 路径仍是当前开源驱动的主要不稳定区域。

4. Rubin 平台定义下一代互连规格

GTC 2026 发布的 Rubin 平台配备 NVLink 7 Switch、14.4 Tbit/s 聚合带宽、CPO 光互连。开源驱动需在未来 6-12 个月内为这些新硬件做好准备。
