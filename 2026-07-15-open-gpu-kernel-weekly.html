NVIDIA 官方

NVLink 6 与 Rubin 平台 — 3.6 TB/s 每 GPU

关键进展: NVIDIA 于 2026 年 1 月宣布 Rubin 平台六大芯片,其中 NVLink 6 路由芯片提供每 GPU 3.6 TB/s 带宽,整机柜 260 TB/s 聚合 NVLink 带宽。NVLink 6 芯片内置 in-network processing 用于集合运算,已进入量产阶段,合作伙伴将于 2026 下半年收到产品。
NVLink 代际对比:
  • NVLink 5 (Blackwell): 1,800 GB/s 双向/GPU, 130 TB/s per rack, NVSwitch 4.0 (144 ports @ 100 GB/s each)
  • NVLink 6 (Rubin): 3.6 TB/s 双向/GPU, 260 TB/s per rack, in-network collective processing
  • 带宽提升: NVLink 6 vs NVLink 5 = 2x
Rubin 平台芯片组:
  • Vera CPU — 88 个定制 Olympus 核心 (Armv9.2), NVLink-C2C 连接 Rubin GPU
  • Rubin GPU — 下一代 AI 加速器
  • NVLink 6 路由芯片 — 集成集合运算 offload
  • ConnectX-9 NIC / BlueField-4 DPU / Spectrum-6 交换机

固件要求

Blackwell NVSwitch 固件版本要求: 35.2014.4770+

固件规格: Blackwell 平台要求 NVSwitch 固件版本 35.2014.4770 或更高。NVSwitch 4.0 ASIC 配备 144 端口,每端口 100 GB/s,为 NVLink 5 提供全互连交换矩阵。跨节点内存共享需 NCCL 2.25.2+。
NVSwitch 4.0 技术参数:
  • 144 个端口,每端口 100 GB/s
  • 总交换容量: 14.4 TB/s per NVSwitch ASIC
  • GB200 NVL72 机架需要 9 个 non-scaleout NVSwitch + 3 个 Kubernetes 服务器
  • 支持 Zero Touch Provisioning (ZTP) 和 leader election
  • 固件升级支持在线进行

NVLink C2C 模块追踪

官方规格

NVLink-C2C 规格: 900 GB/s CPU-GPU 一致性互连

核心参数: NVLink-C2C 提供芯片间高带宽一致性数据传输。Blackwell + Grace 组合实现 900 GB/s CPU-GPU 带宽;Grace CPU Superchip 通过 NVLink-C2C 实现 1 TB/s 内存带宽。相比 PCIe Gen 6 PHY,能效提升 6 倍。
NVLink-C2C 平台支持:
  • Rubin + Vera: Vera CPU 88 个 Olympus 核心 (Armv9.2) 通过 "ultrafast NVLink-C2C" 连接 Rubin GPU
  • Blackwell + Grace: 900 GB/s 双向带宽 (GB200, GB300 等)
  • Grace CPU Superchip: 两颗 Grace CPU 通过 NVLink-C2C 连接,1 TB/s 内存带宽
  • GB10 Grace Blackwell Superchip: 面向桌面级 AI 超级芯片
能效优势:
  • 相比 PCIe Gen 6 PHY: 6x 更优的能效比
  • 一致性互连消除显式 DMA 传输,降低 CPU 开销
  • 支持统一内存模型,CPU 和 GPU 可直接 load/store 对方内存
来源: NVIDIA NVLink-C2C Official Page | Rubin Platform Announcement
下一代平台

Vera CPU + Rubin GPU: NVLink-C2C 的下一代演进

平台演进: Rubin 平台中,Vera CPU 采用 88 个定制 Olympus 核心 (Armv9.2 架构),通过 "ultrafast NVLink-C2C connectivity" 直连 Rubin GPU。这延续了 Grace-Blackwell 的 CPU-GPU 一致性互连设计,预计带宽将进一步提升。
来源: NVIDIA Rubin Platform
驱动特性

CDMM: 一致性驱动内存管理 (GB200 平台)

驱动支持: 数据中心驱动 580.126.09 (2026-01-13) 新增了 Coherent Driver-Based Memory Management (CDMM) for GB200 平台。CDMM 利用 NVLink-C2C 的一致性特性,实现驱动层面的高效内存管理,是 CPU-GPU 统一内存栈的关键组件。
来源: Data Center Driver 580.126.09 Release Notes

MMU/UVM 模块追踪

Bug #1187

BAR1 VA-space 耗尽导致系统硬锁死

严重问题: Turing GPU (256 MiB BAR1) 上,浏览器 WebGL 工作负载导致 BAR1 虚拟地址空间耗尽,触发 Xid 31 → Xid 154,RC watchdog 报告 "GPU is probably locked",需硬重启。影响驱动版本 595.71.05。
技术分析:
  • 根因: BAR1 VA-space 仅 256 MiB (Turing 限制),WebGL 工作负载的内存映射请求超出可用空间
  • 触发路径: WebGL context → GPU buffer allocation → BAR1 mapping → VA exhaustion → MMU fault (Xid 31) → GPU hang (Xid 154)
  • 相关代码路径: kernel-open/nvidia/ 中的 BAR1 映射逻辑
  • 影响范围: Turing 架构 (T4, RTX 20-series) 受 BAR1 大小限制影响最大
来源: GitHub Issue #1187
Bug #1134

RTX 3090: __nv_drm_gem_nvkms_map PCI BAR1 越界 → Xid 31 → Xid 154

MMU 相关: Kernel 7.0.3 + nvidia-open 595.71.05 在 RTX 3090 上,__nv_drm_gem_nvkms_map 函数请求范围超出 PCI BAR1 限制,导致 Xid 31 (MMU Fault) → Xid 154 (GPU requires Node Reboot)。Chromium GPU 工作负载触发。
代码路径:
  • 关键函数: __nv_drm_gem_nvkms_map()
  • 问题: DRM GEM 映射请求的地址范围超出 PCI BAR1 物理空间
  • 错误链: BAR1 overflow → MMU page fault (Xid 31) → unrecoverable GPU hang (Xid 154)
  • 环境: Kernel 7.0.3, driver 595.71.05, RTX 3090 (Ampere)
来源: GitHub Issue #1134
Bug #1064

Blackwell 笔记本: GSP heartbeat 卡死 + S0ix 电源管理

GSP 相关: RTX PRO 1000 Blackwell 笔记本上,GSP heartbeat 自启动起卡在 0,S0ix 电源管理无法正常工作。该问题涉及 GSP 固件与开放内核模块的交互,影响 Blackwell 移动平台的电源管理可靠性。
来源: GitHub Issue #1064
Bug #971

B200: nvidia-smi 在 ~66 天 uptime 后无限挂起 (NV-Triaged)

数据中心关键 Bug: 驱动 570.133.20 OpenRM 在 B200 + kernel 6.6.0 环境下,nvidia-smi 在约 66 天 12 小时后无限挂起。该 Issue 已被 NVIDIA 标记为 "NV-Triaged",表明已进入内部处理流程。可能与 UVM/GSP 内部计时器或内存泄漏有关。
来源: GitHub Issue #971
Bug #1151

RTX 5080 (GB203): 随机 Xid 79 — GPU 原子级死亡

高危 Bug: RTX 5080 (GB203) 上出现随机 Xid 79 错误,无 precursor,任何负载水平均可触发,导致 GPU 瞬间 "原子级死亡"。使用开放内核模块 595.71.05。该问题仍在调查中,持续有更新 (最近 2026-07-14)。
来源: GitHub Issue #1151

趋势展望

短期趋势 (1-3 个月)

  • 610 分支快速迭代: 610.43.03 刚发布,预计 610.43.x 修复版本将在数周内推出,重点关注 Blackwell 稳定性 (Xid 79/120) 和 BAR1 映射问题
  • Linux Kernel 7.2 适配: PR #1227 和 Issue #1224 表明社区正在积极推动 7.2 内核支持,预计将在下一个 release 中合入
  • PREEMPT_RT 支持: Issue #1240 反映实时内核用户需求增长,NVIDIA 可能需要在数据中心场景下优先解决
  • 发行版默认切换: Arch Linux 已默认使用开源驱动,预计更多发行版将在 2026 下半年跟进

中期趋势 (3-12 个月)

  • Rubin 平台开源驱动支持: NVLink 6 + Vera CPU 需要新的开源内核模块支持,预计 2026 H2 开始有代码合入
  • NVLink Partition Mode 成熟: 580.126.09 引入的 Multi-tenancy NVLink Partition Mode 将在数据中心多租户场景下广泛部署
  • CDMM 扩展: Coherent Driver-Based Memory Management 将从 GB200 扩展到 Rubin 平台,成为 CPU-GPU 统一内存栈标准
  • 安全审计加强: Quarkslab 等安全研究机构的 GPU 内核漏洞研究将推动 NVIDIA 加强开源模块的安全审计

核心洞察

1. 610 新分支开启新周期: 610.43.03 是继 580/595 之后的新驱动分支,通常意味着新架构支持或重大 API 变更。结合 Rubin 平台 2026 H2 交付时间表,610 分支很可能包含 Vera-Rubin 平台的初步驱动支持。
来源: Release 610.43.03
2. BAR1 内存映射成为高频痛点: 本周 #1187 (Turing) 和 #1134 (Ampere) 两个独立 Issue 都指向 BAR1 VA-space 溢出导致的 MMU fault。这反映了随着 Web 应用 GPU 使用增加,旧架构 BAR1 空间不足的问题正在加剧。
来源: #1187 | #1134
3. Blackwell 消费级稳定性待提升: RTX 5080 (#1151, Xid 79) 和 RTX 5070 Ti (#1239, GSP crash) 的严重 Bug 表明 Blackwell 消费级 GPU 在 Linux 开源驱动上的稳定性仍有显著差距。数据中心 B200 的长期运行挂起 (#971) 同样值得关注。
来源: #1151 | #1239 | #971
4. NVLink 生态加速扩展: NVLink 6 (3.6 TB/s/GPU) 进入量产,NVLink Fusion 引入 Marvell,NVLink-C2C 向 Vera-Rubin 延伸。NVLink 正从 "GPU 间互连" 演变为 "数据中心 AI fabric",涵盖 GPU-GPU、CPU-GPU、和第三方芯片互连。
来源: Rubin Platform | NVLink Fusion
