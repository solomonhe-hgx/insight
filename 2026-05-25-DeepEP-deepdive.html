DeepEP 是什么

DeepEP (Deep Every Parallel) 是由 DeepSeek 开源的、专为 Mixture-of-Experts (MoE) 模型中 Expert Parallel (EP) 场景设计的高性能通信库。它提供了高吞吐、低延迟的 GPU 全对全（all-to-all）通信内核，覆盖 MoE 的 dispatch（token 分发）和 combine（结果聚合）两个核心阶段，支持 FP8/BF16 等低精度数据类型。

V2 是 2026 年 4 月 23 日发布的完全重构版本，核心变化包括： (1) 弃用 NVSHMEM，切换为更轻量的 NCCL Gin 后端； (2) 全部 CUDA kernel 改为 JIT 运行时编译； (3) 统一的 ElasticBuffer API 替代了 V1 中分离的 high-throughput / low-latency 接口； (4) SM 占用从 24 降到 4-6，同时保持或超越 V1 性能。

V1 → V2 关键变化

  • 后端切换：从 NVSHMEM 切换到 NCCL Gin — NVSHMEM 是重量级 PGAS 运行时，NCCL Gin 是 header-only 轻量模块，复用已有 NCCL communicator GitHub 设计原理
    NCCL Gin 后端切换详细设计原理
    核心问题：V1 使用 NVSHMEM 作为 RDMA 后端。NVSHMEM 是一个完整的 PGAS (Partitioned Global Address Space) 编程模型——它需要独立的初始化流程（nvshmem_init）、独立的内存管理（nvshmem_malloc）、独立的同步机制（nvshmem_barrier）。这不仅增加了代码复杂度，还意味着 DeepEP 和 NCCL（用于其他 collective 如 all-reduce）运行在两套独立的通信栈上，无法共享 communicator 和拓扑信息。在 DeepSeek-V3 这样的大模型训练中，一个 rank 既要用 NCCL 做 all-reduce（data parallel），又要用 DeepEP 做 dispatch/combine（expert parallel），两套通信栈并存会带来：(1) 双倍的 NIC queue pair 占用；(2) 独立的内存注册和地址翻译开销；(3) 调试困难——两套栈的日志和诊断工具不同。
    设计原理：NCCL Gin (GPU-Initiated Networking) 是 NCCL 2.30+ 引入的 Device API 扩展。它允许 GPU kernel 直接发起 RDMA 操作（put/get/signal），而不需要 CPU 协调。关键优势：(1) Header-only 轻量集成：NCCL Gin 只是一个头文件库（nccl/device_api.h），不需要独立的运行时进程。只需在构建时包含 NCCL header，kernel 中调用 ncclGin_Put/Get 等函数即可。(2) 复用 NCCL communicator：DeepEP V2 的 ElasticBuffer 初始化时复用已有的 NCCL communicator（来自 PyTorch 的 ProcessGroup），共享同一个 NIC queue pair 和内存注册表。(3) 统一的拓扑感知：NCCL 的拓扑检测（PCI 扫描、NVML、NUMA 感知）同样服务于 DeepEP 的通信路径选择，不需要单独的实现。
    适用场景：所有同时使用 Data Parallel + Expert Parallel 的训练场景。特别是在 DeepSeek-V3 这类模型中，DP 和 EP 混合使用，统一通信栈可以显著降低系统复杂度和内存开销。
  • 全 JIT 编译：所有 CUDA kernel 运行时 JIT 编译，安装时无需 CUDA 工具链，按硬件和配置动态生成最优代码 GitHub 设计原理
    JIT 编译详细设计原理
    核心问题：DeepEP 的 dispatch/combine kernel 有大量编译时常量参数：hidden 维度、top-k 数量、是否 FP8、专家数量、每 rank token 上限、SM 数量等。V1 的做法是预编译一组 kernel 变体（类似 "template instantiation matrix"），在 setup.py 中用 nvcc 编译所有组合。这导致：(1) 安装时间长（需要完整 CUDA 工具链）；(2) 二进制文件巨大（数百 MB）；(3) 新的参数组合需要重新编译库。
    设计原理：V2 改为 JIT 运行时编译。具体流程：(1) Kernel 模板：源代码中维护 CUDA kernel 的 C++ 模板，使用宏和常量参数化所有可变维度。(2) 运行时生成：当 ElasticBuffer 初始化时，根据传入的 hidden、topk、fp8 等参数，动态生成一个特定于此配置的 CUDA C 源码字符串。(3) 在线编译：调用 PyTorch 的 JIT 编译接口（类似 torch.utils.cpp_extension.load_inline），在首次 dispatch/combine 调用时将源码编译为 PTX → SASS。(4) 缓存：编译结果缓存到本地磁盘，下次相同配置时直接加载，无需重新编译。
    适用场景：(1) 快速部署——用户只需 pip install，无需 nvcc；(2) 动态配置——不同模型的 hidden 维度、expert 数量差异很大，JIT 可以按需生成最优 kernel；(3) 迭代开发——开发者修改 kernel 源码后，用户无需重新安装库，下次运行时自动编译新代码。
  • SM 占用大幅降低：V1 需要 24 个 SM 做通信，V2 仅需 4-6 个 SM（同等性能下），释放更多计算资源给 GEMM GitHub 设计原理
    SM 占用优化详细设计原理
    核心问题：在 MoE 训练/推理中，GPU 的 Streaming Multiprocessor (SM) 是 dispatch/combine 通信 kernel 和 GEMM 计算 kernel 的共享资源。V1 中通信 kernel 占用 24 个 SM（H100 共 132 个 SM，约占 18%），这意味着可用于 GEMM 的 SM 减少了 18%。对于 MoE 这种 compute-bound 的场景，通信 SM 占用直接影响 GEMM 吞吐。
    优化手段：(1) NCCL Gin 的零-copy 特性：V1 使用 NVSHMEM 时，数据需要先拷贝到 NVSHMEM 的 staging buffer，再由 NVSHMEM 的 host proxy 或 SM 发起 RDMA。V2 使用 NCCL Gin 后，GPU kernel 直接发起 RDMA put/get，数据从用户 buffer 直接传输，消除了中间拷贝步骤和对应的 SM 开销。(2) 解析式 SM 计算：V2 的 ElasticBuffer 根据消息大小、NIC 带宽、SM 频率等参数，通过解析公式计算出最优 SM 数量（而非 V1 的 heuristic 固定值）。例如 EP 8x4 + RDMA 场景计算出只需 6 个 SM 就能饱和 400 Gbps 带宽。(3) Warp 级并行优化：每个 SM 内的 warp 调度更高效——用更少的 warp 完成相同的传输量（通过更大的 per-warp 传输块大小和更好的 L2 cache 复用）。
    适用场景：所有 MoE 训练和推理场景。特别是推理场景（如 vLLM + DeepEP），SM 资源直接决定每 GPU 的 token throughput。
  • 统一 API：High-throughput 和 low-latency 模式合并到单一 ElasticBuffer 接口，简化集成 GitHub
  • 规模扩展：支持最大 EP2048（2048 个 expert 并行域），适配超大模型训练 GitHub

架构 DeepEP 整体架构设计

ElasticBuffer — 统一通信缓冲区

V2 核心组件

ElasticBuffer 是 DeepEP V2 的核心抽象，替代了 V1 中分散的 buffer 管理类。它统一管理 dispatch 和 combine 的内存分配、SM 调度、QP (Queue Pair) 管理，并通过解析式计算自动确定最优 SM/QP 数量，无需手动调优 GitHub

技术细节

  • 解析式 SM/QP 计算：根据消息大小、NIC 数量、带宽、GPU SM 频率等参数，通过解析公式自动计算最优 SM 数量和 QP 数量，取代 V1 的 auto-tuning
  • 统一 high/low-latency：单一接口支持高吞吐和低延迟两种模式，内部自动选择最优策略
  • 内存消耗增大：官方文档注明 V2 buffer size 消耗比 V1 更大（trade-off for performance）

NCCL Gin 后端 — GPU 直接发起网络操作

V2 后端

NCCL Gin (GPU-Initiated Networking) 允许 GPU kernel 直接发起 RDMA put/get/signal 操作，无需 CPU 协调。V2 用它完全替代了 V1 的 NVSHMEM 后端，实现 header-only 轻量集成，复用 NCCL communicator GitHub NCCL EP Paper

NCCL Gin 核心原语

  • ncclGin_Put：RDMA Write — 将本地 GPU 内存数据写入远程 GPU 内存地址（通过 memory window 寻址）
  • ncclGin_Get：RDMA Read — 从远程 GPU 内存读取数据到本地 GPU 内存
  • ncclGin_SignalAdd：原子操作 — 原子地增加远程信号值，用于同步通知
  • ncclGin_WaitOnSignal：等待原语 — 阻塞等待远程信号达到阈值后继续执行

实验特性：0 SM 模式

Experimental

DeepEP V2 在实验分支中提供 0 SM 模式——通信完全不占用 SM 资源。包括 0 SM Engram (RDMA)、0 SM PP (RDMA)、0 SM CP (Copy Engine)。这些模式利用硬件 copy engine 和 RDMA 引擎直接在后台传输，无需 SM 执行 kernel 代码 GitHub

0 SM 模式分析

0 SM 模式的核心思想是将通信操作从 SM 转移到专用的硬件引擎：(1) 0 SM Engram：使用 RDMA NIC 的 hardware engine 直接执行 put/get，GPU 侧只需设置一次 doorbell register，之后 NIC 自主完成传输。(2) 0 SM PP (Pipeline Parallel)：类似 Engram 但针对 pipeline 场景优化，支持流水线式连续传输。(3) 0 SM CP (Copy Engine)：使用 GPU 的 copy engine（如 Hopper 的 TMA, Tensor Memory Accelerator）执行设备内和设备间的数据搬运，完全不占用 SM。需要注意的是，这些模式目前仅在实验分支中，且需要特定硬件支持（ConnectX-7 + Hopper 及以上）。

Dispatch Token 分发机制深度解析

Dispatch 流程概览

核心操作

Dispatch 阶段负责将每个 token 从源 GPU 路由到其被分配的 expert 所在的 GPU。在 MoE 中，每个 token 经 gate 网络计算后被分配给 top-k 个 expert（通常 k=8），这些 expert 可能分布在不同 GPU 上。DeepEP 的 dispatch 核心是 all-to-all 通信——每个 GPU 既是发送方也是接收方。

Dispatch 分步流程

1
路由计算：Gate 网络输出每个 token 的 top-k expert 分配（topk_idx），以及对应的权重（topk_weights）。这决定了每个 token 需要发送到哪些 GPU。
2
布局计算：根据 topk_idx 统计每个目标 rank 需要接收的 token 数量，确定 dispatch 的内存布局。这涉及对 token 按目标 rank 排序和对齐（expert_alignment）。
3
通知阶段（Notify）：通过 NCCL Gin 的 Put 操作，将元数据（每 rank 发送的 token 数量）写入对端的小型协调缓冲区。对端收到通知后知道要从哪里拉多少数据。
4
数据传输：接收方 GPU 通过 NCCL Gin Get（RDMA Read / NVLink Load）从发送方拉取 token 数据。节点内使用 NVLink（~740 GB/s），节点间使用 RDMA（~90 GB/s）。
5
接收完成：接收方 GPU 将 token 数据按 expert 分组存入连续的 receive buffer，返回 recv_x, recv_topk_idx, recv_topk_weights 供后续 expert 计算使用。

Dispatch 为什么用 Load (Pull) 而非 Store (Push)

核心设计

Dispatch 阶段接收方采用 Load/Pull 语义——接收方主动从发送方拉取数据。这源于信息不对称：发送方知道每个 token 该去哪，但接收方不知道会收到多少 token、来自哪些源 GPU。

Dispatch Pull 详细设计原理

核心问题：在 MoE dispatch 中，每个 token 的路由（去哪个 expert）是 gate 网络在运行时动态计算的。发送方 GPU 拥有完整的路由信息（topk_idx），但接收方 GPU 对此一无所知——它不知道会有多少 token 到达、来自哪些源 GPU、分配给哪些本地 expert。如果用 Store/Push，接收方必须预分配 worst-case buffer（假设所有源的所有 token都涌来），即 O(E × B × P) 的内存，而实际 token 分布极度不均衡（standard deviation 可达 0.05），浪费极大。
设计原理：采用两阶段 Pull 方案：(1) 元数据交换：发送方通过 NCCL Gin Put（RDMA Write with immediate data）将路由元数据（每 rank 的 token 数量）写入接收方的小型协调缓冲区。这一步数据量极小（仅 count 信息）。(2) 按需拉取：接收方收到元数据后，知道自己要从每个源拉多少数据，然后通过 NCCL Gin Get（RDMA Read / NVLink Load）精确拉取。拉取是连续的、大块的数据传输，带宽利用率远高于分散的 push write。(3) 内存优化：接收方只需分配 O(N × B × P) 的实际需要缓冲区（N=目标 rank 数），而非 O(E × B × P) 的 worst case（E=expert 总数）。
适用场景：(1) 训练阶段——token 分布高度动态，无法预测；(2) 大规模 EP（EP64+）——源 GPU 多，worst-case buffer 爆炸性增长；(3) 不均衡 expert 负载——某些 expert 接收更多 token，pull 模式能自适应。

FP8 Dispatch — 低精度传输

性能优化

DeepEP 支持 FP8 (E4M3/E5M2) 格式的 dispatch 数据传输，将 token activation 从 BF16 压缩到 8-bit，带宽需求减半 GitHub

FP8 Dispatch 详细设计原理

核心问题：Dispatch 是带宽密集型操作。以 DeepSeek-V3 为例：8K tokens × 7168 hidden × 2 bytes (BF16) = ~117 MB 每批次。在 RDMA（400 Gbps = 50 GB/s）下传输需要 ~2.3 ms。而 FP8 只需 58.5 MB，传输时间降到 ~1.2 ms，减少近一半。
设计原理：(1) 量化点：FP8 将 activation 从 16-bit 压缩到 8-bit，使用 E4M3 (4 exponent, 3 mantissa) 或 E5M2 格式。在 MoE dispatch 场景下，activation 的数值范围有限（经过 layer norm 后），FP8 的精度损失可接受。(2) 反量化：接收方 expert 在计算前将 FP8 反量化为计算所需的精度（通常是 FP8×FP4 用于 DeepGEMM 的 MegaMoE kernel，或 BF16 用于传统 GEMM）。反量化和 expert 计算可以 overlap 或在同一个 fused kernel 中完成。(3) 粒度：DeepEP 支持 per-token 或 per-group 的 FP8 scaling，避免全局 scale factor 导致的精度损失。
适用场景：所有带宽受限的 MoE 通信场景。特别是跨节点 RDMA 传输，FP8 dispatch 可以显著降低通信时间占比。

NVLink 节点内优化

节点内

节点内 token 传输使用 NVLink 而非 RDMA，带宽可达 740 GB/s（SM100 EP8）。同时使用 ld.global.nc.L1::no_allocate.L2::256B PTX 指令优化读取路径 BAAI Hub

NVLink 优化详细设计原理

核心问题：在 8-GPU 节点内，如果所有跨 GPU 通信都走 RDMA（需要通过 NIC → PCIe → NVLink 的迂回路径），会引入不必要的延迟。NVLink 提供 GPU 间的直接高速互连（H100 上 900 GB/s 双向带宽），但需要正确的内存访问模式才能饱和带宽。
设计原理：(1) LDG.NC 指令：DeepEP 在 NVLink 传输中使用 PTX 指令 ld.global.nc.L1::no_allocate.L2::256B。其中 .nc (non-coherent) 表示此 load 不参与 L1 cache coherence，适用于"读一次就丢弃"的通信数据，避免了 cache line 填充和 coherence traffic。L1::no_allocate 表示不分配 L1 cache 行，数据直接进入 register。L2::256B 指定 L2 传输粒度为 256B（匹配 NVLink 的 flit 大小）。(2) 带宽数据：H800 测试中 NVLink dispatch 达到 160 GB/s（单方向），SM100 (Blackwell) 下双向可达 740 GB/s。(3) 路径选择：DeepEP 自动判断传输是否在节点内——如果是，走 NVLink 路径；否则走 RDMA。这种分层策略减少了跨节点 RDMA 的负载。
适用场景：所有多 GPU 节点内的 MoE 通信。特别是 EP 规模 ≤ 8 时（单节点可容纳所有 expert），NVLink 性能优势最大。

Combine 结果聚合机制深度解析

Combine 流程概览

核心操作

Combine 阶段负责将 expert 计算结果从 expert 所在 GPU 返回给 token 的源 GPU，并按 gating 权重加权求和。与 dispatch 不同，combine 阶段的路由信息完全已知（来自 dispatch 阶段的缓存），因此采用 Store/Push 语义。

Combine 分步流程

1
Expert 计算完成：每个 GPU 上的 expert 对其分配的 token 完成前向计算（通常是 Linear → SwiGLU → Linear 的 FFN 结构），输出 expert output。
2
缓存路由复用：Combine 直接使用 dispatch 阶段缓存的 handle（包含每个 token 的源 rank、在源 buffer 中的偏移、gating 权重），无需重新计算路由。
3
Scatter Write (Push)：Expert GPU 根据缓存的路由信息，将每个 token 的 expert output 通过 NCCL Gin Put（RDMA Write / NVLink Store）直接写入源 GPU 的预计算偏移位置。由于写入目标地址已知，不需要元数据交换——一步到位。
4
加权求和：源 GPU 收到来自多个 expert 的结果（每个 token 有 top-k 个 expert 输出），按 dispatch 阶段缓存的 topk_weights 加权求和，得到最终的 combined_x。

Combine 为什么用 Store (Push) 而非 Load (Pull)

核心设计

Combine 阶段发送方采用 Store/Push 语义——发送方直接推送数据到接收方的预计算地址。这是因为 dispatch 阶段已经缓存了完整的路由信息，发送方知道每个 expert output 该去哪里。

Combine Push 详细设计原理

核心问题：与 dispatch 不同，combine 阶段的信息状态完全反转——expert GPU 知道每个 token 的 expert output 应该返回给哪个源 GPU，且源 GPU 上的写入偏移已经在 dispatch 阶段预计算好了。如果此时还用 Pull 模式，接收方需要先告诉发送方"我要什么"，发送方再等待、再响应，白白多一次 round-trip 延迟。
设计原理：(1) 已知路由：Dispatch 阶段返回的 handle 包含了完整的路由元数据：每个 token 的源 rank、目标偏移、gating 权重。Combine 直接使用这些信息，无需额外的元数据交换。(2) Scatter Write：Expert GPU 将 output 按目标 rank 分组，对每个目标 rank 执行一次 bulk RDMA Write（或 NVLink Store），直接写入接收方 buffer 的预计算偏移。这是一阶段操作——打包 + 写入，没有"等待-响应"循环。(3) Buffer 复用：Combine 的目标 buffer 就是 dispatch 时分配的 receive buffer——内存已经就位，combine 只需往已知地址写入。(4) 带宽优势：Push 模式下写入是连续的、大块的（per-destination-rank 的 bulk write），带宽利用率高。Dispatch 的 Pull 模式因为要按源拉取，写入相对碎片化。
适用场景：所有 MoE combine 场景。特别是 top-k > 1 时，同一个 token 有多个 expert output 需要加权求和，push 模式可以高效地将多个输出写入同一位置（利用 RDMA 的原子操作或接收方的 accumulate 逻辑）。

Combine 也是 Dispatch 的反向 — 训练中的对称性

训练

在 MoE 训练中，dispatch 的反向传播等价于 combine，combine 的反向传播等价于 dispatch。DeepEP 利用这种对称性复用同一个 kernel 实现，减少代码量和维护成本 GitHub

训练对称性详细设计原理

核心问题：MoE 的前向传播是 dispatch → expert compute → combine。反向传播需要计算 gradient 对 dispatch 和 combine 的偏导。数学上，dispatch 的 Jacobian transpose 就是 combine，combine 的 Jacobian transpose 就是 dispatch——它们互为转置操作。
设计原理：DeepEP 在前向 dispatch 时缓存 handle（路由信息），在反向传播时：(1) Combine 的反向 = Dispatch：gradient of combined_x 需要按 dispatch 的逆路径分发回 expert，等价于一次 dispatch 操作。(2) Dispatch 的反向 = Combine：gradient of expert output 需要按 combine 的逆路径聚合回源 token，等价于一次 combine 操作。因此 DeepEP 只需实现 dispatch 和 combine 两个 kernel，前向和反向通过交换角色复用，无需额外的 backward kernel。
适用场景：所有 MoE 训练场景。这种对称性在 PyTorch autograd 中通过 custom Function 的 backward() 方法实现。

通信-计算重叠

async_with_compute_stream

async_with_compute_stream=True 使 DeepEP 的 dispatch/combine 与 GPU 计算 stream 重叠执行。DeepEP 内部使用 CUDA event 同步，确保数据传输和专家计算并发执行，隐藏通信延迟 GitHub

通信-计算重叠详细设计原理

核心问题：在 MoE 中，通信（dispatch/combine）和计算（expert GEMM）在同一个 GPU 上执行。如果串行执行（通信→计算→通信→计算），通信延迟会直接加到 total latency 中。但 GPU 有足够的 SM 资源同时执行通信 kernel 和计算 kernel——只要正确调度。
设计原理：(1) CUDA Stream 模型：PyTorch 默认使用一个 CUDA stream 做计算。当 async_with_compute_stream=True 时，DeepEP 的通信 kernel 也提交到同一个 stream（或通过 CUDA event 与 compute stream 同步）。(2) 硬件级重叠：在 Hopper/Blackwell GPU 上，SM 可以同时调度不同的 kernel block。DeepEP 的通信 kernel 只占用 4-6 个 SM，剩余的 126-128 个 SM 可以同时执行 GEMM kernel。此外，GPU 的 copy engine（DMA engine）可以独立于 SM 执行数据传输，与 SM 上的计算并行。(3) Event 同步：DeepEP 返回的 CUDA event 可以在 PyTorch 侧用于同步——compute stream 等待 dispatch event 完成后再启动 expert GEMM，而 dispatch 的传输过程（由 copy engine 或空闲 SM 执行）与 compute stream 的其他操作重叠。
适用场景：所有 MoE 训练和推理场景。通信-计算重叠是 MoE 性能优化的关键——理想情况下通信完全被计算隐藏，total latency ≈ max(通信时间, 计算时间)。

各方案核心差异分析

DeepEP V2 优势

  • 极致性能：NVLink 740 GB/s dispatch/combine，RDMA 90 GB/s
  • 极低 SM 占用：4-6 SM，释放 126+ SM 给 GEMM
  • 零编译部署：JIT 运行时编译，pip install 即可
  • 生产验证：DeepSeek-V3/R1 训练和推理已验证

DeepEP V2 局限

  • EFA 不支持：仅支持 InfiniBand/RoCE，不支持 AWS EFA
  • ConnectX 绑定：NCCL Gin 依赖特定 NIC 能力
  • Buffer 消耗增大：V2 比 V1 需要更多 GPU 内存
  • 生态孤立：独立于 NCCL 标准 collective 栈

NCCL EP 优势

  • 统一栈：与 NCCL all-reduce/broadcast 共享 communicator
  • 拓扑感知：复用 NCCL 的硬件拓扑检测
  • 弹性容错：继承 NCCL 的 fault-tolerance
  • 内存效率：比 DeepEP 原始 buffer layout 减少一个数量级

UCCL-EP 优势

  • EFA 支持：唯一支持 AWS EFA (SRD 协议) 的高性能方案
  • 异构 GPU：同时支持 NVIDIA 和 AMD GPU
  • 可移植性：Host proxy 架构，不依赖特定 NIC 能力
  • RL 权重传输：万亿参数模型权重传输仅 1.3 秒

趋势 MoE 通信库技术趋势展望

短期趋势 (3-6 个月)

  • NCCL EP 持续追赶：NCCL EP 论文（arXiv:2603.13606）已展示在低延迟 dispatch 上匹配/超过 DeepEP 的性能。预计 NVIDIA 会在后续版本中集成 ncclEpDispatch/ncclEpCombine 原语，形成 DeepEP 的直接竞争对手
  • UCCL-EP EFA 生态：UCCL-EP 在 EFA 支持上的独特优势将推动 AWS 生态的 MoE 训练。ConnectX 到 EFA 的 bridge 需求持续增长
  • 0 SM 模式成熟：DeepEP 的 0 SM Engram/PP/CP 实验分支将逐步成熟，最终合并到主线——完全消除通信 SM 占用
  • MegaMoE 融合：DeepGEMM 的 MegaMoE（dispatch+GEMM+combine 单 kernel）将继续发展，推动通信和计算的深度融合

中期趋势 (6-18 个月)

  • 统一通信栈：NCCL + NIXL + NVSHMEM 的分层定位将逐渐清晰——NCCL 负责标准 collective，NIXL 负责跨框架数据移动，NVSHMEM 负责 fine-grained RMA。DeepEP 可能被 NCCL EP 吸收为原生原语
  • 异构通信：随着 AMD MI300/MI400 系列和 Intel GPU 的竞争，跨厂商 GPU 的 MoE 通信需求增加。UCCL-EP 的异构方案将成为重要选项
  • FP4 普及：DeepGEMM 已提出 FP4 (E2M1) activation dispatch。随着 Blackwell 原生支持 FP4，FP4 dispatch 将成为标准配置
  • 推理优先：MoE 推理（vLLM + DeepEP）将比训练更快采纳新通信优化——推理对延迟更敏感，且部署环境更标准化

竞争格局总结

DeepEP V2 目前代表 MoE Expert Parallel 通信的最高性能标杆——NVLink 740 GB/s、RDMA 90 GB/s、SM 占用仅 4-6 个。但它面临着来自两个方向的竞争压力：

一方面，NCCL EP 以统一通信栈为核心优势（与 NCCL 共享 communicator、拓扑感知、容错），性能已接近甚至在某些场景超过 DeepEP，预计将成为 NVIDIA 的官方 MoE 通信方案。

另一方面，UCCL-EP 以 EFA 支持和异构 GPU 能力开辟了差异化赛道，为 AWS 生态和跨厂商部署提供了 DeepEP 无法覆盖的选项。

整体而言，MoE 通信领域正从"DeepEP 一家独大"向"多方案分层竞争"演进——极致性能选 DeepEP，统一栈选 NCCL EP，异构/EFA 选 UCCL-EP，深度融合选 MegaMoE。
